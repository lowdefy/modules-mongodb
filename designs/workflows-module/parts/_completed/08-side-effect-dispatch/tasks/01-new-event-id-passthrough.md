# Task 1: Extend `new-event.yaml` to honor a caller-supplied `_id`

## Context

The events module's [`new-event` Api](../../../../../modules/events/api/new-event.yaml) currently mints a fresh UUID for every inserted event doc:

```yaml
_id:
  _uuid: true
```

The workflows engine ([SubmitWorkflowAction.js:15](../../../../../plugins/modules-mongodb-plugins/src/connections/WorkflowAPI/SubmitWorkflowAction/SubmitWorkflowAction.js)) generates its own `eventId` on handler entry and threads it through every action transition's `status[0].event_id` ([handleSubmit.js:196,210,298](../../../../../plugins/modules-mongodb-plugins/src/connections/WorkflowAPI/SubmitWorkflowAction/handleSubmit.js)). The engine spec commits this as a hard contract ([engine/spec.md:213](../../../../workflows-module-concept/engine/spec.md)): *"eventId: string — generated by SubmitWorkflowAction on entry; threaded through every write in this invocation"*. One id per invocation.

If the engine calls `new-event` without override, the event doc's `_id` is a fresh UUID that diverges from every action's `status[0].event_id` — breaking the spec's one-id-per-invocation guarantee. This task adds the override path so the engine can pass `_id: context.eventId` on the payload and have the doc inserted with that id.

The change is backwards-compatible: existing app-level callers (e.g. `apps/demo`'s contact creation flows in [api-routines.md](../../../../../apps/demo/.claude/guides/api-routines.md)) don't pass `_id` on the payload, so they fall through to `_uuid: true`.

## Task

Edit [modules/events/api/new-event.yaml](../../../../../modules/events/api/new-event.yaml). In the `MongoDBInsertOne` request's `doc._object.assign` block, replace:

```yaml
_id:
  _uuid: true
```

with:

```yaml
_id:
  _if_none:
    - _payload: _id
    - _uuid: true
```

The rest of the routine is unchanged — `_object.assign` order still favors payload `display`/`references` first, then the engine-supplied core fields, with no behavioural change for app callers.

## Acceptance Criteria

- [modules/events/api/new-event.yaml](../../../../../modules/events/api/new-event.yaml) uses `_if_none: [_payload: _id, _uuid: true]` for the `_id` field.
- `pnpm ldf:b` (Lowdefy build) succeeds in `apps/demo` — confirms YAML is still valid.
- Calling the api without an `_id` on the payload still inserts a fresh UUID (the demo's existing event flows keep working).
- Calling the api with `_id: '<some-uuid>'` on the payload inserts a doc whose `_id` matches.

## Files

- `modules/events/api/new-event.yaml` — modify — replace the `_uuid: true` line with the `_if_none` block.

## Notes

- The `_if_none` operator is the established pattern for "use payload field if present, otherwise default" — see [contact-fields.md](../../../../../apps/demo/.claude/guides/contact-fields.md) for several existing call sites. Don't substitute `_if`/`_eq` — `_if_none` reads better and is the convention.
- Don't bump the events module version — this is an additive, backwards-compatible change inside the same Api routine.
- Do **not** touch the `eventId` return — the routine still returns `eventId: { _step: insert-event.insertedId }`, which now reflects either the caller-supplied id (when present) or the generated UUID.
