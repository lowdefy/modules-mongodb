# Review 1 тАФ Side-effect dispatch (Part 8)

Cross-checked Part 8's design against the submit-pipeline spec, the events / notifications modules already in `modules/`, the engine spec's commitments on `eventId`, and the in-progress Part 6 handler at `plugins/modules-mongodb-plugins/src/connections/WorkflowAPI/SubmitWorkflowAction/handleSubmit.js`. Several findings tie back to mismatches between the design's wording and what's already running in code.

## Correctness

### 1. `event_id` is generated in Part 6, not "captured from the new-event return"

> **Resolved.** Picked option (a). `dispatchLogEvent` now passes `_id: context.eventId` into the `new-event` payload and returns `context.eventId` directly. Added an in-scope `new-event.yaml` extension that honors a caller-supplied `_id` with `_if_none: [_payload: _id, _uuid: true]` fallback so existing app-level callers stay unchanged.

The design (┬з `dispatchLogEvent.js`) says: "Fires `context.callApi({ id: 'new-event', module: 'events' }, ...)`. Captures `event_id` and threads it into the return payload."

This is inconsistent with what's already on disk and with the engine spec:

- [SubmitWorkflowAction.js:15](../../../../plugins/modules-mongodb-plugins/src/connections/WorkflowAPI/SubmitWorkflowAction/SubmitWorkflowAction.js) generates `eventId: randomUUID()` on handler entry and stuffs it into `context.eventId`.
- [handleSubmit.js:196,210,298](../../../../plugins/modules-mongodb-plugins/src/connections/WorkflowAPI/SubmitWorkflowAction/handleSubmit.js) threads `context.eventId` into every `action.status[].event_id` via `updateAction`.
- [engine/spec.md:213](../../../workflows-module-concept/engine/spec.md) commits: *"eventId: string тАФ generated by SubmitWorkflowAction on entry; threaded through every write in this invocation"*. One id per invocation, by design.
- But [modules/events/api/new-event.yaml:14](../../../../modules/events/api/new-event.yaml) inserts a doc whose `_id: { _uuid: true }` is generated **inside the API routine**, and returns `eventId: { _step: insert-event.insertedId }` тАФ a different UUID.

If Part 8 "captures `event_id` from `new-event`'s response and threads it into the return payload", the returned `event_id` will not match the id stamped on every action's `status[0].event_id` written in step 4. Downstream consumers reading `result.event_id` will fail to join back to action transitions in the same submit.

**Fix.** Either:

- (a) Pass `_id: context.eventId` into the `new-event` payload and extend [new-event.yaml](../../../../modules/events/api/new-event.yaml) (or a workflow-specific variant) to honor a caller-supplied id instead of always generating one, then return `context.eventId` from Part 8; or
- (b) Reverse the order: call `new-event` **first** in step 7, then thread its returned id back through step 4 (much bigger change; conflicts with the spec's "engine generates on entry" commitment).

(a) is the right call тАФ it preserves the spec's one-id-per-invocation guarantee. State this explicitly in ┬з `dispatchLogEvent.js` and add the `new-event.yaml` extension to "In scope" (or carve out a sibling task for it).

### 2. `display` shape mismatch against the live events module

> **Resolved.** Picked (a-revised): workflows module declares `app_name` as a manifest var (mirroring notifications module). Part 20 plumbs it through `WorkflowAPI.connection.app_name`; Part 8's `buildDefaultLogEventPayload` reads it and emits `display: { [appName]: { title: ... } }`. Consuming apps wire `app_name` such that `events.display_key === workflows.app_name`, matching the established `app_name` idiom in [docs/idioms.md](../../../../docs/idioms.md). The literal `default` key in the concept spec is wrong against the events module's per-`display_key` projection; spec-level cleanup tracked under finding #5.

The design quotes the spec's default-event template:

```yaml
display:
  default:
    title:
      _nunjucks:
        template: "{{ user.profile.name }} marked {{ action_type }} as {{ status_after }}"
```

But the live events module treats `display` as a **per-`display_key` map** keyed by the consuming app's `display_key` var, not by the literal key `default`:

- [modules/events/module.lowdefy.yaml:18-25](../../../../modules/events/module.lowdefy.yaml) declares `display_key` as a required var: *"App identifier for display objects. Events store per-app display titles keyed by app name"*.
- [modules/events/components/events-timeline.yaml:34-50](../../../../modules/events/components/events-timeline.yaml) extracts `$<display_key>.title` etc. via `$addFields` тАФ so the key under `display` must equal the consuming app's `display_key`, otherwise the timeline shows nothing.

If Part 8 inserts events with `display: { default: { title: ... } }` literally, those events will render in any app whose `display_key === 'default'` and silently render blank in all others. The worked example would fail the smoke test in ┬з Verification.

**Fix.** The engine doesn't know the consuming app's `display_key` from JS. Two options:

- (a) Have the workflows module manifest declare a `display_key` var of its own (mirroring events module), have Part 20 plumb it down to `WorkflowAPI`'s `connection` config, and use `connection.display_key` as the key here. State this dependency in Part 8 (or carve a sibling sub-task; flag in Open Questions).
- (b) Sidestep the per-app indirection: pass `display` already keyed by `connection.display_key` from the resolved connection at handler entry. Same plumbing, different naming.

Either way, Part 8 cannot ship with a hard-coded `default` key тАФ flag this and pick one.

### 3. `status_before` / `status_after` are not defined anywhere in scope

> **Resolved.** ┬з `dispatchLogEvent.js` now pins `status_before` as `context.action.status?.[0]?.stage` captured before step 4 mutates the in-memory cache, and `status_after` as the engine-resolved `targetStatus` computed at step 1.

Metadata field list includes `status_before` and `status_after`, and the spec's display template uses `{{ status_after }}`. Neither is in `handleSubmit`'s `params` shape, nor obviously derivable without re-reading the action doc post-write.

Concretely:

- `status_after` тЙИ the engine-resolved `targetStatus` from [handleSubmit.js:132-136](../../../../plugins/modules-mongodb-plugins/src/connections/WorkflowAPI/SubmitWorkflowAction/handleSubmit.js) for the user-submitted action.
- `status_before` тЙИ `action.status[0].stage` before the step-4 write (i.e. `context.action.status?.[0]?.stage` at handler entry).

The design should pin both. Otherwise Part 8's implementer is left guessing тАФ and given that the in-memory `context.workflowActions` is mutated in-place during step 4 ([handleSubmit.js:209-216](../../../../plugins/modules-mongodb-plugins/src/connections/WorkflowAPI/SubmitWorkflowAction/handleSubmit.js)), reading the "before" stage *after* step 4 already reflects the new value. **Capture `status_before` at step 1, before step 4 mutates the cache.**

**Fix.** Add to ┬з `dispatchLogEvent.js`:

> - `status_before` is captured at handler entry as `context.action.status?.[0]?.stage` before step 4 mutates the in-memory cache; `status_after` is the engine-resolved `targetStatus` for the user-submitted action (already computed at [handleSubmit.js:132](../../../../plugins/modules-mongodb-plugins/src/connections/WorkflowAPI/SubmitWorkflowAction/handleSubmit.js)).

### 4. Error-path interaction with Part 6's `error_transition` is undefined

> **Deferred to open questions.** Picked (a). Added an open question to ┬з Open questions pinning that Part 8 does not fire a log event for error transitions (Part 6's catch block or hypothetical step 7 failure). The action's `status` array carries durable error metadata; second-event semantics will be settled alongside Part 9's `hook_error` emission so both error paths get the same treatment.

[handleSubmit.js:265-290](../../../../plugins/modules-mongodb-plugins/src/connections/WorkflowAPI/SubmitWorkflowAction/handleSubmit.js) Part 6 returns early with `event_id: null` when a write step throws. The design's Open Question 1 says *"log-event failure throws an engine error transition, but the action transition is already durable"* тАФ but it doesn't address the inverse direction:

- If steps 4тАУ6 throw, the catch block writes the error transition with `event_id: context.eventId` ([handleSubmit.js:298](../../../../plugins/modules-mongodb-plugins/src/connections/WorkflowAPI/SubmitWorkflowAction/handleSubmit.js)). Should the engine **also** fire a log event documenting the error transition? Or are error transitions invisible in the events log?
- If Part 8's step 7 throws (the `new-event` call fails post-write), per the design's lean, the engine pushes an error transition on the user-submitted action. But the action already has a `done` (or `in-review`) stage written. Now `status[0]` is `error`, `status[1]` is the success transition. Is `status_before` reported in the error event the original status, or the success status?

**Fix.** Either:

- Pin in ┬з Open questions: error-path log events on step 4-6 failures are out of scope for Part 8 (the action's `status` array carries the error metadata for now; an audit of the error transition is post-v1 work); OR
- Add a "step 7 error тЖТ error transition" subsection that defines `status_before` / `status_after` for the second event, and confirms it bypasses the priority rule with `force: true` (the action's previous `error` status entry is already terminal-ish but unranked).

### 5. The `references` payload is too narrow тАФ `entity_ids` should be there too

> **Resolved.** Picked (c): added `<entity-ref-key>: [workflow.entity_id]` to `buildDefaultLogEventPayload`'s default, pinned the derivation rule (strip trailing `-collection`, kebabтЖТsnake, append `_ids`), and amended [submit-pipeline/spec.md ┬з Default log event](../../../workflows-module-concept/submit-pipeline/spec.md) to match (the spec's literal `default` key under `display` was also incomplete against the events module тАФ both fixed in the same edit).

Spec template: `references: { workflow_ids: [<id>], action_ids: [<id>] }`. The design copies this verbatim.

But this is the action timeline an *entity page* (lead, lot, contact, etc.) wants to render. The convention across the live events module is to also include `<entity>_ids: [<workflow.entity_id>]` so the entity's [events-timeline](../../../../modules/events/components/events-timeline.yaml) can match its own `reference_field`. Without it, the workflow event won't appear on the entity page тАФ only on a hypothetical workflow-detail page that filters by `workflow_ids`.

[apps/demo/.claude/guides/events.md:7](../../../../apps/demo/.claude/guides/events.md) confirms the convention: *"timeline component queries by {entity}_ids"*.

`workflow.entity_id` and `workflow.entity_collection` (per [Part 21](../../21-entity-type-to-collection/design.md)) are available on `context.workflow` at this point.

**Fix.** Decide one of:

- (a) Add `references[<entity-ref-key>]: [workflow.entity_id]` to the default, where `<entity-ref-key>` derives from `workflow.entity_collection` (e.g. `tickets-collection` тЖТ `tickets_ids`). Document the derivation rule in Part 8 (likely just `entity_collection.replace('-collection', '_ids')` тАФ pin it).
- (b) Defer to Part 9's `event_overrides` (authors add their own `references.{entity}_ids`). This pushes work onto every author who wants their workflow events on entity timelines тАФ undesirable.
- (c) Spec-level deviation: pick (a) and update [submit-pipeline/spec.md ┬з Default log event](../../../workflows-module-concept/submit-pipeline/spec.md) to match. The spec template is incomplete, not deliberately exclusive.

Pick (c). Otherwise the worked-example "Submitting `qualify` writes an event in `events` with the expected display, references" line in ┬з Verification will pass mechanically but the event won't show on the lead's timeline.

## Scope and ordering

> **Resolved.** Picked (a): dropped the ┬з Cancel-workflow event sub-section entirely. **Fix.** Either:

- (a) Drop the cancel-workflow-event sub-section from Part 8 entirely. Carry it as a concept open question (or a Part 23 successor) since it spans `CancelWorkflow` ownership ([CancelWorkflow.js](../../../../plugins/modules-mongodb-plugins/src/connections/WorkflowAPI/CancelWorkflow/CancelWorkflow.js), Part 5) and is logically separate from the submit-pipeline side effects this part owns.

Scoping a `CancelWorkflow` side-effect into a part whose repo path is the submit-action subdirectory is a code-organization smell.

### 7. Notifications dispatch should also pass `references` for the `send_routine`'s benefit

> **Resolved.** ┬з `dispatchNotifications.js` now states that `event_ids` is the only field and the `send_routine` re-fetches the event doc for `references` and `metadata` тАФ preventing implementers from "helping" by passing extras.

┬з `dispatchNotifications.js`: *"Fires `context.callApi({ id: 'send-notification', module: 'notifications' }, ...)` with `{ event_ids: [event_id] }`."*

This matches the spec exactly. But per [submit-pipeline/design.md:413](../../../workflows-module-concept/submit-pipeline/design.md): *"that routine reads the event doc by id, resolves recipients (typically from `event.references` or the action's role declarations), and dispatches via whatever channels the app wires"*.

So the routine **does** re-fetch the event doc; passing only `event_ids` is correct. Confirming this passes muster тАФ no change needed, but flag for clarity in the design body: a one-liner *"`event_ids` is the only field; `send_routine` re-fetches the event doc to read `references` and `metadata`"* prevents the implementer from adding extra fields "to help" and breaking the spec.

### 8. Ordering of side effects vs. step 4-6's in-memory mutations

> **Resolved.** ┬з Lifecycle integration now says to capture the log-event input bag at step 1, after validation resolves `context.action`/`context.workflow`/`targetStatus` and before step 4 mutates `context.workflowActions`.

┬з Lifecycle integration says: *"Step 7 must complete before step 8"*. Correct.

Not stated: the **inputs** Part 8 reads (`workflow_id`, `action_id`, `status_before`, `status_after`, `interaction`, `current_key`, `action_type`, `workflow_type`) must be captured **before step 4 runs** тАФ because step 4 mutates `context.workflowActions` in-memory ([handleSubmit.js:209-216](../../../../plugins/modules-mongodb-plugins/src/connections/WorkflowAPI/SubmitWorkflowAction/handleSubmit.js)). For `status_after` this doesn't matter (it's the resolved target). For `status_before` it does (see finding #3).

**Fix.** Add to ┬з Lifecycle integration:

> Capture the log-event input bag at step 1 (after the validation block resolves `context.action`, `context.workflow`, `targetStatus`), before step 4 mutates `context.workflowActions` in-memory. Pass the captured bag into `dispatchLogEvent` at step 7.

### 9. ┬з Contract to neighbours hands Part 9 a "pure function" hook but the design's prose doesn't define it

> **Resolved.** ┬з `dispatchLogEvent.js` is now split into `buildDefaultLogEventPayload` (pure function with the named arg shape) and `dispatchLogEvent` (wraps + dispatches). Both names pinned in the body.

Last sentence of ┬з Contract to neighbours: *"Make the default-payload assembly its own pure function so part 9 can call it before merging overrides."*

This is the right design intent, but the body of the in-scope section above (┬з `dispatchLogEvent.js`) makes no mention of the function тАФ it describes the dispatch behaviour but not the assembly/dispatch split. Part 9 needs the assembly as a standalone import.

**Fix.** Restructure ┬з `dispatchLogEvent.js`:

- `buildDefaultLogEventPayload({ workflow, action, actionConfig, interaction, current_key, status_before, status_after, user, eventId })` тАФ pure function returning `{ type, display, references, metadata }`. Tested without `context.callApi`. Part 9 imports it and layers `event_overrides`.
- `dispatchLogEvent(context, { interaction, status_before, status_after })` тАФ wraps the pure function, calls `context.callApi('new-event', module: 'events', payload, { user })`, returns the dispatched payload's `event_id` for confirmation (which should equal `context.eventId` per finding #1).

Pin function names; Part 9 design already references "the default-payload function" generically.
