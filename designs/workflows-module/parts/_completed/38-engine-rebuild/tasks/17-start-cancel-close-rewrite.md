# Task 17: Rewrite Start / Cancel / Close around phases

## Context

The other three engine entry points restructure into the same load-plan-commit shape as Submit (task 15), with different planners and commit batches but the same helpers. They have **no pre-hook in v1**. Critically, this rebuild extends event emission so **every** engine handler invocation produces exactly one `event_id` and a lifecycle entry in the events timeline (today only Submit dispatches events). These handlers compose the phases (tasks 9–13) and feed `trackerFires` into the cascade (task 16). They can be built in parallel with task 15.

## Task

**`WorkflowAPI/StartWorkflow/StartWorkflow.js`** — restructure:

- Load: `workflowConfig` + parent action (if started as a tracker child). **Not `loadWorkflowState`** — its `{ workflowId }` mode throws `workflow_not_found` and Start has no workflow yet; Start performs its own reads: config lookup via `workflowsConfig.find(type)` (in-memory) + the optional parent-action read by `_id` via the task-1 `findDocs` helper.
- Plan: workflow doc (`Plan.workflow.operation: "insert"` — D3/task 13: commit step 1 dispatches to `insertOneDoc`, no CAS filter) + initial action docs **seeded directly at the declared status** — `starting_actions` / payload `actions:` entries carry the `{ type, key?, status }` grammar (legal seeds: `action-required`, `blocked`; `key` is payload-only — the keyed-action guard below directs keyed seeds to the `actions:` payload, and the landed planner already accepts `key`). Drafts are built by **`planActionTransition` in a new `seedStage` mode**: an optional input, mutually exclusive with `signal`, insert-only (no `action`; bypasses the `upsert` gate, which guards the signal path) — it skips `resolveSignal`, takes the declared status as the target stage, and runs every downstream step unchanged (full draft shape, `access`/`workflow_type` denormalisation, `status_map` render at the seed stage, `computeEngineLinks`, change-log `{ before: null, after: doc }` — `planActionTransition.js:138–205`). Do **not** hand-build the drafts like today's `createAction` — one composition site. The planner stays generic (no legal-seed check inside it); enforcement is build-time + runtime per the payload bullet below. Creation at workflow start is not an FSM transition; the `none` row is the pre-hook spawn path only (Part 45 review 1 #2; state-machine.md "Creation"). The planned workflow doc carries **`entity_ref_key`** from the workflow config alongside `entity_collection` (same copy-onto-doc mechanic) — `planEventDispatch` reads it for the event's entity reference key (task 12; design "Event references"). When started as a tracker child, the planned workflow doc also carries **`parent_workflow_id`** beside `parent_action_id`, stamped from the loaded parent action's `workflow_id` (schema addition; design "Schema additions" — it's what makes downstream tracker-mirror fires purely derivable at plan time). The workflow doc's derived fields (`summary`, `groups[]`) come from **`planWorkflowRecompute`**, the design's single workflow-doc composition site — do **not** hand-build them like today's `StartWorkflow.js:97–108`: compose the base insert doc (status seeded `active`, `form_data: {}`), then run `planWorkflowRecompute({ loadedState: { workflow: baseDoc, workflowConfig }, plannedActions: seededDrafts, event_id, now })` — no `lifecyclePush`; auto-complete can't fire (legal seeds are non-terminal, and the `total > 0` guard covers the zero-action workflow), and the planner stamps `updated: now` consistently. Today's `notRequiredCount` logic dies naturally (under the legal-seed rule summary is always `{ done: 0, not_required: 0, total: N }` — the planner computes it without Start knowing that). The parent-tracker transition is **not** in Start's plan — the parent action belongs to a different workflow, and the Plan is per-aggregate (D3/D10); it runs as a cascade level via the `internal_mirror_child_active` fire (see "Tracker fires" below). Event = `workflow-started` (workflow-lifecycle context).
- Commit through `commitPlan` like every other handler; when started as a tracker child, the parent-tracker mirror runs through `runTrackerCascade` (the fire below) like every other cross-workflow push.
- `start-workflow.yaml` payload gains `metadata` (handled in task 19), and **StartWorkflow consumes it**: every seed-mode `planActionTransition` call receives `payload.metadata` from `params.metadata`, so the start-payload bag merges onto every seeded draft's `metadata` (the landed planner already does the merge — insert-side `planActionTransition.js:166`, the path every seed takes — this is pure threading). Part 30 carry-over semantics ("Proposed change" #5 / D10); review-16 #4 names this task as the consumer. The `actions:` override stays on `{ type, key?, status }` — no signal grammar at start. Per-entry `fields` and `references` from today's entry shape (`createAction.js`) are **dropped** — nothing in-repo passes them ("build for what exists"); the workflow-level `references` and `metadata` payload channels carry over unchanged. **StartWorkflow enforces the legal-seed rule at runtime**: any `actions:` entry (and any `starting_actions` entry, defense-in-depth) with a status other than `action-required` | `blocked` throws `WorkflowEngineError` — build validation can't see payloads, so the runtime check is what makes the rule real for the override path. **Migration:** existing configs seeding other stages (e.g. a `starting_actions` entry at `in-review`) must re-author to a legal seed — the build-time check makes the break loud, not silent.

**`WorkflowAPI/CancelWorkflow/CancelWorkflow.js`** — restructure:

- Load: workflow + all actions via `loadWorkflowState` `{ workflowId }` mode. This throws `workflow_not_found` where today's Cancel proceeds silently against a missing workflow (`CancelWorkflow.js:35–41` has no null check) — an intended tightening, the one deliberate exception to the "no new guards" rule below.
- Plan: mark all **non-terminal** actions `not-required` via FSM signal `internal_cancel_action` — Cancel's sweep is **unconditional** (today's behaviour: `CancelWorkflow.js:71–96` has no `required_after_close` filter); recompute via `planWorkflowRecompute` with **`lifecyclePush: { stage: "cancelled", reason: payload.reason }`** (see "Lifecycle status entries" below); event = `workflow-cancelled` (workflow-lifecycle context). **Done actions are preserved** (their status stays `done`).
- Commit; tracker cascade.

**`WorkflowAPI/CloseWorkflow/CloseWorkflow.js`** — same shape as Cancel, with two deltas; event = `workflow-closed`:

- **Close pushes `completed`**, not `closed` — `closed` is not a workflow stage (`CloseWorkflow.js:78–82`); every `status.0.stage === 'completed'` consumer (the D2 load gate, the recompute guard, `get-entity-workflows`) depends on it.
- **Close's sweep keeps today's `required_after_close` exception** (`CloseWorkflow.js:66–71, 116–121`): sweep only non-terminal actions where `required_after_close !== true` **or** currently `blocked` (the blocked-action exception). Survivors stay at their stage — this is what keeps the post-close submit carve-out (D2 / task 9 / review-4 #3) reachable: the form FSM's `not-required` row is empty, so a swept action could never move again.

**Lifecycle status entries — `lifecyclePush`:** Cancel/Close must not let the recompute's auto-complete fire: the sweep makes every action terminal while the loaded stage is still `active`, so the recompute's auto-complete branch (`planWorkflowRecompute.js:93–107`) would push a phantom `completed` under Cancel's `cancelled`, and a duplicate `completed` under Close's. The landed `planWorkflowRecompute` takes an optional **`lifecyclePush: { stage, reason }`** input (implemented by task 23 — don't re-extend): when present, the auto-complete check is **skipped entirely** and the planner pushes the declared entry instead — `{ stage, event_id, created: now, ...(reason ? { reason } : {}) }` — composed in the planner, the single status-entry composition site (`payload.reason` carries over from today, `CancelWorkflow.js:49–53`). Cancel passes `{ stage: "cancelled", reason }`; Close passes `{ stage: "completed", reason }` — the skip-entirely semantics is also why Close can't delegate to the auto-complete: a `required_after_close` survivor keeps the action set non-terminal, but Close pushes `completed` regardless. Submit and tracker levels pass no `lifecyclePush` — auto-complete behaviour unchanged. Lifecycle entries thus carry the invocation `event_id` (the landed recompute already stamps it on its auto `completed` push; today's lifecycle entries carry none). Start seeds the workflow status at **`active`** (`StartWorkflow.js:82`), same `event_id` stamp, and passes no `lifecyclePush` either (Start inserts; it runs the recompute against the planned insert doc, not loaded state — see the plan bullet).

**Lifecycle preconditions live here, not in `loadWorkflowState`** (task 9's stage check is Submit-specific). Preserve today's actual semantics, no new guards: Close on a `completed` workflow is an **idempotent no-op** (returns the empty result, `CloseWorkflow.js:52–54`); Close on a `cancelled` workflow **throws** — now `WorkflowEngineError` with `code: "stage_rejects_close"` (D13). Cancel deliberately has **no stage guard** today (cancelling a completed workflow is unguarded) — keep it that way per "build for what exists". Start inserts a fresh workflow doc, so it has no started-already check; its config-shaped preconditions (unknown `workflow_type`, keyed `starting_actions`, tracker-parent checks) carry over from the current handler.

**Payload and return surfaces:** `payload.references` carries over — Start spreads it into the planned workflow doc at insert (`StartWorkflow.js:74–75`); Cancel/Close merge it into the planned workflow doc minus `RESERVED_WORKFLOW_KEYS` (`CancelWorkflow.js:5–18, 44–47` — same reserved-key filter, now applied at plan time so the planned whole-doc `$set` carries it). Returns, pinned per handler: Start returns `{ workflow_id, action_ids, event_id }` — `event_id` is **new** return surface, uniform now that Start mints a real event (task 19 extends `start-workflow.yaml`'s `:return`); Cancel/Close return `{ action_ids, event_id, tracker_fired }` with `event_id` now real (today hardcoded `null`) and `tracker_fired` from `runTrackerCascade`'s `fires` in today's shape (task 16; review-11 #2's lifecycle half).

**Post-commit dispatch failures — `throwIfDispatchFailed`:** D9/D13 require **every** handler to end its invocation by throwing `WorkflowEngineError` `code: "post_commit_dispatch_failed"` when commit steps 3–5 recorded `dispatchErrors[]` or the cascade recorded `cascadeErrors[]` — `commitPlan` and `runTrackerCascade` only _record_ these; the end-of-handler throw is the sole surfacing (committed state is durable; no engine side-channel logging). Task 15 implemented this inline for Submit (`handleSubmit.js:88–102`). Extract that block to a shared **`shared/phases/throwIfDispatchFailed.js`** (handler name parameterised into the message) and call it at the very end of all four handlers — after the cascade, before the success return; `handleSubmit` adopts the helper too ("one correct way"). Close's idempotent no-op returns the empty result before commit, so it never reaches the throw; a handler invocation with no cascade (Start not as tracker child, no fires) passes empty `cascadeErrors`.

**Event emission (per the "Engine entry points emit events" table):** one `event_id` per invocation, used as the dispatched event doc's `_id`. Workflow-lifecycle context only (`{ user, workflow, signal }`) — `planEventDispatch` already branches on type (task 12).

**Tracker fires follow the D3 producer rule** (same as task 15's `planSubmit` step — composed purely from ids in hand, no cross-workflow read): Start emits `internal_mirror_child_active` with ids from the loaded parent action (`payload.parent_action_id` + that action's `workflow_id`), plus **`payload: { fields: { child_workflow_id, child_entity_id, child_entity_collection } }`** — the parent↔child link fields today's handler sets directly (`StartWorkflow.js:120–125`); `planTrackerLevel` forwards `fire.payload` into `planActionTransition`'s `payload.fields` (task 16; D3 fire shape). Two deliberate behaviour deltas vs today's silent direct push: the parent's timeline gains an `action-internal-mirror-active` event (every cascade level emits one), and the parent's `groups[]`/`summary` recompute (today's `updateAction` push leaves them stale). Cancel emits `internal_mirror_child_cancelled` (parent tracker → `not-required`) and Close emits `internal_mirror_child_completed` (parent tracker → `done` — close is forced completion, the child's status reads `completed`; today's `CHILD_STAGE_MAP` behaviour), both iff the loaded workflow has `parent_action_id != null`, ids read off the loaded workflow doc (`parent_action_id` + `parent_workflow_id`). Close's idempotent no-op (already `completed`) returns before planning, so it fires nothing.

## Acceptance Criteria

- Start/Cancel/Close each run load → plan → commit (→ tracker cascade) with no mutable shared `context`.
- Each emits exactly one lifecycle event (`workflow-started` / `workflow-cancelled` / `workflow-closed`) with the workflow-lifecycle render context — except Close's idempotent no-op (already `completed`), which returns the empty result without an event.
- Cancel sweeps all non-terminal actions to `not-required` via `internal_cancel_action` (unconditional); Close sweeps with the `required_after_close` exception (non-`blocked` survivors keep their stage); **`done` actions are preserved**.
- Test: close a workflow with a non-terminal `required_after_close` action → the action survives at its stage → a post-close submit on it succeeds (the D2 carve-out stays reachable).
- Test: cancel an active workflow → status history gains exactly **one** new entry (`cancelled`, carrying the invocation `event_id`) — no phantom `completed` from the recompute's auto-complete; the parent tracker (if any) lands `not-required`.
- Test: start as a tracker child → the parent tracker lands `in-progress` **with the child link fields set** (`child_workflow_id`, `child_entity_id`, `child_entity_collection`); the parent's timeline gains an `action-internal-mirror-active` event and its `groups[]`/`summary` are recomputed (both deliberate deltas vs today's silent stale push).
- Every handler ends with the shared `throwIfDispatchFailed`: recorded `dispatchErrors[]`/`cascadeErrors[]` surface as a `post_commit_dispatch_failed` throw after the cascade (D9/D13); Close's idempotent no-op returns before any of this. Test: a failing event dispatch on Cancel throws `post_commit_dispatch_failed` while the swept docs stay committed.
- Per Q5: **workflow-level event only** for Cancel/Close (no per-action `action-internal-cancel-action` events) in v1 — the change-log captures per-action mechanics for forensic audit.
- Integration tests `StartWorkflow.test.js`, `CancelWorkflow.test.js`, `CloseWorkflow.test.js` pass, including: Start emits `workflow-started`; Cancel preserves `done` actions and pushes `cancelled`; the change-log records per-action `not-required` transitions.

## Implementation deviations (as landed)

Error-code vocabulary only — behaviour matches the task. Start's config-shaped preconditions threw codes the design's D13 list didn't yet name, resolved post-review (D13 now carries all three):

Initially also covered missing-required-param throws — those were split out post-review.
- `invalid_params` — new code for a missing required request param (Start's `workflow_type`/`entity_id`/`entity_collection`; Cancel/Close's `workflow_id`), so "bad request shape" is distinguishable from "bad seed grammar".
