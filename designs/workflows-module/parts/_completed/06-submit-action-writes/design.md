# Part 06 — `SubmitWorkflowAction` core writes

**Source rationale:** [workflows-module-concept/engine/spec.md](../../../workflows-module-concept/engine/spec.md), [workflows-module-concept/submit-pipeline/spec.md](../../../workflows-module-concept/submit-pipeline/spec.md). **Layer:** engine handlers. **Size:** L. **Repo:** `plugins/modules-mongodb-plugins/src/connections/WorkflowAPI/SubmitWorkflowAction/`.

## Goal

Ship the bones of the user-submit path: a `SubmitWorkflowAction` handler that takes an `interaction`, applies the priority rule, writes the action transition and `form_data`, recomputes the workflow summary, and returns. The 11-step lifecycle skeleton lands here with all 11 steps stubbed; only the writes-and-priority slice executes. Parts 7–11 light up the other steps.

## In scope

### Payload

The per-action endpoint ([part 13](../13-resolver-apis/design.md)) hands `SubmitWorkflowAction` a flat per-endpoint payload — what the user posted plus the build-time-baked endpoint config:

```
{
  action_id, interaction, current_key,
  form, form_review, fields,
  current_status,                        // task submit_edit only — status-selector pattern
  hooks, event_overrides,                // baked-in config maps; consumed in parts 8, 9
}
```

Step 1 of the lifecycle (Validate) translates this into the engine's internal payload shape — what every sub-module (`computeAutoUnblocks`, `updateAction`, `dispatchLogEvent`, …) reads from for the rest of the call:

```
{
  currentActionId: string,
  actions: [
    { type, status, keys?, fields?, references?, force? }
                                         // force: optional per-entry bypass of the priority rule
                                         //   for that entry only. The only force surface — there
                                         //   is no top-level force on the handler payload (see
                                         //   Priority rule below).
  ],
  eventId: string,                       // generated by handleSubmit on entry; threaded into every
                                         //   write in this invocation.
}
```

The translation in step 1 looks the action up (by `action_id`), resolves the interaction to a target status (see [Interaction → target-status mapping](#interaction--target-status-mapping-engine-default-only) below), and constructs a single-entry `actions[]` with the `currentActionId` slot populated. The internal shape is a superset of what v1 uses — `actions[]` is an array because parts 7 and 9 plumb additional entries (auto-unblocks; pre-hook-returned actions) through the same loop without re-shaping the seam.

**v1 fan-out posture.** The per-entry write loop ships here; only the `currentActionId` slot is populated. Auto-unblock entries land in [part 7](../07-group-state-machine/design.md); pre-hook-returned `actions[]` entries land in [part 9](../09-hook-invocation/design.md). Both pour into the same internal `actions[]` array before the loop runs.

**`keys: []` silent no-op.** Per the engine spec ([§ SubmitWorkflowAction payload](../../../workflows-module-concept/engine/spec.md#submitworkflowaction-payload)), a payload-supplied entry with `keys: []` writes zero ops — documentation-only mitigation in v1, no validator. Callers gate with `_if` / `skip` on `keys.length`.

### Lifecycle scaffold

Full 11-step skeleton in `handleSubmit.js`; only steps 1, 3 (auto-unblocks for action types only — group ids defer to part 7), 4, 5, 6 execute. Other steps no-op with TODO comments pointing at their part. Numbering follows [submit-pipeline/spec.md § Flow](../../../workflows-module-concept/submit-pipeline/spec.md#flow) (the canonical user-facing lifecycle view), not [engine/spec.md § Ordering inside one `SubmitWorkflowAction` invocation](../../../workflows-module-concept/engine/spec.md#ordering-inside-one-submitworkflowaction-invocation) (a different write-ordering view of the same flow):

1. **Validate**: payload schema, action exists, access role gate (`access.roles ∩ user.roles`), and terminal-workflow gate (reject when `workflow.status[0].stage ∈ {completed, cancelled}` AND `!action.required_after_close`, per [action-authoring/spec.md § `required_after_close`](../../../workflows-module-concept/action-authoring/spec.md)). Verb filter is implicit at submit time — the page wouldn't have been generated if the verb wasn't allowed in the calling app (see [engine/spec.md § Capabilities](../../../workflows-module-concept/engine/spec.md#capabilities)).

   **Failure shape.** Two paths per [engine/spec.md § Action `error` transition](../../../workflows-module-concept/engine/spec.md#action-error-transition):

   - **Before action lookup** (payload schema, action-not-found, role gate, terminal-workflow gate): no action doc to attach error context to → throw with a structured error. Caller sees it through the CallApi result. Matches `StartWorkflow`'s posture ([StartWorkflow.js:19](../../../../plugins/modules-mongodb-plugins/src/connections/WorkflowAPI/StartWorkflow/StartWorkflow.js)).
   - **After action lookup, mid-write** (sub-step failure inside steps 4–6): force-push `{ stage: error, created, reason: <step-name>, error_message, error_metadata }` onto the action's `status[]` via `updateAction(...force: true)`; skip the remaining lifecycle steps; return partial `{ action_ids, event_id, ... }`. Pre-hook `hook_error` returns ([part 9](../09-hook-invocation/design.md#pre-hook-return-merge)) take the same path with `reason: 'pre-hook'`.
2. **Pre-hook** — no-op, → [part 9](../09-hook-invocation/design.md).
3. **Compute auto-unblocks** — action-type entries in `blocked_by` only. Group ids resolved in [part 7](../07-group-state-machine/design.md).
4. **Write action transitions** — per-entry loop over `actions[]`; each entry's write applies the priority rule + `currentActionId` self-exception + per-entry `force: true` bypass. Push to `status[]` with change stamps.
5. **Recompute workflow summary** — counts only. `groups[]` defer to part 7.
6. **Write `form_data`** — merge `form` and `form_review` from the endpoint payload into one flat bag (per [engine/spec.md § Form data layout](../../../workflows-module-concept/engine/spec.md#form-data-layout) — no `.review` sub-key), then `$set` per-field at `form_data.{action_type}[.{key}].{field}`. Field collisions are author error; engine doesn't disambiguate. Part 9's `form_overrides` merge layers on top of this in the same write.
7. **Generate log event** — no-op, → [part 8](../08-side-effect-dispatch/design.md).
8. **Dispatch notifications** — no-op, → [part 8](../08-side-effect-dispatch/design.md).
9. **Group `on_complete` fan-out** — no-op, → [part 11](../11-group-on-complete-fanout/design.md).
10. **Tracker subscription** — no-op, → [part 10](../10-tracker-subscription/design.md).
11. **Post-hook** — no-op, → [part 9](../09-hook-invocation/design.md).

Then return: `{ action_ids, completed_groups: [], event_id: null, tracker_fired: null, pre_hook_response: null, post_hook_response: null }`. `completed_groups: []` is a literal placeholder in v1; [part 7](../07-group-state-machine/design.md#completed_groups-return-shape) swaps it for the real `[{ workflow_id, id, on_complete? }]` entries computed during step 5's group recompute.

### Interaction → target-status mapping (engine default only)

Applies to the `currentActionId` entry only — step 1's translation uses it to derive that entry's `status`. Auto-unblock entries (computed in step 3) carry `status: action-required`; pre-hook-returned `actions[]` entries (part 9) carry the author-supplied `status`. Neither passes through this table.

No `action.interactions:` override (part 9), no pre-hook override (part 9). Defaults:

- `submit_edit` → `in-review` if any `access.{app}` has `review`, else `done`.
- `not_required` → `not-required`.
- `resolve_error` → same target as `submit_edit`.
- `approve` → `done`.
- `request_changes` → `changes-required`.
- Task `submit_edit` honors caller-supplied `current_status` instead of computing (status-selector pattern).

### Priority rule

- Allow transition iff `priority(new) < priority(current)`.
- Self-exception: same-stage allowed for the `currentActionId`. A re-click writes a fresh status entry — audit history is the source of truth for "user did this again." Idempotency on non-self entries falls out of the priority rule (priority(X) < priority(X) is false).
- `force: true` per-entry (in `actions[]`) bypasses the rule for that entry only. **Per-entry is the only force surface** — no top-level `force` on the handler payload. v1 user is the pre-hook path (part 9), which sets `force` on individual entries it returns. Engine-internal force-pushes split by mechanism (per [engine spec § Priority rule](../../../workflows-module-concept/engine/spec.md#priority-rule)): per-doc force via `updateAction(...force: true)` covers the `error` transition, tracker subscription's parent push, and `StartWorkflow`'s parent-link push; bulk bypass via `MongoDBUpdateMany` covers `CancelWorkflow`'s sweep and [part 23](../23-close-workflow-handler/design.md)'s `CloseWorkflow` sweep. Neither path reconstructs a handler payload.
- Priority is read from `connection.actionsEnum[stage].priority` at handler entry. A resolved target status not present in `actionsEnum` throws — guards against typos in `action.interactions[*].status` (part 9) or pre-hook returns that resolved to unknown stages.
- `not-required` (priority 0) is universal terminal.

### Idempotency

Falls out of the priority rule + `currentActionId` self-exception. A retried submit issues the same writes; the priority check rejects status pushes that already landed on **non-self** entries, and the self-exception allows the no-op same-stage push on the user-submitted action (writing a fresh audit entry — see Priority rule).

- `utils/shouldUpdate.js` wraps the priority check for the per-entry write loop.
- `utils/getCurrentAction.js` fetches the action doc once per entry so the priority comparison reads consistent state.

### Sub-modules

- `SubmitWorkflowAction/handleSubmit.js` — orchestrator.
- `SubmitWorkflowAction/computeAutoUnblocks.js` — action-type-only resolution; commented for extension by part 7.

## Verification

- Unit tests on `handleSubmit`:
  - Priority rule honored; same-stage on `currentActionId` allowed (writes fresh audit entry); same-stage on others rejected.
  - Per-entry `force: true` bypasses the priority rule for that entry only.
  - `form_data` writes land at the right path for keyed and non-keyed actions; `form` and `form_review` merge into one flat bag before write.
  - Summary counts match after a transition.
  - Each interaction default maps correctly (form action with `review` verb → `in-review`; without → `done`; etc.).
  - Task `submit_edit` honors caller-supplied `current_status`.
  - Terminal-workflow gate rejects when workflow is `completed`/`cancelled` and `!action.required_after_close`.
- Idempotency: a retried submit no-ops on non-self entries (priority rule rejects same-stage) and writes a fresh audit entry on the `currentActionId` self-exception.
- End-to-end coverage lands in [part 22](../22-workflows-e2e-suite/design.md). This part's verification is unit-tests + handler-level integration smoke only.
