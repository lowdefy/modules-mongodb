# Part 50 ‚Äî Collapse to one config-free entity events timeline

> **Supersedes [Part 46](designs/workflows-module/parts/_completed/46-debundle-workflow-config/design.md) D6.** Part 46 D6 _split_ the entity timeline into two components ‚Äî an events-only generic timeline (events module) and an action-enriched timeline (`workflows-events-timeline`, on the `WorkflowAPI` connection) ‚Äî to avoid an events ‚áÑ workflows module cycle. This part **re-merges them into one** timeline that the events module owns and that enriches with action cards when an `actions` collection is configured. Two facts that did not hold when D6 was written make this safe (see D1): the action-card access policy is **data-on-the-action + session roles, no workflow config**, and the events module **already depends on the `@lowdefy/modules-mongodb-plugins` package**, so the one shared policy implementation is reachable without a module cycle. The earlier Part 50 design (a replaceable `components.events_tile` slot that let an app _swap_ the events-only tile for the workflows one) is **dropped** ‚Äî with one timeline there is nothing to swap.

Entity modules (companies, contacts, activities) each render an events-timeline "History" tile. Today that tile is events-only; an app that runs workflows on the entity gets a _second_, action-enriched timeline component and has to swap one for the other per entity. This part makes the single events timeline **enrich itself**: it joins each event's referenced actions and renders their cards inline, gated only by the session user's roles against access data already denormalised onto each action. Enrichment is **data-driven** ‚Äî wherever an app's events reference actions, _every_ entity timeline shows their cards; a pure-CRM app whose events reference no actions renders identically to the events-only timeline of today, with no skip branches and no second component (D4). The events module entry carries **one `actions_collection` var** (plus `contacts_collection`) that points the engine at the app's collections, defaulting to `actions` / `user-contacts`. No per-entity wiring, no second component, no slot.

**Layer:** plugin engine (config-free + relocatable) + events-module connection/manifest/timeline + action write-path denormalisation + workflows-module cleanup + docs + demo adoption. **Size:** M. **Repo:** `plugins/modules-mongodb-plugins/`, `modules/{events,workflows}/`, `docs/`, `apps/demo/`.

## Proposed change

1. **The action sort key is denormalised onto the action doc at write time.** Today `makeWorkflowOrderComparator` (`compareActionOrder.js`) is the _only_ consumer of `workflowsConfig` in the timeline read ‚Äî it maps each action to its declaration position (group index in `cfg.action_groups`, action index in `cfg.actions`). The two declaration indices are computed **once at build time** by the `makeWorkflowsConfig` resolver (`modules/workflows/resolvers/makeWorkflowsConfig.js`) ‚Äî it already walks every group and action to normalise them, so in the same pass it attaches each action's **group index** (position in `action_groups[]`) and **action index** (position in `actions[]`) onto that action's config entry. These indices are pure functions of the authored config, so build time is their natural home ‚Äî nothing about a live action instance affects them. `planActionTransition.js` already writes `doc.access`, `doc.workflow_type`, `doc.title`, `doc.tracker` onto the action on every plan (`:209`); it now copies `actionConfig.group_index` / `actionConfig.decl_index` onto the doc in the same block ‚Äî a plain copy beside `doc.access`, no signature change and no per-transition computation. The comparator reads the stored indices instead of resolving config; the rest of its sort key ‚Äî `[group_index, notRequired, decl_index, key, _id]` ‚Äî is unchanged, because only the two indices were ever config-derived: `notRequired` still reads off `status`, and the `key` / `_id` tiebreakers read straight off the action doc. In the timeline these survive to the in-memory sort: the actions `$lookup` sub-pipeline rewrites `status` to a scalar and applies **no narrowing `$project`**, so `status`, `key`, `_id`, and the new `group_index` / `decl_index` all pass through. **Result: the timeline read needs no `workflowsConfig`.**

   The comparator is _not_ timeline-local: `compareActionOrder.js` is the single source of truth for action display order across **all four** read engines (`GetEventsTimeline`, `GetEntityWorkflows`, `GetWorkflowOverview`, `GetWorkflowActionGroupOverview` ‚Äî its own header, Part 54). So this change is **global** ‚Äî all four engines move to doc-stamped ordering, and the three overview engines drop the now-unused `workflowsConfig` argument at their `makeWorkflowOrderComparator(...)` call. Keeping config-based ordering for the three config-holding engines while forking a doc-based path for the timeline is rejected: it would produce two ordering implementations and break the Part 54 single-source-of-truth. The staleness consequence of the global change is covered in D5.

2. **The timeline engine becomes config-free and moves to a new events-owned connection type.** `GetEventsTimeline` (`plugins/.../WorkflowAPI/GetEventsTimeline/`) drops its `workflowsConfig` dependency (change 1). "Relocating" it is not an in-place edit: it currently lives on the `WorkflowAPI` connection, whose schema requires workflows-only fields (`entry_id`, `endpoints`), and the events module's only connection is a generic `MongoDBCollection` (`events-collection`) that cannot host a plugin-defined request type. So this part **introduces a new plugin connection type, `EventsTimeline`** (parallel to `WorkflowAPI`; registered in `plugins/.../src/connections.js` alongside it), which exposes the `GetEventsTimeline` request. It is read-only (`meta.checkRead`/`checkWrite: false` ‚Äî no change-stamp/write fields), and its schema is fully determined by what the engine reads off its connection:
   - `databaseUri` (secret) ‚Äî Mongo access;
   - `app_name` ‚Äî the display/access namespace key (the events module wires its existing `display_key` var here ‚Äî see D3);
   - `eventsCollection` ‚Äî default `log-events`;
   - `actionsCollection` ‚Äî wired from the `actions_collection` var; defaults to `actions` (the `?? 'actions'` engine fallback stays ‚Äî see D4);
   - `contactsCollection` ‚Äî wired from the `contacts_collection` var; defaults to `user-contacts`;
   - `user` ‚Äî via `_user: true` (session user, for role-gating).

   The events module adds `EventsTimeline` as a **second** connection; `events-collection` (`MongoDBCollection`) stays put for the event writes and change-log. The relocation has two concrete steps: (a) `GetEventsTimeline` is **removed from `WorkflowAPI.js`'s `requests` map** and registered only on `EventsTimeline` ‚Äî one request type binds to one connection type, never both; and (b) the engine **directory moves** from `.../connections/WorkflowAPI/GetEventsTimeline/` to `.../connections/EventsTimeline/GetEventsTimeline/` (config-free, it no longer belongs under the workflows connection; its `../../shared/render/...` imports sit at the same depth under either parent, so the move is path-neutral).

3. **The events module gains `actions_collection` (and `contacts_collection`) vars** ‚Äî collection-name overrides, each defaulting to the engine's existing fallback (`actions` / `user-contacts`). The engine keeps **both `$lookup` stages unconditional ‚Äî no skip branches** (D4): the actions `$lookup` keys on `localField: action_ids`, so for a CRM app whose events carry no `action_ids` it matches nothing and returns `actions: []` (the dedup stages then no-op on the empty array, and the block's `event.actions.length > 0` guard renders no cards) ‚Äî the stage is inert without action data, so there is nothing to skip. The contacts `$lookup` keys on `created.user.id` (always present), so it does run, but degrades gracefully (`$ifNull` ‚Üí initials) when unmatched or when the collection is absent. The events timeline request routes through the engine method on the events connection rather than the inline `MongoDBAggregation` it uses today.
4. **The action access policy is data-driven and lives in one place.** `computeAllowed` reads `action.access[display_key].{view,edit,review,error}` (gate values written by the engine at transition time) and matches them against the session user's roles via `gateAllows`; `collapseLink` picks the highest-priority accessible link from `action[display_key].links`. Neither touches config. This is the single implementation Part 46 D2 consolidated ‚Äî preserved, just reached through the events timeline.
5. **The workflows module's timeline duplication is deleted**: `modules/workflows/components/workflows-events-timeline.yaml`, its `requests/get_events_timeline.yaml`, and the `WorkflowAPI`-bound `GetEventsTimeline` request wiring. The demo `lead-view` (the one direct consumer) reads the events timeline instead. The check-action click handler / modal wiring (Part 55) moves to the events timeline component as the `onActionClick` hook, since that is now the only timeline.
6. **The events module's dead action-card _display_ config is deleted** (carried over from the original Part 50 scope): drop the `action_statuses_display` merge term from the `actionStatusConfig` prop wiring (`events-timeline.yaml:88‚Äì91`) ‚Äî **keeping the `../shared/enums/action_statuses.yaml` enum ref**, which becomes the unified timeline's status-config source ‚Äî plus the `action_statuses_display` manifest var and its README section. The `EventsTimeline` _block_ keeps its `actionStatusConfig` prop, and the block gates all card rendering on that prop being truthy (`EventsTimeline.js:525`), so the enum ref must survive or the unified timeline renders zero cards even with enrichment on.
7. **The demo turns enrichment on** (`apps/demo/modules/events/vars.yaml`): set `actions_collection: actions` (and `contacts_collection`) on the events module entry. The demo's `company-setup` and `onboarding` workflows already target `companies-collection` / `leads-collection`, so action cards appear on the company and lead timelines with no per-entity config ‚Äî exercising the whole path through one var.

## Key decisions

### D1 ‚Äî Why collapsing is now safe (and was not when Part 46 D6 split them)

Part 46 D6 split the timeline for **one reason ‚Äî a module-dependency cycle**, in its words:

> The events module is **foundational ‚Äî it declares no dependencies**. The naive port (have the events timeline call into the workflows plugin) would make **events ‚Üí workflows** ‚Ä¶ a **workflows ‚áÑ events cycle** ‚Ä¶ The fix is to **invert ownership**: workflows exposes `GetEventsTimeline` ‚Ä¶ reading events itself.

Two facts dissolve that constraint:

- **The cycle was about a _module_ dependency; the policy lives in the _plugin package_.** `@lowdefy/modules-mongodb-plugins` is the shared layer _below_ both modules, and the events module **already declares it** (`module.lowdefy.yaml:87‚Äì89`). Events reaching the enrichment engine is `events ‚Üí plugin`, not `events ‚Üí workflows`. **No cycle.** D6 could not take this path because the engine then needed `workflowsConfig`, a workflows-module artefact riding the `WorkflowAPI` connection.
- **The enrichment no longer needs workflow config.** Access/link/card-worthiness are all functions of data already on the action (`access`, `[display_key].links`, `status`) plus the session user's roles (verified: `computeAllowed`/`collapseLink`/`gateAllows` in `resolveActionAccess.js`; the card-worthiness `$match` reads only `status.stage`). The sole config use ‚Äî declaration-order sort ‚Äî is removed by denormalising the sort key (change 1). With config gone, the engine is a pure function of stored data + roles and is free to live anywhere.

D6's _other_ principle ‚Äî **one policy implementation, no stragglers** (D2) ‚Äî is **preserved**: there is still exactly one JS implementation of verb/link policy in the plugin. This part changes _which surface reaches it_, not how many implementations exist.

### D2 ‚Äî One enriching timeline, not a slot that swaps two timelines (the original Part 50)

The original Part 50 kept two timeline components and added a `components.events_tile` slot so an app could _replace_ the events-only tile with the workflows one per entity. That solved the symptom (two timelines, append-only slots) while keeping the cause (the split). Collapsing removes the cause: one component, enrich-in-place. The consumer cost drops from a per-entity card-swap to a single module-entry var, and the lossless-superset property the original D3 leaned on (`GetEventsTimeline` ‚äá `get-events`) becomes the _literal_ behaviour of one query rather than a guarantee about two. The slot, the pre-wired `events-tile` drop-in, and the `docs/idioms.md` "replaceable-region" flavor are all dropped ‚Äî unbuilt, so nothing to retract.

### D3 ‚Äî `display_key` is the per-app namespace key the engine calls `app_name`

The engine reads display blocks as `$<app_name>.title` and access as `access[app_name]`. The events module already has this exact value as its required `display_key` var ("App identifier for display objects") and builds `$<display_key>.title` in `events-timeline.yaml` today. They are the same key; the engine is configured with `display_key`. No new app-name plumbing.

### D4 ‚Äî Enrichment is inert without action data; no skip branches

D6's instinct that the foundational events module should not assume workflows exist is honoured by the **data**, not by a separate component and not by conditional pipeline assembly. The engine keeps both `$lookup` stages **unconditional** ‚Äî there is nothing to skip:

- the actions `$lookup` keys on `localField: action_ids`. A pure-CRM app's events carry no `action_ids`, so the join matches nothing and returns `actions: []`; the dependent dedup stages (`$unwind`/`$setWindowFields`/`$group`/`$replaceRoot`) then no-op on the empty array, and the `EventsTimeline` block guards card rendering with `Array.isArray(event.actions) && event.actions.length > 0` (verified, `EventsTimeline.js:527`), so no cards render. The stage is inert without action data, so a skip branch would buy nothing. (`$lookup` against a non-existent `from` collection is valid in MongoDB and returns empty, so the `?? 'actions'` fallback is safe even when the app has no `actions` collection.)
- the events module's default config names no workflows concept it **acts on**: the `actions_collection` var defaults to null on the module entry, and the `actions` collection name lives only in the plugin engine's `??` fallback. The one exception is the `onActionClick` check hook that change 5 moves onto the events timeline (`kind: check` / `check_action_modal` vocabulary): it sits in the events component unconditionally, but ‚Äî exactly like the always-present actions `$lookup` ‚Äî it is **inert without action data**. The hook fires only when an action card is clicked, and a card only renders for an app that already has actions referenced by its events; a pure-CRM app renders no cards, so the hook is unreachable dead config there, never a concept the app exercises. Build-time-gating it on `actions_collection` being set is rejected: enrichment is now data-driven (the `?? 'actions'` default means cards can render with the var unset), so gating the hook on the var would leave clickable cards with no handler.

This means **no conditional skip work** for the engine ‚Äî the change in change 1/2 is dropping `workflowsConfig` and reading the denormalised sort key, not net-new pipeline gating. The trade: a pure-CRM app's timeline does run one inert actions `$lookup` per load (cheap; matches nothing) rather than literally no lookup ‚Äî "renders identically," not "executes the identical pipeline."

The contacts `$lookup` is **not** inert the same way: it keys on `created.user.id`, which every event has, so it runs unconditionally against `user-contacts` and adds a `created.user.picture` field (today's events-only request emits no such key). This is not user-visible ‚Äî an unmatched join (system events, deleted contacts, or an absent collection) yields `$ifNull` ‚Üí null ‚Üí the block falls back to initials. Keeping it unconditional is consistent with "no exclusion logic"; it only ever _adds_ a picture, never breaks rendering.

Consequence for framing: the `actions_collection` / `contacts_collection` vars are **collection-name overrides**, not on/off gates. Enrichment effectively works whenever the data is present (events with `action_ids` + a matching collection); the var only matters when your collection isn't named the default. The "one var switches enrichment on" narrative softens to "one var points the engine at your collections (defaulting to `actions` / `user-contacts`); enrichment shows up wherever the data exists." This is a _better_ default-preservation than two components: the events-only path is the same code, not a parallel one, so the two cannot drift.

### D5 ‚Äî Denormalised sort key: the staleness trade, already accepted for access

Denormalising declaration indices means a config reorder (move an action/group) does not retroactively reorder already-written actions until their next transition rewrites the doc. Because the comparator is shared (change 1), this trade applies to **every config-ordered surface, not just the timeline** ‚Äî the overview, entity, and action-group pages served by the three `WorkflowAPI` overview engines also order by the doc-stamped indices, where today they order from live `workflowsConfig`. This is the standard denormalisation trade ‚Äî and the codebase **already makes it for `access`** (`planActionTransition.js:209` stamps config access onto the doc; a config gate change does not propagate to in-flight actions until the next plan). Extending it to a _cosmetic_ ordering field is strictly lower-stakes than the permission field already denormalised this way. Actions written before the field exists sort deterministically last (the comparator already maps a missing index to `+‚àû`, then `_id`), and **no migration is needed ‚Äî workflows has not shipped to consumers**, so there is no live action data to backfill.

## Worked example ‚Äî turn on enrichment for the whole app

```yaml
# apps/demo/modules/events/vars.yaml  (the events module entry)
- id: events
  source: "github:lowdefy/modules-mongodb/modules/events@v1"
  vars:
    display_key: demo
    actions_collection: actions # ‚Üê collection-name override; matches the engine default
    contacts_collection: user-contacts
```

Every entity timeline in the app now renders action cards for actions referenced by its events; entities whose events reference no actions render exactly as before (the actions `$lookup` matches nothing and returns `actions: []`). No entity-module change, no per-entity vars, no second component. A pure-CRM app whose events carry no `action_ids` renders identically to the events-only timeline whether or not the var is set ‚Äî the lookup is inert without action data (D4).

## Non-goals

- **Framework optional-dependency support** ‚Äî not needed; the plugin-package seam (D1) and the `actions_collection` var (D4) give the opt-in without it.
- **Per-entity choice of enriched vs events-only** ‚Äî enrichment is an app-wide switch on the events entry. A mix (enriched on one entity, plain on another) is not supported; if a concrete need appears, it is a follow-up, not speculative surface added now.
- **A data migration for the denormalised sort key** ‚Äî workflows is unshipped; pre-field actions sort deterministically last (D5).
- **Changing the action access/verb model** ‚Äî this part relocates and de-configs the existing read engine; the policy (D4) is unchanged.
