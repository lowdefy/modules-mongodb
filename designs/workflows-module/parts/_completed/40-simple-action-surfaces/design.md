# Part 40 — Check-action surfaces: signal buttons, error recovery, in-context modal

> **Terminology note.** The action kind once called `simple` was renamed to `check` ([Part 43](../43-rename-simple-kind-to-check/design.md)); `simple` was itself the interim name `task` carried before [Part 35](../35-rename-task-kind-to-simple/design.md). This part is written in the final `check` vocabulary throughout — the kind is `check`, the components are `check-action-surface` / `check-action-modal`. The folder name (`40-simple-action-surfaces`) keeps its original `simple-*` spelling as a historical identifier; only path/link tokens and statements describing what Part 35/38 _did_ still read `simple`.
>
> **Part 24 deviation (follow-on, do not reopen this folder).** `check-action-surface.yaml`'s universal-fields `_ref` now passes the full Part 24 var set — `workflow_type` (`_state: current_action.workflow_type`), `action_id`, `allowed_edit`, `on_complete`, and `action_data.assignee_docs` — so the component renders a standalone **Update** button (edit mode) that calls `{workflow_type}-update-fields` independent of any transition, plus display-mode avatars. `show` is omitted (the component defaults to all three — the "no behavioural change for check" floor; the `GetWorkflowAction` envelope carries no presence list). The `submit`/`progress` payloads still carry `fields` (check writes fields on transition). See [parts/24-universal-fields/design.md](../../24-universal-fields/design.md) and its task 11.

**Layer:** module shared pages + two new module components (surface + modal) + a generic `ActionSteps` block event (+ concept-doc reconciliation, applied 2026-06-10 at design time). The surfaces render **dumb** off [Part 46](../46-debundle-workflow-config/design.md)'s `GetWorkflowAction` (per-verb `allowed` + per-signal `buttons`, resolved server-side); the `allow_not_required` flag's server machinery moved to Part 46. **Size:** M (smaller post-46 — the client button/access computation is gone). **Repo:** `modules/workflows/pages/`, `modules/workflows/components/`, `plugins/.../blocks/ActionSteps/`.

The three shared check-action pages (`workflow-action-edit` / `workflow-action-view` / `workflow-action-review`, renamed `task-*` → `simple-*` by [Part 35](../35-rename-task-kind-to-simple/design.md), then `simple-*` → `workflow-action-*` by [Part 38 task 18](../38-engine-rebuild/tasks/18-display-surface-renames.md) per review-14 #1) still run the **old interaction model**: `workflow-action-edit` carries a status **selector** with a `_js` priority filter (`workflow-action-edit.yaml:135`) and a `current_status` payload, and all three pages fire `interaction:` rather than `signal:`. The engine moved to **signals + FSM** ([state-machine](../../../../workflows-module-concept/state-machine/design.md), [Part 38](../38-engine-rebuild/design.md)), and the form templates were migrated by [Part 39](../39-form-submit-buttons/design.md), which shipped the `enums/button_signal_sources.yaml` build-time visibility enum (read via `_ref`) and **explicitly handed the check surfaces to this sibling** ([Part 39 § Check actions are separate](../39-form-submit-buttons/design.md)).

This part rewrites the three shared pages to the signal model, deletes the status selector, resolves the check-action **error-recovery** question, and adds an **in-context modal** so the live working surfaces (`actions-on-entity` and the event-timeline action items) can open a check action without a full page navigation. With [Part 46](../46-debundle-workflow-config/design.md) sequenced first, the surfaces **consume `GetWorkflowAction`'s server-resolved `action.buttons` and `allowed`** — no client-side visibility computation, no `action_role_check` mirror, no `button_signal_sources` `_ref`.

## Proposed change

1. **Delete the `workflow-action-edit` status selector** and its `current_status` / `target_status` payload (`workflow-action-edit.yaml:121–156, 210–211`). Replace with the same nullary signal button bar as the form edit template — `submit`, `progress`, `not_required`. `submit` carries no target; the engine resolves `in-review` vs `done` from the action's `review` verb.
2. **`interaction:` → `signal:` on all three pages** (`submit_edit` → `submit`; `approve` / `request_changes` keep their names), dropping `current_status`. No interaction→status logic on the page — the engine's FSM owns it.
3. **Each signal button reads its server-resolved `action.buttons.{signal}` boolean** from `GetWorkflowAction` ([Part 46 D5](../46-debundle-workflow-config/design.md)) — the FSM source-stage check, per-verb role gate, and `allow_not_required` are AND-ed server-side. No client visibility computation, no enum `_ref`, no `action_role_check` ([D2](#d2--button-visibility-reads-actionbuttons-from-getworkflowaction)).
4. **Add the `progress` button to `workflow-action-edit`** (titled "Mark Started"): `signal: progress`, persists the universal fields without advancing, lands `in-progress`.
5. **Resolve error recovery: a `resolve_error` button on `workflow-action-view`**, rendered only when stage is `error` (FSM `error → resolve_error → in-review`). **No `check-error` page.** Closes [ui Open Question 4](../../../../workflows-module-concept/ui/design.md).
6. **One per-action policy flag: `allow_not_required`** — authored at the action root, gating the `not_required` signal for **every** kind. Its validation, server-side enforcement, and button resolution live in [Part 46](../46-debundle-workflow-config/design.md) (D5); this part only consumes `action.buttons.not_required`. No other per-action button config ([D3](#d3--allow_not_required-the-single-engine-enforced-opt-in-all-other-buttons-fixed)).
7. **Extract the body into a shared `check-action-surface` component** (universal fields + comment + signal buttons, `mode: edit|view|review`). The three pages and the modal both `_ref` it — one body, two containers.
8. **Ship a standalone `check-action-modal` component** and a generic `onActionClick` event on the `ActionSteps` block. `actions-on-entity` bundles the modal and wires the event; any page hosting the event timeline can drop the modal and wire it independently ([D5](#d5--in-context-modal-standalone-component--generic-onactionclick)).

## Surfaces, the engine link, and where the modal fits

Every surface that renders an action reads a **single server-resolved `action.link`**: the engine writes the per-verb map `action[app_name].links = { view, edit, review, error }` — one `{ pageId, urlQuery }` per verb, `null` where the stage has no page for that verb — on every transition ([Part 34 D7/D9](../34-action-access-model/design.md), implemented by [Part 38](../38-engine-rebuild/design.md)), and the reads collapse that map to the one user-appropriate link by static priority `edit > review > error > view` ([Part 42 D5](../42-timeline-action-cards/design.md) moved this server-side in the shared `visible_verbs.yaml` + `resolve_action_link.yaml` stages; [Part 46 D2/D6](../46-debundle-workflow-config/design.md) ports that selection into the engine read methods for **all** reads — overviews, entity, and the cross-stream timeline (`GetEventsTimeline`) — and deletes the shared YAML stages outright (zero stragglers)). The blocks stay dumb and keep reading the singular `action.link`. The page is therefore the **canonical, addressable target** and ships regardless — a notification can only deep-link to a module-owned page (the engine never learns the host app's entity-page URL), so it always navigates.

The modal is an **in-app shortcut layered on the live working surfaces**, never a replacement for the page:

| Surface                                | Renderer               | Default behaviour                              | Modal?                                                                                       |
| -------------------------------------- | ---------------------- | ---------------------------------------------- | -------------------------------------------------------------------------------------------- |
| `actions-on-entity`                    | `ActionSteps` block    | navigate via the server-resolved `action.link` | **Yes** — bundled (D5)                                                                       |
| event timeline (action items)          | `EventsTimeline` block | navigate via the server-resolved `action.link` | **Yes** — host composes the modal ([Part 42](#event-timeline-action-items-part-42--shipped)) |
| `workflow-overview` / `group-overview` | `Link` button          | navigate                                       | No — these _are_ full pages                                                                  |
| notifications / email                  | URL                    | navigate                                       | No — no page loaded to host a modal                                                          |

**The "separate component" design keeps the modules decoupled.** The `EventsTimeline` and `ActionSteps` blocks fire a **generic** `onActionClick(action)` event — neither knows what a workflow check-action surface is. The workflows module ships the modal as a standalone component, and the **host app page composes the two** (drops the modal, wires the event). So there is no events→workflows _code_ coupling — the wiring lives in app composition, exactly where module composition belongs. Navigation remains the block default, so notifications, overviews, and direct deep-links are unaffected.

## Decisions

### D1 — Shared `check-action-surface` component; pages and modal both `_ref` it

The body of a check action — header + universal fields + comment + the signal button bar — becomes one component, `components/check-action-surface.yaml`, parameterised by `mode`:

| `mode`   | Renders                                                                                                   | Button bar                                   |
| -------- | --------------------------------------------------------------------------------------------------------- | -------------------------------------------- |
| `edit`   | universal fields (editable) + comment                                                                     | `submit`, `progress`, `not_required`         |
| `view`   | header + universal fields (editable iff `allowed.edit`, else read-only — see below) + status-history (a List over `current_action.status` — no request) + comment (only at stage `error`, for the `resolve_error` recovery note — binds `current_action.comment`) | `resolve_error` (only at stage `error` — D4) |
| `review` | header + universal fields (editable iff `allowed.edit`, else read-only) + comment                         | `approve`, `request_changes` (modal)         |

The three `check-*` pages `_ref` it; the `check-action-modal` (D5) `_ref`s the same. One body, two containers — this DRY payoff is the reason the surface is extracted now rather than editing three pages in place.

**Universal-fields editability is edit-verb-gated, not mode-gated.** The surface passes the [Part 24](../../24-universal-fields/design.md) universal-fields component `mode: edit` when the user holds the `edit` verb (`current_action.allowed.edit`) and `mode: display` otherwise — **independent of the surface's own mode**. So Part 24's Update operation (its own button, a write with no transition) is reachable on **every** surface the user can open — including the always-available `view` page — for whatever stages Part 24's operation permits. That is what makes reassign-without-transition reachable without routing through the stage-gated edit page: an `edit`-verb user opening a non-actionable action's view page (the canonical link target for `done`/`not-required`/`error`) sees editable fields + Update, while a pure viewer sees the read-only avatar display. The surface `mode` in the table governs only the **signal button bar** and the comment field, not field editability. (Whether the operation should be available on terminal stages such as `done` is [Part 24](../../24-universal-fields/design.md)'s open question, not decided here.)

**The [Part 33](../../33-comment-rendering/design.md) events timeline is page-level chrome, not part of the surface** (review-2 #4). `workflow-action-view` renders the action-filtered timeline below the surface `_ref`; the modal omits it. Rationale: the modal's hosts are entity pages whose own entity timeline already shows the action's events and comments, so an in-modal copy duplicates what's directly behind the modal — and the full action-filtered stream lives on `workflow-action-view`, one navigation away (the modal is a shortcut, never the canonical surface). It also avoids a hard conflict: a second `events-timeline` instance on an entity page would collide on the component's fixed `get-events` request id (Lowdefy build throws `Duplicate requestId`; request ids are not `_ref`-scoped). Part 33's "the timeline rides the surface's `view` mode" line is amended to page-level accordingly.

**State contract.** The surface reads from a single `_state.current_action` namespace. The **`GetWorkflowAction` response is spread into it** — the action doc plus the server-resolved `allowed` (`{ view, edit, review, error }`) and `buttons` (`{ submit, progress, … }` booleans) sit at the top (`current_action.status`, `current_action.buttons.submit`, `current_action.allowed.edit`, `current_action.workflow_closed`), with the working inputs on sub-keys (`current_action.fields`, `current_action.comment`). The page's `onMount` populates `current_action` from the `get_workflow_action` request (now routed to `GetWorkflowAction`, Part 46) and seeds `current_action.fields`; the modal's open handler does the same for the clicked `action_id`. One read convention, two writers.

**`mode` rides the same namespace — `current_action.mode`** _(mechanism settled at task time)_. The pages set a literal in `onMount` (`current_action.mode: edit` / `view` / `review`); the modal sets the **derived** value in its open handler (D5). The surface gates every mode-dependent block on `_state: current_action.mode` — not an `_ref` var: a build-time var can only carry the modal's runtime derivation as a spliced operator object, which works but silently breaks the moment `_var: mode` is used in any build-time/structural position — a convention no mechanism enforces. State keeps the one read convention and removes that failure class.

**No client role-check (supersedes review-2 #3).** The shipped `action_role_check` mirror is deleted by [Part 46 D8](../46-debundle-workflow-config/design.md); access and button visibility arrive resolved in the `GetWorkflowAction` response, so there is no per-page role computation and no `SetState` copy into the namespace — the surface reads `current_action.allowed.{verb}` and `current_action.buttons.{signal}` directly. The earlier trailing-`SetState` workaround (a copy from root `action_allowed` because the shared component's write target was fixed) is retired entirely.

**Universal-fields namespace contract ([Part 24](../../24-universal-fields/design.md) — review-2 #5).** `Validate` matches state keys, and inputs bind state at their block IDs — so the renderer's edit-mode inputs must live at `current_action.fields.*` for the scoped `Validate` and the `fields` payload to see them. Part 24's renderer takes a `state_path` var (default `fields`, so the form sidebars and shared-page defaults are untouched); the surface passes `state_path: current_action.fields`, making the input block IDs `current_action.fields.{assignees, due_date, description}` (threading into the `user-multi-selector` id passthrough). Recorded on both sides — Part 24's design carries the var.

Each button's `CallAPI` payload is nullary on target. The **base payload is identical for every signal**; only the two edit-mode content signals add `fields`:

```yaml
# base — every signal
payload:
  action_id: { _state: current_action._id }
  signal: submit # submit / progress / not_required / approve / request_changes / resolve_error
  current_key: { _state: current_action.key }
  comment: { _state: current_action.comment }
  # submit + progress ONLY — the universal fields are this signal's live submission content:
  fields: { _state: current_action.fields } # assignees / due_date / description
```

**`fields` rides `submit` and `progress` only** (review-4 #7). The engine writes `payload.fields` onto the action doc on every check transition (Part 24's kind-based rule), so a signal carrying the open-time `fields` seed when the user can't edit it — `approve` / `request_changes` (review mode is read-only), `resolve_error` (recovery note in view mode), or `not_required` (a skip) — would silently revert a concurrent editor's field change. Those four send the base payload **without** `fields`; only `submit` and `progress`, where the edit-mode universal fields are the live content the user is entering, carry it. `comment` rides **every** signal — unlike `fields` it's a fresh per-transition note folded into that signal's event ([Part 33](../../33-comment-rendering/design.md)), never an overwrite. (Independent field edits without a transition — e.g. reassigning a `done` check action — go through Part 24's shared universal-fields Update operation, available for every kind, not through a signal.)

No `form` / `form_review` (check actions have no form body), and **no `current_status` / `target_status`** — the v0 selector payload is gone ([state-machine "What disappears"](../../../../workflows-module-concept/state-machine/design.md)). The endpoint resolves **at runtime** — `_string.concat` of `{ _module.id: true }` + `/update-action-` + the action type off the response (`_state: current_action.type`) — producing the same scoped id the form templates build with `_module.endpointId: { _build.string.concat: [update-action-, <type>] }`. The build-time form is unavailable here: the shared pages serve every check action, so the action type is runtime-only, where the generated form pages know theirs at generation time. _(Amended at task time — an earlier draft said "aligns with the form templates", but `_module.endpointId`/`_build.*` evaluate at build and cannot consume a runtime type; the shipped pages' runtime concat stands.)_ `progress` has no `Validate` step (a draft is intentionally partial) and fires **no author hook** — unlike the form template's njk-baked `page_config.events.onProgress` ([Part 39 D2](../39-form-submit-buttons/design.md)), the shared static pages have no per-action baking point to splice one in (the same structural fact D3 stands on). _(Amended at task time — an earlier draft claimed an `onProgress` hook here by form analogy, contradicting D3.)_ The engine-side `progress_saved` log event is Part 38 (scoped out below). `submit` keeps a `Validate` step, **scoped** to the surface's own field namespace — `params: { regex: ^current_action\.fields\. }` (the repo idiom for namespaced validation, cf. `^entity\.` on edit pages). The scope matters because the surface is rendered both as a page and inside the modal: an unscoped `Validate` inside the modal would validate the **entire host entity page** (every unrelated input). Scoping to `current_action.fields.*` makes validation identical in both containers and confined to this action's fields.

### D2 — Button visibility reads `action.buttons` from GetWorkflowAction

Each signal button's `visible` is a single server-resolved boolean — the client computes nothing:

```yaml
visible: { _state: current_action.buttons.submit } # or progress / not_required / approve / request_changes / resolve_error
```

[Part 46 D5](../46-debundle-workflow-config/design.md) resolves `buttons.{signal}` server-side as the AND of the three read-time-knowable dimensions — FSM source-stage, the per-verb role gate (`allowed.{verb}` for the signal's required verb, [Part 34 D6/D8](../34-action-access-model/design.md)), and `allow_not_required` (for `not_required` only). No `button_signal_sources` `_ref`, no per-verb AND on the page, no `allow_not_required` term — all of it collapses into the boolean.

**The per-verb gate still applies — server-side.** A button's required verb (`submit`/`progress`/`not_required` → `edit`; `approve` → `review`; `request_changes` → any of `view`/`edit`/`review` per [Part 49](../49-request-changes-verb-gate/design.md), which superseded [Part 34 D6](../34-action-access-model/design.md)'s `review`-only row; `resolve_error` → `error`) is folded into its `buttons.{signal}` boolean by `GetWorkflowAction`. This is what lets **one shared surface** gate a mixed-verb button bar correctly: the server resolves each signal against the action's own verbs, stage, and the requesting user, so the same surface renders right for an editor, a reviewer, or an error-recoverer. The client never sees `allowed.{verb}` for button purposes (it stays available on the response for the modal's mode derivation — D5).

This deletes the `_js` priority lookup on the selector (`workflow-action-edit.yaml:144–156`) outright. A button shows exactly when its signal is coherent from the action's current stage — and [Part 38 D13(3)](../38-engine-rebuild/design.md) makes that matter: a **user-driven** signal with no FSM entry **throws**, so a button shown from an incoherent stage would surface a user error. A button shown from a now-stale read (a concurrent stage push the `GetWorkflowAction` response didn't see) is still FSM-checked at submit, and per the same D13(3) rule the **user-driven** signal **throws** (`WorkflowEngineError`) — it does **not** silently no-op (the no-op path is reserved for cascade/mirror signals). This thrown error is the accepted outcome for the rare read-skew race — the read was honest when rendered — and it surfaces through the signal call's standard error path; the surfaces add no bespoke per-signal error handling, matching the form templates' signal calls. The `resolve_error` button on `workflow-action-view` falls straight out — `buttons.resolve_error` is true only at stage `error` for an `error`-verb user.

### D3 — `allow_not_required`: the single engine-enforced opt-in; all other buttons fixed

An earlier draft imported Part 39's full per-button author opt-out map to check actions as "form parity", carried by a resolver-emitted runtime global. Review-2 #2 showed the mechanism didn't exist, and the requirements dig behind it showed the _requirement_ didn't either: [ui](../../../../workflows-module-concept/ui/design.md) specifies check actions as "**no per-action customisation; identical experience for every check action across every workflow**", and no button except `not_required` has a per-action story — hiding `submit` makes the page useless, `progress` is a harmless draft-save, and the rest are already stage- and role-gated. The opt-out was also UX-only: the engine accepts any user-driven signal with an FSM entry, so a hidden button never restricted anything.

`not_required` _is_ a per-action policy ("is skipping this check ever legitimate?" — varies per action in a way FSM and roles can't express). It becomes the **single** authored flag, validated and enforced server-side (its machinery moved into Part 46 under the order flip):

- **Semantics** — `allow_not_required: true` at the action root, **any kind** (default absent = `false`, opt-in — preserving [Part 39 D3](../39-form-submit-buttons/design.md)'s safety rationale).
- **Server machinery — [Part 46](../46-debundle-workflow-config/design.md) (D5)** — validation (`makeWorkflowsConfig`, boolean); the kind-agnostic load-phase gate rejecting a **user-driven** `not_required` off **live config** (`access_denied`, ahead of pre-hooks — closing the gap where Part 39's client opt-out was only advisory); and the `not_required`-button resolution inside `GetWorkflowAction`. Because `GetWorkflowAction` reads the flag from live config, **no doc-persist-for-display is needed** — the earlier per-transition denormalisation plan is dropped. Engine-driven signals are unaffected.
- **Client (this part)** — the surface gates the `not_required` button on the server-resolved `_state: current_action.buttons.not_required`; zero config plumbing.
- **Form alignment ([Part 46](../46-debundle-workflow-config/design.md))** — `edit.yaml.njk`'s `not_required` term reads the same server-resolved boolean; the form's `page_config.buttons.not_required.visible` remains a plain author opt-out (default **`true`**), ANDed client-side (D5 layer 2).

All other buttons (`submit`, `progress`, `approve`, `request_changes`, `resolve_error`) are **fixed** on the check surface — no config map, no resolver, no global. This is justified **structurally**, not just on "build for what exists": author button config (`.visible`/`.disabled`) only earns its place as a **runtime operator** reacting to live client state, which must be baked into the page at build time — and only the **generated form pages** have a per-action baking point. The three shared check pages have none, so a check action structurally **cannot** host per-action runtime button config (short of generating per-check pages, rejected by [Part 46 D4](../46-debundle-workflow-config/design.md)). [Part 46](../46-debundle-workflow-config/design.md) resolves the _engine-policy_ button booleans (FSM source-stage ∧ per-verb gate ∧ `allow_not_required`) server-side for every kind via `GetWorkflowAction` — uniform by construction, and all a shared-page surface can consume. So 46 does **not** "reopen" configurable check buttons; it confirms the boundary: form buttons can react to page state (generated pages), check buttons obey engine policy only (shared pages).

### D4 — Error recovery: `resolve_error` on `workflow-action-view`

A check action reaches `error` only via a pre-hook `error` cascade ([state-machine "Check kind"](../../../../workflows-module-concept/state-machine/design.md)) — no check page surfaces an `error` button (the engine never self-sets `error`). [ui Open Question 4](../../../../workflows-module-concept/ui/design.md) left the recovery surface open. **This part resolves it as the lighter option: a `resolve_error` button on `workflow-action-view`, rendered only when stage is `error`** (FSM `error → resolve_error → in-review`).

Rationale:

- A fourth static page for a rare cascade is heavier than the case warrants, and `workflow-action-view` already loads the full action — it's the natural recovery context.
- The Part 38 engine's per-verb link table needs one matching special case (Part 38 review-14 #4): for `kind: check`, the `error` verb links to the **view** page (`workflow-action-view`) — there is no error page to link to — so an error-verb-only user still gets a working link from timeline cards and overviews. (The old engine's `linkDefaults` already routed the `error` stage to the view page, per the [Part 30 D4 table](../../_rejected/30-status-map-rendering/design.md); the rebuilt table initially pointed the error verb at a nonexistent `check-error` page — fixed via Part 38 task 18's link-table coordination.) Form kind is unaffected: generated `{workflow_type}-{action_type}-error` pages exist per verb.
- Visibility is the server-resolved `action.buttons.resolve_error` (true only at stage `error` for an `error`-verb user — [Part 46 D5](../46-debundle-workflow-config/design.md)).

`resolve_error` reuses the comment field (recovery note) and fires the standard payload. There is **no `check-error` page** in v1.

### D5 — In-context modal: standalone component + generic `onActionClick`

Two new pieces plus one generic block event.

**`components/check-action-modal.yaml` (standalone, reusable).** A single **`Modal`** block, fixed blockId `check_action_modal`, whose body `_ref`s `check-action-surface`; the modal's open handler **sets the derived `current_action.mode`** from the fetched action's stage (the surface reads mode from state — D1). With the events timeline page-only (D1 / review-2 #4), every mode is light — universal fields, status history, comment, signal buttons — so a centred `Modal` fits the content better than a `Drawer` (an earlier draft chose a `Drawer` to hold a timeline-bearing `view` mode; that rationale died with the in-modal timeline). One block type for all modes keeps the one fixed blockId / one open contract intact (a runtime block cannot switch its type by mode, so two container types would mean two blockIds and break the contract).

- stage `error` → `view` (surfaces `resolve_error`); stage `in-review` and `allowed.review` → `review`; an actionable stage with `allowed.edit` → `edit`; otherwise → `view`. (The per-verb bools come from the `GetWorkflowAction` response — `current_action.allowed` — see the open sequence in this decision.) This derivation drives the **signal button bar** only; the universal-fields component's editability is the separate edit-verb gate (`allowed.edit ? edit : display`, D1), so a `done`/`not-required` action opened in the modal is still reassignable by an `edit`-verb user even though its derived mode is `view`.

Open contract (fixed, so any host wires it the same way):

```yaml
# Host wiring — both ActionSteps and EventsTimeline fire onActionClick(action),
# where action carries { _id, kind, status, link, … }. Branch on kind: only
# `check` actions open in the modal; every other kind navigates via action.link.
- type: SetState
  skip: { _ne: [{ _event: action.kind }, check] }
  params: { check_action_modal: { action_id: { _event: action._id } } }
- type: CallMethod
  skip: { _ne: [{ _event: action.kind }, check] }
  params:
    { blockId: check_action_modal, method: setOpen, args: [{ open: true }] }
- type: Link # non-check kinds: navigate to the resolved page
  skip: { _eq: [{ _event: action.kind }, check] }
  params:
    pageId: { _event: action.link.pageId }
    urlQuery: { _event: action.link.urlQuery }
```

The branch reads `_event.action.{kind, _id, link}` — so the feeding read must carry `_id` **and** `kind` per action. Both feeding reads do: `GetEntityWorkflows` (entity page) and the cross-stream `GetEventsTimeline` (event timeline) each project `_id` and `kind` per action — added by [Part 46](../46-debundle-workflow-config/design.md) (its "The read methods" table) expressly because both are check-modal hosts. Linkless rows never reach the `Link` branch — the blocks suppress the click for them (next paragraph).

On open the modal runs the **same fetch the page's `onMount` runs**, for `check_action_modal.action_id` — the surface depends on it (D6), and none of it runs unless the modal replicates it:

1. `get_workflow_action` (now `GetWorkflowAction`, fresh — the list/timeline data may be stale) → spread the response into `current_action` (action doc **+ resolved `allowed` + `buttons` + `workflow_closed`**), seed `current_action.fields`, and set the derived `current_action.mode` (the stage/verb derivation above). **One read** — no separate role-check, and no separate `get_workflow`: Part 46 folds the parent-workflow read into `GetWorkflowAction` and deletes the request, so the workflow-closed banner and the `required_after_close` gate on the submit buttons read the resolved `current_action.workflow_closed` (`stage ∈ {completed, cancelled}`, collapsed server-side — [Part 46 D8](../46-debundle-workflow-config/design.md)) and `current_action.required_after_close` straight off the same response.

Then it renders — D2's button visibility and the mode derivation both read straight off `current_action`. On a successful signal call it runs the host-supplied `onComplete` refetch (passed as a `_var`) and closes. `mode: view` renders the surface's status history (a List over `current_action.status` — no request, modal-safe); the [Part 33](../../33-comment-rendering/design.md) events timeline is **not** rendered in the modal (page-level only — D1 / review-2 #4): the host entity page's own timeline already carries the action's events and comments, and the full action-filtered stream lives on `workflow-action-view`.

**Both blocks fire a generic `onActionClick(action)` event.** When the host wires `onActionClick`, the block fires it with the clicked **action object** (`{ _id, kind, status, link, … }`) **instead of** navigating; when not wired, it navigates via the server-resolved `action.link`. This is `ActionSteps`' existing default (unchanged — notifications/overviews/deep-links preserved); **this part converges `EventsTimeline` onto the identical contract** (see _Event-timeline action items_, below — it shipped firing `{ pageId, urlQuery }` and with no navigate-default). Both blocks also **suppress the click for linkless rows** (keeping `ActionSteps`' existing disabled-row behaviour, `ActionSteps.js:170`), so `onActionClick` only ever fires for actionable rows — the host's `Link` branch never sees a null link. The event is **generic** (carries the action object); neither block gains workflow-surface knowledge.

**`actions-on-entity` — bundles the single modal instance.** The component drops `check-action-modal` once and wires `ActionSteps.onActionClick` to it with the kind-branch above (`check` → open the modal; other kinds → `Link` to `action.link`), passing `entity-workflows-refetch` as `onComplete`. Apps that already use `actions-on-entity` get the modal for free — no per-app wiring (satisfies "one correct way"). Because the blockId is fixed (`check_action_modal`), any other action surface on the same page (e.g. a co-present event timeline) targets **this same instance** by id — it does **not** drop its own. The rule is simple: **the modal is dropped exactly once per page, and `actions-on-entity` is what drops it when present.**

**Reusable without `actions-on-entity` — and graceful when absent.** The modal is **opt-in**, so there is no collision and no broken wiring on a page that lacks `actions-on-entity`:

- A page that renders the event timeline but not `actions-on-entity`, and **wants** the in-context modal, drops `check-action-modal` itself (once) and wires `EventsTimeline.onActionClick` to it (passing its own timeline refetch as `onComplete`). Post-46 the timeline ships wrapped in the module's `workflows-events-timeline` component (Part 46 task 11), so this part adds an `on_action_click` events-passthrough var to that wrapper — absent by default, preserving the navigate fallback below.
- A page with the timeline and **no modal at all** simply leaves `EventsTimeline.onActionClick` unwired — the timeline falls back to its default and **navigates to the action page** (the same server-resolved `action.link` default as `ActionSteps`). No missing-blockId target, no double-drop.

A developer never drops the standalone modal on a page that already has `actions-on-entity` (it's already bundled there), so the "exactly once" rule has no failure mode. The `EventsTimeline.onActionClick` event shipped with [Part 42](#event-timeline-action-items-part-42--shipped); this part defines the modal component and its open contract that the timeline hosts consume, and **converges the `EventsTimeline` event/default onto the `ActionSteps` contract** so the same kind-branch wiring drives both (see _Event-timeline action items_, below).

### D6 — What carries over unchanged

The shared-page scaffolding stays: the `action_id` presence guard, the `get_workflow_action` (now `GetWorkflowAction`) request, the workflow-closed banner + `required_after_close` gate on the submit buttons (now reading the resolved `current_action.workflow_closed` + `current_action.required_after_close` off the single `GetWorkflowAction` response — D1/D5), the stale-URL guard on `workflow-action-edit` (`[action-required, in-progress, changes-required]`) and `workflow-action-review` (now `[in-review]` — `error` dropped: a check action at `error` renders no review buttons and the engine routes its `error` verb to `workflow-action-view`, so the guard redirects an `error`-stage hit to the view page where `resolve_error` lives rather than stranding it on a buttonless review render — #8), and `workflow-action-view`'s status-history card (absorbed into the surface's `view` mode) + the events-timeline `_ref` (Part 33's swap — rendered by the page below the surface, not inside it; D1 / review-2 #4). Replaced: the selector, the `interaction:`/`current_status` payloads, and the per-button `_js` visibility. **Gone** (deleted by [Part 46](../46-debundle-workflow-config/design.md) before this part runs): the `action_role_check` step and its `SetState`-into-namespace copy, **and the separate `get_workflow` request** — Part 46 folds the parent-workflow read into `GetWorkflowAction`, so access, buttons, and the workflow-closed state all arrive in its response (D1, D2).

## Event-timeline action items (Part 42 — shipped)

The reference project's entity timeline shows the **most recent event per action** as a status-coloured card linking to the action. This was a discovered gap when this part was first drafted; it is now **[Part 42 — Timeline action cards](../42-timeline-action-cards/design.md), shipped** (end-to-end testing pending the wave coming together). A host page that wants in-context opening composes **this part's** `check-action-modal` with the timeline (D5). **Convergence (this part) — was the "payload caveat".** As shipped, `EventsTimeline.onActionClick` fires `{ pageId, urlQuery }` (the resolved link) and the block has no navigate-default, while the modal contract and the kind-branch need `_event.action.{_id, kind, link}`. - **`EventsTimeline` block** — `onActionClick` fires the **action object** (matching `ActionSteps`), and the block **navigates via `action.link` by default** when `onActionClick` is unwired (it previously only fired the event). Identical contract to `ActionSteps`; linkless rows suppress the click.
- **Data path (no Part 40 edit).** Post-46 the timeline's action cards are served by Part 46's cross-stream `GetEventsTimeline`, whose `$project` already carries `{ _id, kind, status, link, message, … }` — `_id`/`kind` added by Part 46 expressly because the timeline hosts the check modal. The shipped `timeline_action_lookup.yaml` is **ported into `GetEventsTimeline` and deleted** by [Part 46 D6](../46-debundle-workflow-config/design.md) (all three shared YAML stages go, zero stragglers), so this part owns **no** data-side timeline edit — only the block-side convergence above.

With the block converged and `GetEventsTimeline` supplying `_id`/`kind`/`link` per card, the same kind-branch wiring (D5) drives the timeline exactly as it drives `actions-on-entity`.
