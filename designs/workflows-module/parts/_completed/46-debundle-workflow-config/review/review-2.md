## Blocking — verify before building

### 1. The read methods' user source (line 120) does not match the Lowdefy resolver contract — and the shipped submit gate's user source needs the same audit

> **Resolved.** Confirmed against source: `callRequestResolver` (both pinned api versions) passes no top-level `user`; `createEngineContext.js:45` reads a top-level `user` that is `undefined` on the real path; the submit endpoint declares no `user` property; `gateAllows` denies role-array gates with empty roles. The audit confirms the latent submit-gate bug. **Chose the connection-property mechanism** over the request-property one: declare `user` in `schema.js`, set `user: { _user: true }` on the connection (resolves to `lowdefyContext.connection.user`), and change `createEngineContext` to read `connection.user` → `context.user`. Because `createEngineContext` is shared, this single edit fixes both the read methods and the shipped submit gate. Line 120 rewritten with the corrected mechanism + files-changed (`schema.js`/`workflow-api.yaml`/`createEngineContext.js`); a Ripple bullet records the submit-gate fix. (The connection-property choice also covers part of finding #2's `schema.js` work.)

Line 120 states the user reaches each read method "via a **connection-level `user: { _user: true }` property** — resolved per-request in the request context, exactly as `changeLog.meta.user` already is (`workflow-api.yaml:24`) — exposed as `context.user`." Traced through the shipped resolver path, none of the three clauses hold:

- **The resolver is handed no top-level `user`.** `callRequestResolver` (`.../routes/request/callRequestResolver.js:46–56`) invokes the connection resolver with exactly `{ blockId, callApi, connection: connectionProperties, connectionId, endpointId, pageId, payload, request: requestProperties, requestId }`. There is no `user` key. Yet `createEngineContext` reads `user` off that object (`createEngineContext.js:47`, `const { …, user, … } = lowdefyContext`), so `context.user` is `undefined` on this path.
- **`changeLog.meta.user` is not `context.user`.** The `_user: true` under `changeLog.meta` (`workflow-api.yaml:24`) is evaluated by `evaluateOperators` into `connectionProperties`, so it arrives as `lowdefyContext.connection.changeLog.meta.user` and is read from there by `planChangeLog` — it never populates the top-level `context.user` the verb gate uses (`loadWorkflowState.js:171`, `context.user?.apps?.[currentApp]?.roles`).
- **A new top-level `user` connection property would be rejected.** The connection schema is `additionalProperties: false` (`schema.js:4`) and `validateSchemas` runs on every request (`callRequest.js:78`). `user` is not a declared property, so adding `user: { _user: true }` to `workflow-api.yaml` fails validation until `schema.js` declares it — and even then it lands at `connection.user`, not `context.user`.

The session user *is* available during operator evaluation (`createApiContext.js:19`, `context.user = context?.session?.user`), which is why `_user` resolves inside connection/request **properties** — but only as a value embedded in `connection`/`request`, not as a resolver argument. The shipped `SubmitWorkflowAction` tests pass `user` by hand-building the context (`SubmitWorkflowAction.test.js:198`), which masks this: in the real request path the submit access gate appears to receive `context.user === undefined`, i.e. `userRoles = []`, denying every role-array gate. **Audit the production submit path first** — whatever actually feeds `context.user` there is the only proven mechanism, and the read methods must reuse it. If nothing feeds it, that is a latent gate bug this part now depends on.

**Fix.** Pin the real mechanism, don't assert the connection-property one. The Lowdefy-native option that actually reaches the resolver is a **request property** on each read endpoint — `user: { _user: true }` in the routine's `properties`, arriving as `lowdefyContext.request.user` (`params.user`), read in `createEngineContext` — mirroring how `makeWorkflowApis` already threads payload via request properties (`makeWorkflowApis.js:58–69`). Alternatively declare `user` in `schema.js` and read `connection.user`. Either way, correct line 120 and add the `createEngineContext` read to the files-changed.

## Gaps — under-specified mechanics

### 2. Moving `entities` (and the user input) onto the connection requires `schema.js` changes the design never mentions

> **Resolved.** Confirmed: schema is `additionalProperties: false` (`schema.js:4`); the `entities` module-var shape (`module.lowdefy.yaml:66–79`) and the overview back-link derivation (`urlQuery = { [id_query_key]: entity_id }`) are as stated. `user`'s declaration was added under finding #1; added the `entities` top-level declaration to "The read methods" §3, plus a note in "Validated config additions" that the config-field additions (`title`/`form_meta`/`allow_not_required`) need no schema work (`workflowsConfig.items` is `additionalProperties: true`, `schema.js:87`/`:100`) — only the two new top-level connection properties (`user`, `entities`) do.

The connection schema is closed (`schema.js:4`, `additionalProperties: false`). The design adds two new top-level connection inputs — the `entities` routing map (lines 117, 146: "The `entities` map moves onto the connection config") and whatever carries the user (finding #1) — but never mentions `schema.js`. Both must be declared there or the connection fails `validateSchemas` at build/request time.

Note the **config-field** additions are fine without schema work: `workflowsConfig.items` is `additionalProperties: true` at both the workflow and action level (`schema.js:87`, `:100`), so the new `title`, `form_meta`, and `allow_not_required` fields validate as-is. It is specifically the two **new top-level connection properties** that need schema declarations. `entities` already has a fully-specified shape as a module var (`module.lowdefy.yaml:66–79`, `{ page_id, id_query_key, title }` keyed by `entity_collection`) — port that shape into the schema, and confirm the resolved `entity_link` `{ pageId, urlQuery, title }` derives as `urlQuery = { [id_query_key]: entity_id }` (it does, per `workflow-overview.yaml:147–177`).

### 3. The read methods require a `requestResolver.meta`; "no `checkRead`/`checkWrite` meta" (line 120) is incorrect and would throw

> **Resolved.** Confirmed `checkConnectionRead.js:17` dereferences `meta.checkRead` unconditionally, so a resolver with no `.meta` throws — meta is mandatory. Corrected line 120: read methods set `meta = { checkRead: false, checkWrite: false }`. Per the user's call, **both checks are disabled rather than `checkRead: true`** — access control lives in the engine verb gate inside each method, not in the connection-level read/write capability flags, so connection-level read-gating adds nothing here. Replaced the (incorrect) "no meta" rejection with this.

Line 120's parenthetical rejects per-method `meta` ("that optional Lowdefy read/write toggle doesn't fit a single connection that serves both reads and writes"). That misreads the mechanism, and the code makes `meta` mandatory:

- `checkConnectionRead` dereferences `requestResolver.meta.checkRead` unconditionally (`checkConnectionRead.js:17`), and `callRequest` calls it on every request (`callRequest.js:66`). A resolver with no `.meta` throws `Cannot read properties of undefined`. All four shipped handlers set it (`SubmitWorkflowAction.js:22–25`, etc., all `{ checkRead: false, checkWrite: true }`).
- `meta.checkRead`/`checkWrite` is **per-handler capability declaration**, not a per-method toggling of the connection. The check only errors when the handler declares a capability the connection denies (`checkConnectionRead.js:17`: `meta.checkRead && connectionProperties.read === false`). A single connection serving both is exactly the supported case: the connection grants `read: true` (default, `schema.js:46–49`) **and** `write: true` (`workflow-api.yaml:7`), and each handler declares which it is.

**Fix.** The read methods must set `meta = { checkRead: true, checkWrite: false }` (the connection's default `read: true` already grants it). Replace the line-120 rejection with this. This also restores symmetry with the four write handlers and gives the read methods real read-gating instead of silently skipping it.

### 4. `allow_not_required` × the **form** `not_required` button is unspecified — folding it into `action.buttons` risks hiding currently-visible buttons

> **Resolved.** Confirmed shipped `edit.yaml.njk:314` defaults `not_required.visible` to `false` with no `allow_not_required` term. Chose **option (b)** — `allow_not_required` is kind-agnostic (every kind, default `false`, opt-in), enforced as a server-side load-gate on `not_required` for all kinds; this aligns Part 46 with the decision Part 40 D3 already made and gives forms a load-gate they lack today. The form `page_config.buttons.not_required.visible` flips from default-`false` opt-in to default-`true` opt-out, with the opt-in moving to the root `allow_not_required`. Added the migration rule (shipped form actions showing the button get `allow_not_required: true`); no such action exists in this repo's demo. Updated D5 (term 3 + form layer-2 bullet), "Validated config additions", and the Part 39 Ripple.

D5 (line 59) and "Validated config additions" (line 135) move `allow_not_required` resolution into this part as a term of the server-side `action.buttons.not_required`. But on the shipped **form** templates, `allow_not_required` is **not** a term today — the `not_required` button is gated purely on the author flag + FSM stage + edit verb (`edit.yaml.njk:310–322`: `page_config.buttons.not_required.visible` default **false**, ANDed with `button_signal_sources` stage and `action_allowed.edit`). `allow_not_required` is a Part-40 *simple*-kind concept (`40/design.md`).

So if `GetAction` computes `action.buttons.not_required = stage ∧ edit ∧ allow_not_required` for **all** kinds, a form action that today shows the button via `page_config.buttons.not_required.visible: true` will go dark unless its config also sets `allow_not_required: true`. The design doesn't say whether `allow_not_required` applies to the form kind, or its default. Decide and state it: either (a) `allow_not_required` is simple-only and form `action.buttons.not_required` omits that term (no behaviour change for Part 39), or (b) it applies to both and the default must be chosen so shipped form actions don't regress — and any form action currently using the button gets `allow_not_required: true` added in this part's config migration.

### 6. D5's FSM-source-stage prose (line 57) is imprecise vs the authoritative table, and a naive inversion over-includes internal signals

> **Resolved.** Confirmed the discrepancies against `button_signal_sources.yaml` (`progress` is action-required/in-progress only; `submit`/`request_changes` also fire from `done`; `not_required` also from blocked/in-review/error). Replaced the inexact restatement with a pointer to `button_signal_sources.yaml` / `FSM_TABLES.form` as authoritative, and added the derivation hazard note: any server-side inversion must restrict to the six `SIGNAL_VERBS` keys or it surfaces buttons for internal/lifecycle signals.

Line 57 says "`submit`/`progress`/`not_required` at `action-required`/`in-progress`/`changes-required`, `approve`/`request_changes` at `in-review`, `resolve_error` at `error`." The authoritative inversion (`tables.js` form table ≡ `button_signal_sources.yaml`) differs in ways that matter: `submit` also fires from `done`; `request_changes` also from `done`; `not_required` also from `blocked`/`in-review`/`error`; and `progress` fires only from `action-required`/`in-progress` (**not** `changes-required`). Since the design (lines 57, 144) makes the FSM table the normative source, either point at `button_signal_sources.yaml`/the table as authoritative rather than restating an inexact summary, or correct the summary.

Also flag the derivation hazard for the "implementer's choice" in line 144: `FSM_TABLES[kind][stage]` keys include lifecycle/internal signals (`activate`, `block`, `unblock`, `internal_cancel_action`, the `internal_mirror_*` family). A server-side inversion must restrict to the six user signals (the `SIGNAL_VERBS` keys) or it will surface buttons for internal transitions. Keeping the existing `button_signal_sources.yaml` enum and reading it server-side (the offered alternative) sidesteps this entirely, since the enum is already restricted to the six.
