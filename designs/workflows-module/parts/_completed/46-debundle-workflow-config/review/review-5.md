## Gaps — the `GetWorkflowAction` envelope can't feed all its consumers

### 1. Deleting `requests/get_workflow.yaml` breaks the two static detail pages — they read the **workflow stage**, which the envelope doesn't carry

> **Resolved.** `GetWorkflowAction` gains a resolved `workflow_closed` boolean (parent workflow `stage ∈ {completed, cancelled}`), computed from the parent-workflow read it already does. Chose the resolved boolean over the raw `workflow_stage` string per the D8 rule: every consumer (both static pages, all closed-banner/`required_after_close` reads, and Part 40's planned review banner) computes the identical closed/not-closed disjunction, so no consumer needs the raw stage value. Updated D8 envelope (Resolved fields), "The read methods" §4, the response table, and "What gets deleted" to name both consumer sets of `get_workflow.yaml` (four templates → form values; two static pages → `workflow_closed`). Corrected task 10's "only consumers are these templates" (new step 1c + Files/AC) and task 5's envelope (Resolved fields + AC).

The design scopes the `get_workflow` deletion to form-data: "the four form templates (`edit`/`view`/`review`/`error.yaml.njk`) drop their `get_workflow` step and read submitted values off the single `GetWorkflowAction` response" ("What gets deleted", read-methods §4). That is correct for the templates — their only `get_workflow` reads are `form_data` (`edit.yaml.njk:100`, `review.yaml.njk:93,98`, `view.yaml.njk:89`, `error.yaml.njk:93`).

But the request has two more consumers the design never mentions: the **static detail pages** `pages/workflow-action-edit.yaml` (`:19` `_ref`, `:54–56` request step) and `pages/workflow-action-review.yaml` (`:19`, `:50–52`). They read `get_workflow.status.0.stage` — not `form_data` — to drive the **workflow-closed banner** and the `required_after_close` gating (`workflow-action-edit.yaml:96,99,180,183`; `workflow-action-review.yaml:80,83,165,168,193,196`, with the explicit comment "Step 4: load workflow — needed for required_after_close gate"). These pages are live until Part 40 replaces them, and Part 40 is paused **behind this part** — so this part lands on a tree where they must keep working. Worse, the data they need has no home: the curated envelope (D8) carries action fields + form values + `allowed`/`buttons` but **no parent-workflow stage**, so there is nothing to rewire the reads to. And the need outlives Part 40: its planned review surface also carries a "workflow-closed banner" (`40/design.md:51`), which will have the same missing input.

**Fix.** The method already reads the parent workflow doc (the D8 two-read). Add the workflow's lifecycle stage to the envelope — either raw (`workflow_stage: workflow.status[0].stage`) or, more in the spirit of D8's "never ship a raw resolution input when its resolved output is on the response," a resolved `workflow_closed: stage ∈ {completed, cancelled}` boolean (that disjunction is exactly what both banners compute, `workflow-action-edit.yaml:93–100`). Then: (i) extend the deletion bullet and read-methods §4 to name the two static pages alongside the four templates, and (ii) record the field in the D8 envelope allowlist and the response table. Task 10's claim that `get_workflow.yaml`'s "only consumers are these templates" (`tasks/10:51–53`) is wrong on the same point and needs the same correction.

### 2. The envelope's per-action `title` has no source — action docs don't carry one, authors don't write one, and the validated config doesn't gain one

> **Resolved (fully — including the broader title-vs-message question).** Confirmed the gap (action docs carry no `title`; `get_action.title` on the static view header is a broken read). Resolved by reusing the canonical per-action display string `message` (`action.{app_name}.message`, engine-stamped, already on tree + timeline) for the shared detail header, instead of introducing a second display string — the unprovisioned action `title` is dropped from the envelope (D8) and task 5; task 10 repoints the view header's read at `message`. The neighbouring finding that an envelope `title` would need to serve _every_ shared/bespoke surface (not just check) is satisfied — `message` already serves all of them. Workflow-level `title` (WORKFLOW_FIELDS) is unaffected — a separate concern.
>
> The **broader** question — whether the per-verb `pages.{verb}.title` (generated form pages) earns its place against `message`, or `message` should be the single per-action display string everywhere — was briefly held as an open question, then resolved (2026-06-10) by auditing the reference implementation (the production app the module is extracted from; ~50 action configs across 10 workflows). **Both decisions stand, one per kind:**
>
> - **Form actions keep `pages.{verb}.title`.** 30 of 51 multi-verb actions differentiate titles per verb, with a systematic register per verb (edit = imperative "Capture Initial Details" / "Upload Waiver"; view = descriptive "Device Move Initial Details"; review = "Review Site Verification Check"; error = "Technician On Site Issue Resolution"). The page-title and status-`message` string sets are fully disjoint in the reference — messages are stage-keyed, punctuated, often Nunjucks-templated status sentences ("Awaiting installation of device {{ physical_id }}."), the wrong shape and stability for a bespoke page heading. The remaining 21 actions repeat one noun-phrase title across verbs — a convenience an action-level default could serve, but not added speculatively.
> - **Check/external shared surfaces keep `message` as the header — no `title` field is introduced.** The form-kind evidence doesn't transfer: the shared `workflow-action-{verb}` pages statically know their own verb (verb framing is page chrome, not per-action config); check messages _are_ the step statement (the authoring spec's own examples are imperative task copy); instanced actions get per-instance identity via message templating, which a derived type-title can't carry; and the status-map binding rule (no cell → invisible) guarantees a message wherever the action is reachable. `kind: external` has no user-facing surface and inherits the generic view page's behaviour. Escape hatch if a real consumer ever needs a stable noun-phrase header: an optional action-root `title:` — deferred until that need actually surfaces (build-for-what-exists).

D8 lists `title` among "the config-derived display fields the surfaces read (`title`, `required_after_close`)", and task 5 sources it "from the validated action/workflow config". Traced end-to-end, no such source exists:

- **Not on the doc.** The engine writes no `title` anywhere (zero hits across `planActionTransition.js` / `StartWorkflow.js` / the planners). The one surface read the design's claim points at — `workflow-action-view.yaml:75`, `_request: get_action.title` — is a **broken read today** (the field doesn't exist, and `get_action` returns an aggregation array, so even an existing field wouldn't resolve object-style).
- **Not in authored config.** Demo actions author no root `title` (checked all of `onboarding/`/`company-setup/`); per the authoring spec, action titles live only under **`pages.{verb}.title`** (`action-authoring/spec.md:250–264`) — a per-verb, build-time field that `makeWorkflowsConfig` deliberately excludes from the validated config (`makeWorkflowsConfig.js:3–6`: "Build-time-only fields (form, form_review, form_error, **pages**, hooks, event) are excluded"). A single envelope `title` can't be derived from per-verb page titles anyway ("Capture Initial Details" vs "Initial Details" vs "Review Initial Details").
- **Not added by this part.** "Validated config additions" adds `title` to `WORKFLOW_FIELDS` only; `ACTION_FIELDS` (`makeWorkflowsConfig.js:7–18`) has no `title` and the design doesn't add one.

So as designed, `GetWorkflowAction` cannot return an action `title`, and the one consumer that wants it (the view page's `action_title` header — and Part 40's check-surface header, which as a shared page has no baked `page_config.title`) stays empty. **Fix — decide the source:** (a) introduce an authored action-root `title` (validated, picked into `ACTION_FIELDS`, with a documented fallback — e.g. the action `type` prettified — for actions that don't set it), and state it in "Validated config additions"; or (b) drop `title` from the envelope and leave titles where they live today (baked `page_config.title` on generated pages), explicitly deferring the check-modal header source to Part 40. Option (a) is the one consistent with this part's thesis (the API feeds the page); either way the current text promises a field the design never provisions.

## Accuracy — D6's current-state claim and the unowned timeline surface

### 3. "No-op on current data" is false for the demo lead-view, and the design omits the replacement surface its own task list builds

> **Resolved.** Verified: the onboarding workflow targets `leads-collection`/`lead_ids`, the engine writes `entity_ref_key` into event docs (`StartWorkflow.js:163`), and lead-view reads `reference_field: lead_ids` — so started workflows do put action cards there; the no-op claim was false for that consumer. Corrected the D6 sentence to split the four consumers two ways (no-op for `contacts`/`companies` `tile_events` + `activities/pages/view`; lead-view is a real consumer that migrates). Folded the replacement surface into the design (was task-11-only): added the exported `modules/workflows/components/workflows-events-timeline.yaml` (`EventsTimeline` block backed by `GetEventsTimeline`), its manifest export, and the demo lead-view migration to "The read methods" and the Part 42 ripple — so `GetEventsTimeline` no longer ships with zero consumers. Design now matches task 11.

D6 justifies removing the inline lookup from the generic events timeline with: the four shipped consumers "lose action cards. No **workflow** consumer exists to migrate, and no workflow events reference those entities' actions today, so it is a no-op on current data." For one of the four named consumers — `apps/demo/pages/leads/lead-view.yaml` — both halves are wrong:

- The demo's onboarding workflow targets that exact entity: `entity_collection: leads-collection`, `entity_ref_key: lead_ids` (`onboarding/onboarding.yaml:3–4`), and the engine writes `entity_ref_key` references into every workflow event doc precisely "so events surface on the entity" (`WorkflowAPI/schema.js:83`, `StartWorkflow.js:163`).
- Lead-view's timeline reads `reference_field: lead_ids` (`lead-view.yaml:113`).

So any started demo workflow puts workflow events — and their action cards — on the lead-view timeline; this is the very surface Part 42 (shipped) demos through. Lead-view **is** a workflow consumer to migrate, not a bystander whose cards happen to vanish.

The task list already knows this: task 11 creates a new workflows component (`components/workflows-events-timeline.yaml` — same `EventsTimeline` block, fed by `GetEventsTimeline`), exports it in the manifest, and migrates the demo lead-view to it. **None of that is in the design.** D6 says adoption is "out of scope here"; the method table lists only a "_new_ workflows events-timeline request" with no component, no export, no consumer. That's a design-vs-task contradiction (designs are the source of truth), and as the design stands, `GetEventsTimeline` ships with **zero consumers** — a method with no surface.

**Fix.** Fold the workflows-provided timeline surface into the design: the new exported component, the manifest export, and the demo lead-view migration (Proposed change / "The read methods" / Ripples + the current-state table). Correct the D6 sentence to scope the no-op claim to the three consumers it's true for (`contacts`/`companies` `tile_events`, `activities/pages/view`) and state plainly that the demo lead-view migrates to the new surface in this part. (Note consistency-3 marked task 11 "no drift" — it drifted in the right direction; the design is the side to update.)
