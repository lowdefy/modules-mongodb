# Rename `kind: task` → `kind: simple`

The workflow-action kind currently spelled `task` is renamed to `simple` across shipped code, shipped templates, the demo app's `workflow_config`, the three shared pages (`task-edit` / `task-view` / `task-review` → `simple-edit` / `simple-view` / `simple-review`), and the active follow-on parts that key off the name. The word "task" is being freed for the adhoc-todo concept owned by a future tasks module, which will share the `actions` collection with workflows ([tasks-module-plan](../../../../workflows-module-concept/tasks-module-plan/design.md)). The rename is purely a vocabulary swap — no behavioural change, no data migration (the module hasn't shipped to production).

## Proposed change

1. **Rename the kind value.** `ACTION_KINDS` in [makeWorkflowsConfig.js](../../../../../modules/workflows/resolvers/makeWorkflowsConfig.js) becomes `['form', 'simple', 'tracker']`; every shipped check on `action.kind === 'task'` flips to `'simple'`; the `ActionKind` JSDoc typedef in [plugins/.../shared/types.js](../../../../../plugins/modules-mongodb-plugins/src/connections/shared/types.js) flips correspondingly. The validator's error messages mentioning `task` are updated.
2. **Rename the three shared pages** in [modules/workflows/pages/](../../../../modules/workflows/pages/): `task-edit.yaml` → `simple-edit.yaml`, `task-view.yaml` → `simple-view.yaml`, `task-review.yaml` → `simple-review.yaml`. The `id:` inside each page's `_ref: layout/page` vars block flips with the filename. The manifest's [`pages:` list](../../../../../modules/workflows/module.lowdefy.yaml) flips its three `_ref:` entries to the new filenames.
3. **Update every shipped reference to the page IDs.** The demo's `schedule-followup.yaml` uses `_module.pageId: { id: task-edit, module: workflows }` — those flip to `id: simple-edit`. Header comments in `universal-fields.yaml` and the README's shared-pages table all flip to the new IDs.
4. **Update the demo's workflow_config.** Two demo files declare `kind: task`: `installation/install-step.yaml` and `onboarding/schedule-followup.yaml`. Both flip to `kind: simple`. The demo's `schedule-followup.yaml` link cells also flip per item 3.
5. **Update every shipped test.** Test fixtures in `modules/workflows/resolvers/*.test.js` and `plugins/.../WorkflowAPI/**/*.test.js` that seed actions with `kind: "task"` flip to `kind: "simple"`. Strict-string assertions in `makeWorkflowsConfig.test.js`'s validation cases flip to the new error wording.
6. **Touch the active follow-on parts that key off the name.** Parts 22, 24, 28, 33, 34 each reference `kind: task` or the shared page IDs in their design.md. The design.md files flip in-place; review files stay frozen (historical record). (Part 30 — status-map-rendering — is deliberately excluded: [Part 38](../38-engine-rebuild/design.md) supersedes it and moves it to `_rejected/`; see Out of scope.)

## Why a dedicated part

The rename is mechanical but touches: 2 shipped resolvers, 3 page files (renamed + ID-flipped), 1 plugin typedef, 1 manifest entry, 1 README, 5 active-part design.md files, 2 demo workflow_config files, and ~10 test files. Bundling that into any of the active follow-on parts (24, 28, 33, 34) would muddy that part's review with mechanical churn unrelated to its substantive change. Tasks-module-plan's [Open question 1](../../../../workflows-module-concept/tasks-module-plan/design.md) already settled this: standalone, S-sized, runs in parallel.

The rename must land before any host app onboards the module to production. After that, customer YAML files would carry `kind: task` and the cost of the rename grows.

## Why `simple`

`simple` reads alongside the existing kinds — `form` captures input, `simple` captures nothing, `tracker` subscribes, `custom` (Part 28) delegates to the app. The differentiator across the kind enum is "what surface the action presents to the user, and who advances its status." `simple` is the no-surface, user-advances slot.

Rejected alternatives (carried forward from [tasks-module-plan § The `kind: task` → `kind: simple` rename](../../../../workflows-module-concept/tasks-module-plan/design.md)):

- `manual` — misleading; `form` actions are also user-driven. The differentiator is "has input surface" vs "no input surface", not "human vs automated".
- `step` — every workflow action is a step; undifferentiated.
- `status` — every kind transitions status; undifferentiated.
- `user_task` (BPMN's term) — verbose without adding clarity over `simple`.
- `mark` — reads narrower than the semantics (the action also writes universal fields, not just a status flip).

## Out of scope

- **Anything tasks-module-specific.** The tasks-module-plan concept doc remains a _concept_ — its own module manifest, `create-task` API, kanban view, doc-level access model, and external-creation SDK are deferred to a future tasks-module implementation design. This part only frees the word "task" by renaming the workflow-action kind; it does not add the adhoc concept.
- **Tightening or relaxing the `actions` collection schema.** The schema constraints the tasks-module-plan calls out (no required `workflow_id`, no required `type`, nullable `entity_id` / `entity_collection`) are already true in shipped code (verified — `modules/workflows/connections/actions-collection.yaml` is a bare `MongoDBCollection` with no validator). No code change here, just a constraint to keep true.
- **Data migration.** None needed. The workflows module is pre-production; no host app's `actions` collection has docs with `kind: "task"` written by shipped code.
- **Changes to behavior of the simple kind.** This is a pure vocabulary swap. Status semantics, page composition, endpoint emission, hook surface, status-map defaults, and access-model treatment for the kind formerly known as `task` are unchanged.
- **Editing the tasks-module reservation note in the concept docs.** The line at `workflows-module-concept/design.md:314` describing `kind: task` as reserved for the future tasks module must stay — anyone running a terminology sweep across `workflows-module-concept/` will hit it and should leave it alone.
- **Renaming Part 30 (status-map-rendering).** [Part 38 (engine-rebuild)](../38-engine-rebuild/design.md) supersedes Part 30 and moves it to `_rejected/`. Renaming `kind: task` / the `task-*` page IDs across a design slated for rejection is wasted churn — the same reason `_completed/` parts are excluded. Part 30's design.md and `tasks/` rename surface (formerly the largest single cluster in this part) is therefore dropped. If Part 38 stalls and Part 30 is ever revived, its rename folds into whichever part revives it.
- **Adding `kind: task` to the validator as a reserved (rejected) value.** The plan's spec ([action-authoring/spec.md](../../../../workflows-module-concept/action-authoring/spec.md)) reserves `kind: task` for the future tasks module and says workflow-config validation rejects it. The current `ACTION_KINDS = ['form', 'simple', 'tracker']` already produces an "unknown kind" error for any unrecognized value, including `task`. A more specific message ("`kind: task` is reserved for the future tasks module — use `kind: simple` for the no-input workflow action") is a nice-to-have not committed here; it can ship as a one-line follow-on if the generic error proves confusing to authors mid-migration on an external workflow library.
