# `actions` and `workflows` Collection Indexes

The workflows module ships two Mongo collection connections (`actions-collection`, `workflows-collection`) and the engine + read-side APIs hit them on every entity render, every workflow render, and every action submit. Today the module documents no required indexes for either collection — host apps left to discover what to add via slow queries in production. This part closes the gap by documenting the minimum index set the shipped queries require, and pins down the schema-shape constraints that keep those indexes compatible with the future tasks module sharing the `actions` collection ([tasks-module-plan](../../../../workflows-module-concept/tasks-module-plan/design.md) § Constraints this places on the workflows module). No new code — this part adds documentation and verifies the implementation against it.

## Proposed change

1. **Document required indexes in [`modules/workflows/README.md`](../../../../../modules/workflows/README.md).** Add a dedicated top-level `## Indexes` section — placed after `## Exports`, before `## Vars` — listing two indexes the module's shipped queries assume — `actions.{ workflow_id: 1 }` and `workflows.{ entity_collection: 1, entity_id: 1 }`. `activities/README.md` is the only other module documenting per-collection indexes today, but it does so as a prose paragraph inside `## Notes` (`contact_ids` / `company_ids`). This part deliberately uses a standalone section instead: two indexes plus three constraints (non-partial, no validator, schema-shape) read better as a discrete block, and the section is amended into the repo's fixed README template (item 5) so the convention is mechanically enforced rather than diverging per module.
2. **Affirm the schema-shape constraint** that the tasks-module-plan calls out: the `actions` collection must remain free of collection-level required-field validators beyond `_id`, `kind`, `status`, `change_stamp`. The shipped `connections/actions-collection.yaml` is a `MongoDBCollection` carrying `changeLog` + `write: true` but no `validator:` — verified, no change. The README's new `## Indexes` section names this constraint so anyone adding a Mongo validator later sees the reason it's forbidden (future `kind: task` adhoc docs with `workflow_id: null` and no `type`).
3. **Affirm non-partial indexing.** Both new indexes are non-partial — they index `null` values for fields that the future tasks module will leave null. The README states this explicitly so a future contributor doesn't "optimise" by re-issuing them as partial indexes filtered on `workflow_id` / `entity_id` existing, which would silently break tasks-module queries that share an index path.
4. **Correct one factual error in the tasks-module-plan concept doc.** The plan's "Engine boundary" section (line 172 of [`workflows-module-concept/tasks-module-plan/design.md`](../../../../workflows-module-concept/tasks-module-plan/design.md)) says "The tracker subscription queries the `actions` collection looking for `child_workflow_id` matches." It does not — `fireTrackerSubscription.js:46-56` looks up the workflow by `_id`, reads its `parent_action_id`, then fetches that action by `_id`. The constraint the plan is asserting (tasks are transparent to the tracker subscription) still holds; the _mechanism_ is via `workflow.parent_action_id`, not via a query on `actions.child_workflow_id`. Flip the wording in the concept doc.
5. **Amend the fixed README template in [`CLAUDE.md`](../../../../../CLAUDE.md).** The Documentation section pins a fixed per-module README template (line 42: "Description, Dependencies, How to Use, Exports (…), Vars, Secrets, Plugins, Notes"). Adding a standalone `## Indexes` section (item 1) without updating that list would silently break the "fixed template" contract. Insert `Indexes` between `Exports (…)` and `Vars` so the template matches the section's placement, making the convention mandatory for any future module that documents required indexes.

## The index set — query coverage analysis

Walked through every read site and write site on `actions` and `workflows` in the shipped module + plugin code. Two indexes cover all hot paths; one engine field (`workflow.parent_action_id`) is queried only via `_id` and needs no secondary index.

### `actions.{ workflow_id: 1 }`

| Site                                                                                                                                        | Op                | Match shape                                                                 | Hot path?                                                                      |
| ------------------------------------------------------------------------------------------------------------------------------------------- | ----------------- | --------------------------------------------------------------------------- | ------------------------------------------------------------------------------ |
| [`get-workflow-overview.yaml`](../../../../../modules/workflows/api/get-workflow-overview.yaml) `$lookup foreignField: workflow_id`         | aggregate         | `{ workflow_id: <wf._id> }`                                                 | yes — every workflow overview load                                             |
| [`get-action-group-overview.yaml`](../../../../../modules/workflows/api/get-action-group-overview.yaml) `$lookup foreignField: workflow_id` | aggregate         | `{ workflow_id: <wf._id> }` (with sub-pipeline filter on `action_group`)    | yes — every group overview load                                                |
| [`get-entity-workflows.yaml`](../../../../../modules/workflows/api/get-entity-workflows.yaml) `$lookup foreignField: workflow_id`           | aggregate         | `{ workflow_id: <wf._id> }` once per workflow returned for the entity       | yes — every entity page render                                                 |
| [`plugins/.../connections/shared/getActions.js`](../../../../../plugins/modules-mongodb-plugins/src/connections/shared/getActions.js)       | find              | `{ workflow_id: workflowId }`                                               | yes — invoked by `recomputeWorkflowAfterActionWrite` after every action submit |
| `plugins/.../CancelWorkflow/CancelWorkflow.js`                                                                                              | find / updateMany | `{ workflow_id }` plus `{ workflow_id, "status.0.stage": { $nin: [...] } }` | warm — admin cancel                                                            |
| `plugins/.../CloseWorkflow/CloseWorkflow.js`                                                                                                | find / updateMany | same shape                                                                  | warm — admin close                                                             |

Equality on `workflow_id` is the only useful key. The per-workflow `$sort` keys inside the `$lookup` sub-pipelines are a mix of pipeline-computed fields (`groupIndex`, `required_sort`, `sort`) and stored fields (`created.timestamp`, `_id`) — none indexable here, so Mongo does an in-memory sort on the small per-workflow result set (typically <30 actions). A compound `{ workflow_id, action_group }` is tempting for the group-overview lookup but the equality on `workflow_id` already narrows to a handful of docs; the second key wouldn't change the plan.

**Non-partial.** Future `kind: task` adhoc docs carry `workflow_id: null`. A non-partial `{ workflow_id: 1 }` index includes the null entries, costs nothing at read time for the workflow-stream queries (they all filter by a concrete `<wf._id>`), and remains usable if a future tasks-module query joins on `workflow_id` (e.g., "fetch actions for _this_ workflow plus loose tasks referencing the same entity").

### `workflows.{ entity_collection: 1, entity_id: 1 }`

| Site                                                                                                   | Op        | Match shape                                                                                                | Hot path?                                              |
| ------------------------------------------------------------------------------------------------------ | --------- | ---------------------------------------------------------------------------------------------------------- | ------------------------------------------------------ |
| [`get-entity-workflows.yaml`](../../../../../modules/workflows/api/get-entity-workflows.yaml) `$match` | aggregate | `{ entity_collection: <coll>, entity_id: <id> }` then `$sort: { display_order: 1, created.timestamp: -1 }` | yes — every entity page render via `actions-on-entity` |

The compound `{ entity_collection: 1, entity_id: 1 }` matches the equality prefix exactly. Per-entity workflow counts are small (entities typically carry single-digit workflow rows in shipped apps), so the post-match in-memory sort on `display_order` + `created.timestamp` is cheap; an extended `{ entity_collection: 1, entity_id: 1, display_order: 1 }` is a future-proofing knob, not a hot-path need.

### Indexes considered and not shipped

| Candidate                                                             | Why not                                                                                                                                                                                                                                                                                                                               |
| --------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `actions.{ "assignees.user.id": 1, "status.0.stage": 1 }`             | No "my-work" view shipped in workflows module v1. This index belongs to the future tasks module; calling it out now would be the speculation the repo's CLAUDE.md warns against (`Build for what exists, not what might`). The future tasks-module design owns it and the choice between non-partial and per-stream-partial indexing. |
| `actions.{ child_workflow_id: 1 }`                                    | The tracker-subscription claim in the concept doc is wrong — `fireTrackerSubscription.js` never queries by `child_workflow_id`; it joins via `workflow.parent_action_id → action._id`. The field is written on tracker actions for display/UI but not used as a query key. (See item 4 in Proposed change.)                           |
| `actions.{ action_group: 1 }` filtered by workflow                    | `get-action-group-overview`'s sub-pipeline filters the `$lookup` results, not the collection — the equality on `workflow_id` already scopes to <30 docs and an in-memory filter on `action_group` is cheap.                                                                                                                           |
| `actions.{ status.0.stage: 1 }`                                       | Used only inside the `workflow_id`-scoped admin queries (`CancelWorkflow`, `CloseWorkflow`); equality on `workflow_id` is the selective key. A separate global status index would only help a "my-open-actions" cross-workflow scan that no shipped surface performs.                                                                 |
| `workflows.{ parent_action_id: 1 }`                                   | The tracker subscription queries by `_id`, not by `parent_action_id` — the latter is read off the returned doc. No secondary index needed.                                                                                                                                                                                            |
| Compound `workflows.{ entity_collection, entity_id, status.0.stage }` | Listing pages today filter status client-side after the small per-entity fetch; no shipped query pushes the status filter into Mongo.                                                                                                                                                                                                 |

## Why a dedicated part

Three reasons it isn't folded into another active part:

1. **Cross-cuts the concept docs.** The tasks-module-plan is the source of the constraint ("no index assumes `workflow_id` is present"). Folding the README change into one of the active implementation parts (24, 28, 30, 33, 34, 36) would muddy that part's review with index-doc churn unrelated to its substantive change.
2. **One-shot README contribution.** Once the indexes are documented they don't move — future parts that add new query shapes can extend the list, but the structure of the `## Indexes` section is fixed by this part.
3. **Surfaces a concept-doc correction.** The tracker-subscription claim in the plan needs a small flip and this part is the right place for it (the reviewer for this part is already in the indexes-and-engine-mechanism mindset).

## Out of scope

- **A migration that creates the indexes in consumer apps.** Index creation is a host-app concern. Consumer apps in the org typically use the `splice-actions` MongoDB index pattern (see CLAUDE.md's `/r:index-dev` skill); each host app declares its own index definitions referencing this README. Shipping an index-creation migration in the module itself would entangle the module with the host app's deploy pipeline.
- **Tasks-module-specific indexes.** Anything keyed on `assignees`, the adhoc-only "my-work" view, the tasks-side timeline reads, or cross-stream "all my work" aggregations belongs to the future tasks-module design. The constraint this part affirms — non-partial workflow-stream indexes that don't exclude `workflow_id: null` docs — is what makes the tasks-module's later index choices unblocked.
- **Adding indexes to the `events` collection** (which the comment-rendering and event-timeline paths read by `references.action_ids` / `references.workflow_ids`). Owned by the events module and `parts/33-comment-rendering/`.
- **A `validator:` on the `actions` collection.** The schema-shape constraint is the _absence_ of one. If a future part wants to enforce field-level invariants (e.g., "every action has a `kind` in [form, simple, tracker, custom, external, task]"), it does so in the write APIs, not via a Mongo collection validator that would block the tasks-module write path.
- **Indexes the demo app uses internally** beyond `actions` and `workflows` — owned by app-level config, not the module.
