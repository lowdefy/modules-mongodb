- [x] **F22** ‚Äî Timeline cards navigate instead of opening the check-action modal. **Re-decided (supersedes the earlier `on_action_click`-passthrough call).** The handler is now **baked into both hosts by default ‚Äî no consumer onclick, ever.** The old decision rested on a false premise: it claimed the timeline wrapper "must not couple to workflow check actions" to preserve the `workflows ‚Üí events` direction. That conflates the **block** with the **wrapper**. The `EventsTimeline` *block* stays generic ‚Äî it only fires `{ name: onActionClick, event: { action } }`, identical to `ActionSteps`, and the engine already puts `{ _id, kind, link, status, message }` on every card (`GetEventsTimeline.js:25`). The handler lives in `workflows-events-timeline.yaml`, which is **already a workflows-side component** (it already `_ref`s `action_statuses.yaml`), so baking the handler in costs nothing on the dependency direction. **The consumer never needs to write `on_action_click`.**

  **Design (Option A ‚Äî singleton modal + one flag):**
  - **Shared handler, baked in.** Extract the canonical kind-branch handler (`check` ‚Üí `SetState check_action_modal.action_id` + `CallMethod setOpen`; else ‚Üí `Link` to `action.link`) into a shared **`components/check-action-click.yaml`** `_ref`, and wire it **by default** into *both* `actions-on-entity.yaml` (replacing its inline copy at lines 73-105) and `workflows-events-timeline.yaml`. Both blocks emit the same `{ action }` payload, so one handler serves both.
  - **Delete `on_action_click`.** No current consumer needs a custom handler; the passthrough var (and its load-bearing `_build.if` null-guard in the wrapper, lines 64-78) is removed from the component and the manifest. Build for the concrete case, not the escape hatch.
  - **`actions-on-entity` keeps mounting the singleton modal** (unchanged ‚Äî its fixed `check_action_modal` blockId, dropped once per page). The singleton constraint is real: the modal's component-local `get_workflow_action` request id isn't scoped, so two instances = duplicate-id build error; and `CallMethod` *throws* on a missing block (`createCallMethod.js:23` reads `.methods` off `undefined`), so the open-contract is only safe when a modal is present.
  - **New `include_modal` flag on `workflows-events-timeline`** (default `false`): when `true`, the wrapper bundles the `check-action-modal` `_ref` itself ‚Äî for **timeline-only pages** (no action surface). The baked handler always opens `check_action_modal`.
  - **Result:** the canonical entity page (action surface + timeline) is **zero-config** ‚Äî `actions-on-entity` mounts the modal, both surfaces' baked handlers open it. A timeline-only page sets `include_modal: true` (one flag). **Residual footgun:** a timeline-only page that opens a check while neither the flag nor `actions-on-entity` provides a modal will throw on click ‚Äî rare, dev-time-obvious; document the rule "set `include_modal: true` on a timeline that has no action surface" in the manifest + README.
  - **Demo fix:** `apps/demo/pages/leads/lead-view.yaml` already mounts `actions-on-entity` (line 63), so `check_action_modal` is present ‚Üí the timeline's check cards (lines 109-116) open the modal with **no wiring at all**. Leave `include_modal` at its default; nothing to add on the page.
