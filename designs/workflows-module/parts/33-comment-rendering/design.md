# Part 33 — Comments are standard events rendered on the events timeline

> **Re-baselined 2026-06 against shipped Parts 38/40/42/46/48/53.** The design's spine is intact — comment → `display.{app_name}.description`, one `foldCommentIntoEvent` helper called once post-render, deep-merge under the app key, comment-wins precedence. What moved underneath it since first draft: the engine title model is now a curated per-signal map (Part 53, not the old `{{ … marked … as … }}` auto-title); `event_overrides` provenance is Part 48's per-workflow write endpoints (not Part 13's `emitEventOverrides`); the action view page is a Part 40 thin container `_ref`-ing `check-action-surface.yaml` with the comment field at `current_action.comment`; the form review template now carries a **separate** mandatory `change_request_comment` (distinct from the optional `comment`); `get_action` was renamed to `get_workflow_action` and returns a **single object** (Part 46, no `.0`); and the events module's generic `events-timeline` is now **events-only** (Part 46 moved action-card enrichment into the workflows-owned `GetEventsTimeline` / `workflows-events-timeline`). The decisions below and the line references have been refreshed to that baseline. Several engine-side pieces are **already half-landed**: `planEventDispatch` already takes a `comment` param (un-folded), already never writes `metadata.comment`, and already carries Part 24's `UpdateActionFields` handler type — so this part now only adds the fold and both paths inherit it.

A workflow submit (and the Part 24 fields operation) captures an optional rich-text `comment`. Today that comment is written to `event.metadata.comment` and rendered by a **bespoke "Comments" card** that runs its own query — while the standard `EventsTimeline` never surfaces it, and the workflow view pages don't even render the standard timeline. This part makes the comment a first-class part of the standard event: the engine writes the comment HTML into the event's existing **`display.{app_name}.description`** channel, the standard timeline renders it inline under the auto-generated title, and the bespoke comments card is deleted.

## Proposed change

1. **The comment becomes the event's `description`.** The engine writes the comment's HTML into `display.{app_name}.description` — the secondary rich-text channel the timeline already renders (`EventsTimeline.js:240-286`). No new render path, no new event field.
2. **`metadata.comment` is dropped.** The comment is no longer stored under `metadata`. The event carries it once, in `display.{app_name}.description`. No text shadow in v1.
3. **A single shared helper folds the comment into the event** for both write paths — `SubmitWorkflowAction` (the default log event) and Part 24's `UpdateActionFields` (`action-fields-updated`) — so the two cannot drift.
4. **The description channel is owned by the runtime comment.** Authors override per-app **title** (D7); they do not set a static `display.{app}.description` — the comment is the only writer of that slot. An authored `description` is rejected at build, a pre-hook `description` is stripped at merge, and the comment is folded last; a comment-less event simply has no description. This answers the Part 32 question by removing the comment-vs-static contention entirely (the static *title* channel stays; description is comment-only).
5. **The bespoke Comments card is deleted, and the standard events timeline is added to the workflow action pages.** `workflow-action-view.yaml`'s `comments_card` (and its `get_comment_events` query / `comment_events_list` state) is removed and replaced with the shared `events-timeline` component, filtered to the action — and the same timeline `_ref` is added to all four generated form-kind templates (`edit`/`view`/`review`/`error`), which have no comment surface today. The action pages now show the full activity timeline — comments included inline — instead of a comment-only list (or, on form pages, nothing).

## Background: what the code actually does

- **Where the comment is captured.** A `TiptapInput` on the action surfaces. Post-Part-40 the check surface has **two** comment inputs: an **optional** surface comment bound to `current_action.comment` (`check-action-surface.yaml:268`, optional placeholder) shown in edit/review modes and at the `error` stage, and a **separate, mandatory** comment inside the Request Changes modal (`check-action-surface.yaml:626`, `required: true`) — which today shares the same `current_action.comment` id as the optional input (this part splits it to its own path; D5). The form review template mirrors that with an **optional** `comment` (`review.yaml.njk:171`) and a **separate, mandatory** `change_request_comment` (`review.yaml.njk:380`, `required: true`) inside its Request Changes modal. `TiptapInput` writes `{ html, text, markdown, fileList }` to its state path (`@lowdefy/blocks-tiptap` `useTiptapState.js:49-52`).
- **Where the comment is sent.** Every path posts the **whole TipTap object** under the `comment` payload key — the check surface's signal buttons post `comment: { _state: current_action.comment }` (the optional comment) and its Request Changes modal posts the mandatory comment (split to `current_action.change_request_comment` by this part), the form review modal posts `comment: { _state: change_request_comment }`, the regular form submit posts `comment: { _state: comment }`. `makeWorkflowApis` maps `comment: { _payload: 'comment' }` onto the handler regardless of source state key.
- **Where the comment is stored today (pre-Part-38).** The current `SubmitWorkflowAction` threads `params.comment` into `buildDefaultLogEventPayload`, which writes `metadata.comment` **only when `typeof comment === "string"`** (`dispatchLogEvent.js:66`). Because the page sends an object, the guard is `false` and the comment is dropped on the floor — a latent bug. [Part 38](../_completed/38-engine-rebuild/design.md) rebuilds these call sites into shared load→plan→commit planners; this part lands on that new structure, defining the comment's home (`display.{app_name}.description`) and pinning the wire contract there, so the bug doesn't carry forward.
- **The render channel.** The `EventsTimeline` block (now in `plugins/.../blocks/EventsTimeline/`) renders exactly one secondary rich-text slot: `EventDescription` → `dangerouslySetInnerHTML={{ __html: sanitize(event.description) }}` (`EventsTimeline.js:240-286`). Both timeline data sources map `display.{app_name}.description → description`: the events module's generic `events-timeline` aggregation (`events-timeline.yaml:43-50`) and the workflows-owned `GetEventsTimeline` engine method (`GetEventsTimeline.js:211`). So `event.description` must be an **HTML string**, sanitized at render — true on both surfaces.
- **Title resolution is at write time, in the engine.** Per [Part 38](../_completed/38-engine-rebuild/design.md), the engine renders event display as plain Nunjucks **strings** during the plan phase — the engine-default title template plus any author/pre-hook overrides — so the timeline receives already-rendered plain strings. Part 53 replaced the old generic auto-title with a **curated per-signal title map** (`DEFAULT_SIGNAL_TITLES` / `LIFECYCLE_TITLES`, `planEventDispatch.js:14-57`), composed over the denormalised `{{ action.title }}` / `{{ workflow.title }}`. The comment is raw HTML (not a template); it is folded into the rendered display with no read-time templating.
- **The timeline isn't on the workflow action pages.** Post-Part-40 `workflow-action-view.yaml` is a thin container that `_ref`s `check-action-surface.yaml` (status badge + working inputs) and then renders a page-level bespoke `comments_card` below it (querying `metadata.comment: { $exists: true }`) — explicitly tagged in the page header as "kept verbatim pending Part 33." It does **not** compose any `events-timeline`. Workflow events carry `references.{entity_ref}: [entity_id]`, so they surface on the **entity** page timeline (via `workflows-events-timeline`) — but the action's own page has no general timeline.

## Key decisions

### D1 — Comment renders via `display.{app_name}.description`, not a new field

The timeline already has a title + secondary-rich-text model. The engine title (the curated per-signal template, e.g. `request_changes` → `'{{ user.profile.name }} requested changes on {{ action.title }}'`, `planEventDispatch.js:29-57`) is the headline; the comment is the body. This is exactly what `EventDescription` renders. Reusing `description` means **zero block changes** — the comment shows up the moment the field is populated. A `request_changes` row becomes: title "Sam requested changes on Review Onboarding" + the comment HTML in the card below it.

### D2 — Drop `metadata.comment` entirely

The comment lives once, in `display.{app_name}.description` (an HTML string). No `metadata.comment`, no `{ text, html }` shadow.

- **Why no text shadow:** the only consumers proposed for a text shadow were search and email rendering — both out of scope here (email is a Part 33 non-goal; no search-over-comments need exists). Per "build for what exists," it's dropped. If full-text search over comments surfaces later, add a `text` projection then — it's additive.
- **Migration:** existing events with `metadata.comment` keep it (harmless, unread after the tile is deleted). No backfill. New events write `display.description` only. (A one-shot migration to lift legacy `metadata.comment` into `display.{app}.description` is possible but not required for v1 — call it out as deferred.)

### D3 — One shared helper, applied in both write paths

Both the submit pipeline and Part 24's `UpdateActionFields` accept a `comment` and emit an event. The "fold comment into the event's description" logic is a single pure helper — `foldCommentIntoEvent(eventPayload, comment, appName)` — with exactly **one call site**: the shared event-dispatch planner (`planEventDispatch`) both paths route through, so the behaviour can't diverge. It:

- reads the comment's `html` (accepting the `{ html, text, fileList }` rich-text value; ignores `markdown`),
- ensures the `display[appName]` bucket exists before writing (`display[appName] ??= {}`) — a pre-hook/author override merge can produce a `display` without the app key (D7), so writing `display[appName].description` directly would throw,
- sets `display[appName].description = comment.html` when the comment is **non-empty**: `comment?.text` is a non-empty string **or** `comment?.fileList` is a non-empty array. The gate reads the editor's own emptiness signals, not `html` — TipTap emits `'<p></p>'` for an empty document and nulls only `text` (`useTiptapState.js:44-52`), so a type-then-delete leaves `{ html: '<p></p>', text: null, fileList: [] }` and a gate on `html` would store `'<p></p>'` and render an empty description card. The `fileList` clause keeps an image-only comment (screenshot, no text → `text: null`, `fileList` populated) from being silently dropped,
- is a no-op when there's no comment (null, or empty by the gate above).

This supersedes the pre-Part-38 inline `metadata.comment` write (in `dispatchLogEvent.js`'s `buildDefaultLogEventPayload`, both deleted in Part 38). Post-Part-38 the helper is called once inside the shared event-dispatch planner (`planEventDispatch`) that both the submit and `UpdateActionFields` paths reuse, so the two cannot drift. `planEventDispatch` is a **closed enum** over handler types (throws on unknown). The `UpdateActionFields` handler type → `action-fields-updated` event type **already exists** in the planner (`planEventDispatch.js:178-183`, `isFieldsUpdate` branch) and `planFieldsUpdate` already threads `comment` through the same `comment` parameter the submit path uses — there is **no** separate per-event-builder call. The only missing link is the fold itself: once this part calls `foldCommentIntoEvent` after render, both paths fold automatically.

### D4 — The comment owns the description; folded last

The `display.{app}.description` slot is **owned by the runtime comment** — it is the only writer. Authors override per-app **title** (D7), not description. This is the resolution of the Part 32 deferred question: rather than letting a static author description and a runtime comment contend for the slot (and needing a precedence rule between them), the slot is reserved for the comment.

Post-[Part 48](../_completed/48-render-config-off-connection/design.md), `event_overrides` no longer ride the connection config blob: they live on the per-workflow write endpoints and are spliced onto `actionConfig` by `loadWorkflowState`, reaching the planner as `yamlEventOverrides` (`planEventDispatch.js:267-273`; `mergeEventOverrides.js:8`); a pre-hook can also return `event_overrides.display`. (This is **not** Part 32, which dropped the static *status* layer — a different mechanism.) Neither may write `description`: an authored `description` is **rejected at build** (validation, see Files changed) and any `description` arriving on the merged display (e.g. from a pre-hook) is **stripped during the merge**, so only the comment fold writes the slot.

Implementation: `foldCommentIntoEvent` runs **after `renderEventDisplay`**, on the rendered display — the sequence is merge (strips non-comment `description`) → render → fold (`planEventDispatch.js:266-289`). The fold must not sit between merge and render: `renderEventDisplay` compiles **every string** in the display tree as a Nunjucks template, so folding raw user-typed HTML pre-render would compile the comment — a comment containing `{{`/`{%` would throw (failing the submit) or interpolate against the render context (which carries the full `user`/`action`/`workflow` docs — a data-exposure path). The comment is raw HTML, stored verbatim, never templated (Background).

Precedence is per field:

- **title** — an author per-app override wins over the engine-generic title (D7), via the deep-merge under the app key;
- **description** — owned by the comment alone; there is no static author description to contend with it.

A consequence worth naming: a pre-hook `event_overrides` can **no longer override or scrub a typed comment**, nor inject its own description — the fold runs after the merge (last by design) and the merge strips any non-comment description. This is an intended behaviour change from the pre-Part-38 layering (where a pre-hook `metadata.comment` won), not collateral. The static `event:` channel **stays for the title**: authors set a per-app title that renders on every event — comment or not, including comment-less form submits and lifecycle events. A comment-less event simply has an empty description; there is no static description fallback. (Form submits **do** carry comments — an optional one on most signals and a mandatory one on `request_changes`; D5 — so the description is populated whenever a comment is typed, and empty otherwise.)

### D7 — Multi-app event display & per-app author title overrides

Event `display` is **app-keyed** (`display.{app}.{title,description,info}`): one event document is read by multiple apps' timelines, each keyed by its own `display_key` (the events-timeline `$<display_key>.title` projection — and an event surfaces in an app's timeline **only when `display.{that-app}` exists**, per the `$ne: null` filter). This is what makes workflows natively multi-app: a team app and a customer portal read the *same* event document but show different messaging.

The engine writes a generic default title for the submitting app. Authors override **per app** via `event.{interaction}.display.{app}.title` — **title only**; `description` is reserved for the runtime comment (D4) and rejected by the builder validation — e.g. an exact, user-named, detail-rich title for the team app and a generic/opaque one for the customer portal. These overrides are **plain Nunjucks template strings rendered by the engine** at plan/write time against the event render context — `user`, `action`, `workflow`, `signal`, `status_before`, `status_after`, `submitted_form` (`planEventDispatch.js:226-243`; lifecycle events get the smaller `user`/`workflow`/`signal` context). They are **not** `_nunjucks` operators and are not resolved at read time — matching [Part 38](../_completed/38-engine-rebuild/design.md)'s engine-rendered-display model.

For the engine title (or an author per-app title override) + the comment to coexist *within one app bucket* — and so an override targeting one app's `title` doesn't drop sibling engine-written keys in that bucket — the engine's event-display merge must **deep-merge under the app key** — two levels: `display → {app} → {title,description}` (`description` written only by the fold). Today's merge is **still one level deep** at `display` (`mergeEventOverrides.js`'s `overlayObject` → `{ ...base.display, ...override.display }`, `mergeEventOverrides.js:28-43`, called from `planEventDispatch.js:268`) — Parts 38/48 carried that shallow merge over unchanged — so an app-keyed override replaces the whole app bucket and drops the engine title. So **this part adds the deep-merge under the app key** — implemented by calling the shared `deepMerge` helper (`shared/phases/planners/deepMerge.js`, "shared by every planner that layers a patch onto loaded state") on the `display` channel inside `mergeEventOverrides`, not a bespoke second merge (per "one correct way"). While touching it, rewrite `mergeEventOverrides.js`'s docblock — it still documents the pre-rebuild world ("metadata.comment already folded into layer 1 … Do NOT re-inject `comment` here", `mergeEventOverrides.js:1-15`), which this part makes false twice over. With deep-merge in place, the comment folds in last (D4) without clobbering the title, and a comment-less form-submit event still renders the author's per-app title.

### D5 — Pin the wire contract: `comment` is the rich-text value, engine takes `.html`

The pages keep sending `comment: { _state: comment }` (the TipTap value). The engine contract is: `comment` is `{ html, text, fileList, … } | null`; the engine stores `comment.html`, gated on `comment.text` / `comment.fileList` (D3). This fixes the current `typeof comment === "string"` bug (which silently drops object comments) without changing every page's payload.

The `required` validation on `request_changes` does **not** guard emptiness today. Post-Part-40 there are two mandatory-`request_changes` comment inputs, and both gate on `_ne null` (which a type-then-deleted, non-null TipTap object passes):

- **Form review template** (`review.yaml.njk:380`): the `change_request_comment` `TiptapInput` has `required: true` plus a `validate` whose `pass` is `_ne: [change_request_comment, null]`. It is a **separate state path** from the template's optional `comment` (`review.yaml.njk:171`).
- **Check surface** (`check-action-surface.yaml:626`): the mandatory comment lives in the Request Changes modal (`request_changes_modal`, review mode only) — a `TiptapInput` with `required: true` and `validate.pass: _ne: [current_action.comment, null]`, enforced by the modal's `Validate { regex: ^current_action\.comment$ }` (`:599`). **It is *not* a separate path:** today it shares the id `current_action.comment` with the *optional* surface input at `:268`, so the modal's `Validate` (which matches blocks by id) fires the rules of **both** inputs. Adding a rule to the `:268` input — as an earlier draft proposed — would therefore fire that rule inside the modal too (blocking the optional surfaces) while leaving the real mandatory input still on its loose `_ne null` gate.

This part tightens **both surfaces** to mirror the fold's gate — pass when `comment.text` is a non-empty string or `comment.fileList` is non-empty — so a textually empty *mandatory* comment fails at the input with feedback, instead of submitting and then being (correctly) dropped by the engine gate with no comment stored. On the form template that's `change_request_comment`'s `validate.pass`. On the check surface, the shared id is the root problem, so this part **splits the modal input onto its own path** `current_action.change_request_comment` (mirroring the form template's two-path model) and tightens *its* existing `_ne null` validate to the fold gate — updating the modal's `Validate` regex, submit payload, `onClose` reset, and onMount seed to match. The optional `:268` input stays on `current_action.comment` and validate-free. This removes the shared-id coupling rather than relying on it staying benign.

### D6 — Delete the bespoke card; add the standard events timeline to the workflow action pages

Deleting `comments_card` without adding a timeline would leave the action's own view page with no comment surface (only the entity page would show it, via `references`). So this part:

- **removes** `workflow-action-view.yaml`'s page-level `comments_card` (the Part 40 chrome below the `check-action-surface` `_ref`), its `get_comment_events` request, the `metadata.comment: { $exists }` filter, and the `comment_events_list` state/list blocks;
- **adds** the standard `_ref: { module: events, component: events-timeline }`, filtered to the action (`reference_field: action_ids`, `reference_value: { _request: get_workflow_action._id }`), so the page shows the full activity stream — submits, status changes, comments inline — in one standard surface;
- **adds** the same action-filtered `events-timeline` `_ref` to **all four form-kind templates** (`templates/{edit,view,review,error}.yaml.njk`). None has a comment-rendering surface today, yet form actions capture the one *mandatory* comment — `change_request_comment` on the review template's Request Changes modal (`review.yaml.njk:380`) — which would otherwise render nowhere on the action's own pages. Every generated form page gets the timeline. The form templates load the action as a single object (`_request: get_workflow_action`, Part 46 — no `.0`) into `_state.action`, so the filter is `reference_value: { _state: action._id }`.

**Why the events module's generic `events-timeline`, not the workflows `workflows-events-timeline`.** Part 46 split the timeline in two: the events module's `events-timeline` is now **events-only** (a client-side aggregation, no action-card enrichment), and a new workflows-owned `workflows-events-timeline` (backed by the `GetEventsTimeline` engine method) carries the cross-stream **action cards** for entity-page timelines. On a single **action's own page** the timeline is filtered to that one action's id, so the only action card enrichment could ever produce is a self-referential card (which Part 42 D6 suppresses anyway) — action-card enrichment buys nothing here. The events-only generic timeline is the lighter, correct tool: it renders `display.{display_key}.description` (the comment) plus the event stream, with no engine method call, no `GetEventsTimeline` round trip, and no `check_action_modal` to keep clear of the pages' existing `get_workflow_action` request. (The rejected alternative — `workflows-events-timeline` with `include_modal: false` — would also render the comment, but adds the wasted enrichment and the fixed `get_events_timeline` request id for no gain on these pages.) Action cards remain the entity page's job, via `workflows-events-timeline`.

The check view page's status (the Part 40 surface's status badge / `status_history`) is left as-is in v1 (it reads the action doc's `status[]`, a different source from events); folding status history into the timeline is a separate concern.

**Dependency — `events.display_key` must equal `workflows.app_name`.** The `events-timeline` component reads events keyed by its `display_key` var (defaulting to the **events** module entry's `_module.var: display_key`), while the engine writes the event's `display` (title + comment-as-description) under the **workflows** entry's `app_name`. The app must wire these equal or the action-page timeline renders empty. This is already a standing invariant — it's why workflow events surface on the entity-page timeline today (Background) — so it's not new work, just a configuration prerequisite worth stating.

## In scope

- Comment → `display.{app_name}.description` in both the submit and `UpdateActionFields` event paths, via one shared helper.
- Dropping `metadata.comment`.
- Deep-merging the engine event-display merge under the app key so per-app author title/description overrides and the comment coexist (D7).
- Reserving `display.{app}.description` for the runtime comment: authors override **title** only; an authored static `description` is rejected at build (`makeWorkflowsConfig` validation) and any non-comment `description` on the merged display is stripped at merge (D4).
- Tightening the mandatory `request_changes` comment validate on both inputs that host it — the form review template's `change_request_comment` and the check surface's modal comment (split off `current_action.comment` onto its own `current_action.change_request_comment` path) — to match the fold gate (`text` non-empty or `fileList` non-empty).
- Deleting the bespoke comments card and rendering the standard timeline on the workflow action pages — the check view page and all four form-kind templates (`edit`/`view`/`review`/`error`).

## Out of scope / deferred

- **Editing or deleting comments after the fact** — events are immutable; a future "edit comment" is a new operation.
- **Threading / replies** — flat timeline only.
- **Standalone comments (no transition).** Every comment today rides a submit or a fields-update; there is no "just post a comment, no state change" write path. Adding one is a new operation, not a rendering change.
- **Email / notification rendering of comments** — separate channel; needs the `text` shadow, which is why text is reconsidered there, not here.
- **Full-text search over comments** — would reintroduce a `text` projection; add when a real need surfaces.
- **Backfill of legacy `metadata.comment` into `display.description`** — old events keep their `metadata.comment` (unread); no migration in v1.
- **Folding `status_history` into the timeline** — left as a separate block reading the action doc.
