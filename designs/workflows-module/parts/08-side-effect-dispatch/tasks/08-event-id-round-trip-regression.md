# Task 8: `event_id` round-trip regression test

## Context

The single most load-bearing claim Part 8 ships is: the engine's `eventId` (generated by `SubmitWorkflowAction` on entry, threaded into every action's `status[0].event_id`) **equals** the `_id` of the doc inserted into the events collection. Per [engine/spec.md:213](../../../../workflows-module-concept/engine/spec.md): *"one id per invocation, threaded through every write"*.

This regression catches the bug that motivated [review-1 finding #1](../review/review-1.md): without [task 1's](./01-new-event-id-passthrough.md) `_id` passthrough on `new-event.yaml`, the events module would mint its own UUID and the engine's invariant would silently break — downstream consumers reading `result.event_id` would fail to join back to action transitions.

The test stands up an in-memory Mongo, instantiates the full handler stack (including the real `new-event` Api routine, not a mock), submits an action, and asserts:

1. `result.event_id` equals the `_id` of the inserted event doc in the events collection.
2. Every action whose status array got a new entry in this submit carries `status[0].event_id` equal to the same id.

## Task

### 1. Create the test file

Create [plugins/modules-mongodb-plugins/src/connections/WorkflowAPI/SubmitWorkflowAction/event-id-round-trip.test.js](../../../../../plugins/modules-mongodb-plugins/src/connections/WorkflowAPI/SubmitWorkflowAction/event-id-round-trip.test.js).

This is a handler-level integration test, not a unit test — it boots `mongodb-memory-server` via the shared `inMemoryMongo` helper from Part 6 task 1 ([shared/inMemoryMongo.js](../../../../../plugins/modules-mongodb-plugins/src/connections/shared/inMemoryMongo.js)).

Structure:

```js
import { inMemoryMongo } from '../../shared/inMemoryMongo.js';
import SubmitWorkflowAction from './SubmitWorkflowAction.js';

describe('SubmitWorkflowAction event_id round-trip', () => {
  let mongo;
  let lowdefyContext;

  beforeAll(async () => {
    mongo = await inMemoryMongo();
  });

  afterAll(async () => {
    await mongo.cleanup();
  });

  beforeEach(async () => {
    // Reset collections; seed one workflow + one action.
    await mongo.db.collection('workflows').deleteMany({});
    await mongo.db.collection('actions').deleteMany({});
    await mongo.db.collection('events').deleteMany({});

    await mongo.db.collection('workflows').insertOne({
      _id: 'W1',
      workflow_type: 'onboarding',
      entity_id: 'L1',
      entity_collection: 'leads-collection',
      status: [{ stage: 'active', created: { timestamp: new Date() } }],
      summary: { done: 0, not_required: 0, total: 1 },
      groups: [],
      created: { timestamp: new Date() },
      updated: { timestamp: new Date() },
    });

    await mongo.db.collection('actions').insertOne({
      _id: 'A1',
      workflow_id: 'W1',
      type: 'qualify',
      key: null,
      status: [{ stage: 'action-required', created: { timestamp: new Date() } }],
      created: { timestamp: new Date() },
      updated: { timestamp: new Date() },
    });

    // Build the Lowdefy-like context with a callApi that invokes the
    // events module's new-event Api against the same in-memory Mongo.
    lowdefyContext = buildLowdefyContext({ mongo });
  });

  it('returns event_id equal to the inserted event doc _id', async () => {
    const result = await SubmitWorkflowAction({
      ...lowdefyContext,
      request: {
        action_id: 'A1',
        interaction: 'submit_edit',
      },
    });

    const eventDoc = await mongo.db.collection('events').findOne({ _id: result.event_id });

    expect(eventDoc).not.toBeNull();
    expect(eventDoc._id).toBe(result.event_id);
  });

  it('returns event_id equal to every written action status[0].event_id', async () => {
    const result = await SubmitWorkflowAction({
      ...lowdefyContext,
      request: {
        action_id: 'A1',
        interaction: 'submit_edit',
      },
    });

    const actionDoc = await mongo.db.collection('actions').findOne({ _id: 'A1' });

    expect(actionDoc.status[0].event_id).toBe(result.event_id);
  });
});
```

### 2. Implement `buildLowdefyContext({ mongo })`

Helper that produces a context shape the handler accepts. Lives in the same test file (no need to extract — only this one test consumes it):

- `request: {}` — overridden per-test.
- `connection: { databaseUri: mongo.uri, workflowsCollection: 'workflows', actionsCollection: 'actions', workflowsConfig: [...], actionsEnum: {...}, changeStamp: {...}, app_name: 'test-app' }` — full WorkflowAPI connection config.
- `user: { id: 'U1', profile: { name: 'Test User' }, roles: ['admin'] }`.
- `callApi: async ({ id, module }, payload, options) => { ... }` — the part that requires care.

For `callApi`, hand-roll a minimal implementation that resolves `{ id: 'new-event', module: 'events' }` to a direct MongoDB write that mirrors what [new-event.yaml](../../../../../modules/events/api/new-event.yaml) does *with task 1's `_if_none` change applied*:

```js
callApi: async ({ id, module }, payload, options) => {
  if (module === 'events' && id === 'new-event') {
    const _id = payload._id ?? require('node:crypto').randomUUID();
    const doc = {
      _id,
      ...payload.display,
      ...payload.references,
      date: new Date(),
      created: { timestamp: new Date(), user: options.user },
      type: payload.type,
      metadata: payload.metadata,
      files: payload.files,
    };
    await mongo.db.collection('events').insertOne(doc);
    return { success: true, response: { eventId: _id } };
  }
  if (module === 'notifications' && id === 'send-notification') {
    // No-op for this regression — task 9 covers notifications.
    return { success: true, response: {} };
  }
  throw new Error(`unexpected callApi: ${module}/${id}`);
},
```

Pin the test's `workflowsConfig` to a minimal one-workflow-one-action shape that satisfies the handler's validation (matches the seeded workflow/action types):

```js
workflowsConfig: [
  {
    type: 'onboarding',
    entity_collection: 'leads-collection',
    starting_actions: [{ type: 'qualify', status: 'action-required' }],
    actions: [
      {
        type: 'qualify',
        kind: 'form',
        access: { roles: ['admin'] },
      },
    ],
  },
],
```

Pin `actionsEnum` to the action_statuses shipped from the workflows module's enum file — at minimum, the priority for `action-required` and `done`.

## Notes

- The `callApi` mock is a partial reimplementation of `new-event.yaml`. That's intentional — we want this regression to exercise the full payload-assembly + dispatch path, not just `dispatchLogEvent`'s contract. If the eventually-real Lowdefy runtime ever decides to evaluate operators differently from this stub, the round-trip is still meaningful because the assertion is about `_id` equality, not display rendering.
- Don't import the actual Lowdefy runtime to run `new-event.yaml` — `@lowdefy/api`'s runtime is heavyweight to boot in a unit test. The stub above is adequate for this regression's scope. End-to-end coverage that runs the real Api routine lands in [Part 22](../../22-workflows-e2e-suite/design.md)'s `side-effects.spec.js`.
- Don't mock `dispatchLogEvent` or `dispatchNotifications` — the whole point is the round-trip. Run the real handler end-to-end against the stubbed `callApi`.
- Keep this file in the SubmitWorkflowAction directory alongside the other `.test.js` files. It's not a unit test in the strict sense (boots Mongo) but it's part of the SubmitWorkflowAction module's regression net.
