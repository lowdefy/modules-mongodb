# Task 3: Migrate the events module

## Context

Task 1 renamed the `EventsTimeline` connection property from `app_name` to **`display_key`**
in the plugin package. This task updates the events module to match and relaxes the
`display_key` manifest var so the common case no longer needs an explicit setting.

`display_key` is the events module's per-app title-projection key. It is required today, and
consumers set it to the same value as the slug. There is a real divergence case (an ops app
rendering another app's events), so the var stays — but with `{ _app: slug }` as its default,
the common case stops requiring an explicit setting. Because every `display_key` consumer is a
runtime site (the timeline projection), the default may be `{ _app: slug }` (design
§Build-time and runtime usage — a runtime-only var can default to `_app`).

## Task

**`modules/events/module.lowdefy.yaml`** (`display_key` var, line ~25):

- Change `required: true` to `default: { _app: slug }`.
- Update the var `description` if it asserts requiredness or references `app_name`.

**`modules/events/connections/events-timeline.yaml`:**

- The wiring currently reads `app_name: { _module.var: display_key }`. Rename the connection
  property key `app_name:` → `display_key:` (matching the renamed plugin property from task 1).
  The value `{ _module.var: display_key }` is unchanged.
- No connection-level default is needed — the slug default now flows through the
  `events.display_key` var above.

## Acceptance Criteria

- `events.display_key` is optional with `default: { _app: slug }`.
- `events-timeline.yaml` wires `display_key: { _module.var: display_key }`.
- No `app_name` remains in `modules/events/module.lowdefy.yaml` or
  `modules/events/connections/events-timeline.yaml`.

## Files

- `modules/events/connections/events-timeline.yaml` — modify — property key `app_name:` → `display_key:` (value unchanged)
