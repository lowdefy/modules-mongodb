# Submit Pipeline

Templates have the following defined buttons: submit_edit, not_required, submit_error, approve, request_changes. Please validate.

Each action will have a pre and post hook routine defined on the action config for the above interactions as needed. A resolver will be used to turn these into Lowdefy APIs.

Each action type will have an `update-action-<action type>` api (to be generated by a resolver). The above buttons will all call this api, but will need to pass in an interaction type, which maps to the above buttons.

This api will call the SubmitWorkflowAction endpoint of the workflows-api custom connection. It will pass through the interaction types, the payload, as well as a map of the pre and post hook lowdefy api endpoint names.

The SubmitWorkflowAction endpoint will run as follows:

- execute pre hook if present for current interaction
- calculate which actions need to be set to action-required using blocked by field on actions.
- merge this actions array with the one returned in the pre hook (if any)
- update actions
- calculate workflow summary, group by fields
- run action group completed unblocks and run any on completed pipelines (to be confirmed)
- write form, summary and action_group updates to workflow doc
- update parent_action_id if present (async call to UpdateAction?)
- generate log event
- Trigger notifications using notifications module call
- Call post hook

The default log event details should be generic:

- event types: action-submitted, action-changes-requested, etc
- Might need to expose event types as a var, look into how the events module works
- standard event names and labels, titles
- add a default metadata object as well. Should contain action type
- <user> marker <action> as <status>

The log event details can be overwritten though, either in the action yaml or returned by the pre hook.

For this to work we will need the lowdefy update with CallAPI that was originally mentioned in the submit pipeline. Please split this out into a separate design.
