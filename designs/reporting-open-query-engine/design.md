# Reporting open query engine

The reporting analytics engine today lets the AI fill a small structured spec (`{ dataset, select, measures, filters, sort, limit }`) that a compiler turns into a fixed, read-only, single-collection pipeline emitting only `$match/$group/$project/$sort/$limit`. That makes injection _structurally impossible_ — no AI-supplied string ever becomes an operator, field path, or collection — but it also means the AI can only ask the narrow set of questions the spec grammar can express. This design opens the engine up so the AI composes **near-arbitrary read-only MongoDB aggregation pipelines** — `$lookup`, `$unwind`, array work, window functions, faceting — while a new validation layer plus a read-only database principal keep it safe. Charts, saved reports, and exports ride the same engine: their queries become pipelines paired with an AI-declared presentation contract, and the structured spec is removed entirely. It is a design only; nothing here is implemented yet.

## Proposed change

1. Replace the structured query spec **everywhere it appears** — `query_data`, `render_chart`, `generate_report`, `export_data`, and saved-report resolution — with an **AI-authored aggregation pipeline**: the agent emits `{ collection, pipeline: [...stages] }`, where `collection` is a catalog-declared base collection and `pipeline` is a MongoDB aggregation pipeline. This is a **breaking replacement, not a migration**: the structured spec, its compiler, and reports persisted in the old format are dropped; no compatibility path is built (see [Charts, reports, and exports](#charts-reports-and-exports-on-the-open-engine)).
2. Pair every query that feeds a **renderer** (charts, KPIs, tables) with an **AI-declared presentation contract** — which output columns are the chart axes, the KPI value, the table columns, and how numbers format — replacing today's derivation of that contract from the spec's structure. CSV exports carry no contract: headers come from the row keys, as today.
3. Validate every pipeline against a **default-deny allowlist** of read-only stages and expression operators, recursing into every nested sub-pipeline (`$lookup.pipeline`, `$facet` — and `$unionWith`/`$graphLookup` if those deferred stages are ever enabled) and expression tree (`$expr`, `$map`/`$filter`/`$reduce`, `$group` accumulators, `let`). Anything not on the allowlist is rejected.
4. Make the **collections catalog the confidentiality/authorization boundary**: every collection referenced anywhere in the pipeline (including nested lookups) must be declared; the engine enforces the union of those collections' role requirements (role-gating itself is opt-in per collection — an entry with no `roles` is open to any authenticated user). The catalog **replaces** the `dimensions`/`measures` dataset list as the dictionary — with reports on pipelines too, the dataset list has no remaining consumer.
5. Add a **second, independent safety layer**: execute all pipelines as a MongoDB principal with only `read` privileges, so a validator gap or mis-classified operator still cannot mutate data or run privileged commands.
6. Add **resource governance**: structural caps (max stages, sub-pipeline depth, `$lookup` count), a mandatory trailing result `$limit`, `maxTimeMS`, and an `allowDiskUse` policy tuned for joins/large groups.
7. **Accept fan-out/grain correctness as a known, documented risk for now** — the engine does not guarantee aggregates are free of double-counting. Grain-awareness is handled by prompting the agent; engine-side correctness checks are deferred future work, added when experience shows they're needed. Views stop being the required correctness layer.
8. Fold the recently shipped **dotted `field` paths and date `bucket`s into the general model** (they become ordinary field references and `$dateTrunc` usage) and keep MongoDB views as an _optional_ convenience, not a requirement.

## Why this, and why now

The dotted-path + views work (commits `e7a0b95e`, `bc556eae`) deliberately kept the compiler flat and pushed joins/arrays into author-defined views, because emitting `$lookup`/`$unwind` from a _structured spec_ would have meant either enumerating a large grammar or risking silent fan-out. That was the right call for a bounded feature. But the product goal is now different: the reporting module is headed for **production over complicated schemas**, and the agent should feel able to _ask almost anything and get an answer_. A structured grammar can never reach "anything"; an open, validated pipeline can. The cost is that the security guarantee changes from _structural impossibility_ to _validated allowlist + read-only execution_ — a real reduction in assurance that this design has to justify and contain.

## Key decisions and rationale

### Raw pipeline, not an expanded structured spec

Chosen over (a) growing the structured grammar to cover joins/unwind/arrays, and (b) a hybrid structured-default-with-raw-escape-hatch. An expanded grammar can never reach "ask anything" and would be a perpetual catch-up against MongoDB's operator surface; a hybrid doubles the maintained surface. The raw pipeline is the only model that meets the "anything" goal. The trade-off — a large, adversarial validation surface — is accepted and contained by the layers below.

### Allowlist, not denylist

The request was "allow everything except `$out`/`$merge`/`$function`." We invert it: **default-deny**, allow an explicit read-only set. A denylist silently admits any operator a future MongoDB version introduces — including a new write, eval, or introspection operator — the moment the driver supports it. An allowlist fails safe: an unknown operator is rejected until someone reviews and adds it. The _effect_ is the same (most read operators are allowed); the failure mode is the difference.

### Two independent layers (validator + read-only principal)

The validator and the database principal defend against **different** threats, and neither alone is sufficient:

- **Read-only principal** (a Mongo user granted only `read`) stops writes (`$out`, `$merge`) and privileged/introspection stages regardless of validator correctness — but it does **not** stop `$function`/`$where` (JS runs under a read-only user), CPU/DoS, or reading a _different_ collection the user happens to have read on.
- **Validator** stops JS/eval, DoS-shaped pipelines, and confidentiality violations (reading collections outside the dictionary) — but a single missed operator or parsing gap could otherwise be catastrophic.

Together they mean a bug in either layer is not, by itself, exploitable into data loss or a confidentiality breach.

### The dictionary is the confidentiality boundary; the principal is the write boundary

The read-only principal is typically granted DB-wide `read`, so it does **not** enforce _which_ collections the agent may see — the validator does, against the dictionary's declared collections. Every `from`/`coll` in `$lookup`/`$unionWith`/`$graphLookup` (recursively) must be declared, and the engine enforces the **union of roles** across all touched collections. A pipeline that reaches an undeclared collection is rejected. (Per-collection grants on the principal are possible as extra depth but are ops-heavy; not required.)

### Correctness is deferred, not designed-in

Per direction: views are **not** the primary/authoritative layer, and guaranteed-correct aggregates are **not** a launch priority. Free `$unwind`/`$lookup` can double-count (fan-out) — this is documented as a known risk. Mitigation for now is **prompting**: the agent is told about grain and steered toward distinct-counting (`$addToSet` + `$size`) when it unwinds. Engine-side correctness aids (grain declaration, fan-out detection, automatic distinct-counts, result annotations) are captured as future work to add _when experience shows they're needed_. Views remain available as a convenience for pre-baked grains but are no longer required.

## The validation model (the core)

The validator is a pure function over the pipeline tree: `validatePipeline({ collection, pipeline, catalog, roles }) → { collection, pipeline }` or throws. It never executes anything.

### 0. Architecture: reconstruct, don't pass through

**The security posture changes shape here, and the design must own it.** Today's engine is **rebuild-from-normalized-spec**: `compileMongo.js` constructs every operator key itself, so no AI string ever lands in operator position — injection is structurally impossible. The open engine is necessarily **inspect-and-forward**: it walks the AI's object tree and hands (a form of) that same tree to `.aggregate()`. Inspect-and-forward is only ever as complete as the walker, against an adversarial operator surface.

To narrow the gap, the validator **returns a freshly reconstructed structure, not the input by reference.** As the walker validates each node it rebuilds it into a new object, copying only keys/values it has explicitly classified and approved. Anything the walker never visited therefore cannot reach the database — a missed subtree fails closed (dropped) rather than being forwarded verbatim. This also forecloses TOCTOU: the object executed is the object validated, with no re-parse in between, and duplicate keys have already collapsed before the walk on **both entry paths** — `JSON.parse` for chat-time tool input, driver BSON-deserialization when a persisted report's pipeline is read back at resolve time. Because resolve-time input is BSON, the walker treats non-plain scalar instances (`Date`, `ObjectId`) as opaque values — they can appear where `JSON.parse` output never would.

### 1. Structural shape and per-stage argument types

- `pipeline` is an array of stage objects; each stage is a single key (`$stage`) → argument. Reject multi-key stage objects, non-arrays, empty stages, and any stage value whose **type** doesn't match the stage's contract.
- **Per-stage argument-type contract** (checked before walking the argument): `$limit`/`$skip`/`$sample` (`{ size }`) take a number; `$match`/`$group`/`$project`/`$addFields`/`$set`/`$bucket`/`$facet`/`$lookup` take an object; `$unwind` a string or object; `$unset` a string or array of strings; `$sort` an object; `$count` a string; `$facet` an object whose every branch value is an array. `$lookup.from` **must be a string** (reject object/array — type confusion on `from` otherwise depends on membership-check internals). A stage argument of the wrong type is rejected before any allowlist logic runs.
- Enforce caps (see [Resource governance](#6-resource-governance)).

### 2. Stage allowlist (default-deny)

Read/transform stages are allowed; write, JS/eval, and server-introspection stages are rejected. Constrained stages are allowed only with collection scoping.

| ------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Allow**                       | `$match`, `$project`, `$addFields`/`$set`, `$unset`, `$group`, `$sort`, `$limit`, `$skip`, `$count`, `$unwind`, `$facet`, `$bucket`, `$bucketAuto`, `$sortByCount`, `$replaceRoot`/`$replaceWith`, `$redact`, `$sample`, `$fill`, `$setWindowFields`                                                                                                                                                                                                                                                      |
| **Allow, collection-scoped**    | `$lookup` — its `from` must be present and dictionary-declared (the `from`-less form only pairs with `$documents`, which is denied); its sub-`pipeline` recurses through full validation                                                                                                                                                                                                                                                                                                                  |
| **Deny — writes**               | `$out`, `$merge`                                                                                                                                                                                                                                                                                                                                                                                                                                                                                          |
| **Deny — JS/eval**              | `$function`, `$accumulator`, `$where` (and `$expr`/`$match` containing any of these)                                                                                                                                                                                                                                                                                                                                                                                                                      |
| **Deny — introspection/stream** | `$collStats`, `$indexStats`, `$currentOp`, `$listLocalSessions`, `$listSessions`, `$planCacheStats`, `$listSampledQueries`, `$shardedDataDistribution`, `$changeStream`, `$documents`                                                                                                                                                                                                                                                                                                                     |
| **Deferred / opt-in**           | `$unionWith`, `$graphLookup` (powerful but expand the auth/complexity surface most; `$lookup` covers current join needs), `$geoNear` (needs a geo index; must be first stage), `$search`/`$vectorSearch`/`$searchMeta` (Atlas-only), `$densify` (the only allowed-class stage that _synthesizes_ documents — a memory/DoS vector needing its own span×step cap; no concrete gap-filling need exists, and sparse series render fine) — off by default, addable per deployment when a concrete need appears |

Anything not listed is denied (default-deny), so a MongoDB upgrade cannot silently widen the surface.

### 3. Expression operator allowlist — one generic recursive walker

**Do not enumerate expression sites.** An earlier framing listed _where_ expressions appear (`$group` accumulators, `let` bindings, `$map` bodies…). That is structurally fragile: MongoDB's surface has too many expression-bearing positions (`$group._id`, `$bucket`/`$bucketAuto` `groupBy`+`output`+`boundaries`, `$sortByCount`, `$setWindowFields` `partitionBy`/`sortBy`/`output`, `$fill`, `$densify`, `$replaceRoot`/`$replaceWith` `newRoot`, `$switch` branches, `$getField`/`$setField` args, `$sort` `{$meta}`…), and any omission is a hole. Instead the validator is **one generic recursive walk**: at every node, _any_ key that begins with `$` must be classified and be on the allowlist for its grammar context, no matter where it sits. There is no "unwalked" position.

Allowed expression families: arithmetic, comparison, boolean, conditional (`$cond`/`$switch`/`$ifNull`), array (`$map`/`$filter`/`$reduce`/`$size`/`$slice`/`$in`/`$setUnion`…), string, date (`$dateTrunc`/`$dateToParts`/`$dateAdd`…), type, and object (`$mergeObjects`/`$objectToArray`) families, plus accumulators (`$sum`/`$avg`/`$min`/`$max`/`$push`/`$addToSet`/`$first`/`$last`/window operators). Denied: `$function`, `$accumulator`, `$where` — anywhere in the tree, not only under `$expr`.

**Allowlist membership semantics (implementation-critical).** Membership is tested against a `Set` (or null-prototype object) using `Object.hasOwn` — **never** `allowlist[key]` or `key in obj` on a plain object, because keys like `constructor`, `toString`, `valueOf`, `__proto__`, `hasOwnProperty` resolve truthy through `Object.prototype` and would be silently admitted. Separately, the walker **rejects the keys `__proto__`, `constructor`, and `prototype` wherever they appear** in the tree — both to close the allowlist-bypass and to prevent prototype pollution as the walker builds its reconstructed output (§0).

**String values are classified, not ignored.** Operators hide in object _keys_, but references hide in string _values_. Every string value is classified as one of: a literal (kept as-is), a field path (`"$field"`, `"$a.b.c"` — allowed within a permitted collection; see field scoping), or a variable reference (`"$$…"`). A `$$`-token is a variable, not an operator, so it skips the operator allowlist — but it gets its own allowlist: the fixed MongoDB system set **`$$NOW`, `$$ROOT`, `$$CURRENT`, `$$REMOVE`, `$$DESCEND`, `$$PRUNE`, `$$KEEP`** plus any variable bound by an enclosing `let`/`$lookup.let`/`$map`/`$filter`/`$reduce`. Every other `$$`-token is **rejected** — in particular `$$USER_ROLES`, `$$CLUSTER_TIME`, `$$SEARCH_META` and friends, which would otherwise leak principal/cluster/infra state into results. `let`-bound names are tracked in a lexical scope threaded through the walk (a var is in-scope only inside the expression that binds it; `$lookup.let` binds in the _outer_ collection's scope and is consumed by the inner pipeline's `$expr`), and the bound value-expressions themselves still recurse.

**`$literal` stops recursion but not accounting.** `$literal`'s argument is opaque data MongoDB must not interpret, so the walker **does not recurse into it** (otherwise it false-rejects benign data that merely looks like `{ $where: … }`) — but it **does** count toward the size/node/array caps (§6), since a multi-megabyte `$literal` is a memory vector. `$getField`/`$setField` (which compute or address field names containing `.`/`$`) are allowed but noted: they make any future field-level scoping unenforceable, reinforcing the "no field scoping at launch" decision.

### 3b. `$match` query-operator allowlist (distinct grammar)

**Finding from review:** the operator allowlist in §3 is for _aggregation-expression_ operators. A `$match` stage's top-level argument is a **query document**, whose operators are a different grammar (`$eq`/`$gt`/`$in`/`$and`/`$or`/`$nor`/`$not`/`$exists`/`$type`/`$mod`/`$regex`/`$all`/`$elemMatch`/`$size`/`$where`/`$text`/`$near`…). These need their own default-deny allowlist:

- **Allow:** comparison (`$eq`, `$ne`, `$gt`, `$gte`, `$lt`, `$lte`, `$in`, `$nin`), logical (`$and`, `$or`, `$nor`, `$not`), element (`$exists`, `$type`), array (`$all`, `$elemMatch`, `$size`), evaluation (`$mod`, `$regex`), and `$expr` (whose subtree recurses through the §3 expression allowlist).
- **Deny:** `$where` (JS/eval), `$text` (needs a text index; defer), and all geo query operators (`$near`, `$nearSphere`, `$geoWithin`, `$geoIntersects`) — deferred alongside `$geoNear`.
- **Query-document walk rules** (the shape rules, stated so the implementer doesn't re-derive them — the same syntactic position flips between operators and data depending on the keys):
  - **Operator document vs literal:** if _any_ key of an object value under a field key starts with `$`, the object is an operator document and _all_ of its keys must be allowlisted query operators — so `{ f: { $gt: 1, $where: "…" } }` fails even though `$gt` is allowed. Otherwise the value is a literal (document-equality match).
  - **Literals fail closed on `$`-keys.** The query grammar has no `$literal`-style escape marker, so a literal subtree containing a `$`-shaped key deeper down (`{ f: { a: { $where: "…" } } }` — pure data to MongoDB) is **rejected**, accepting the false positive. This mirrors §3's everything-classified stance; the expression grammar's `$literal` remains the one opt-out, and it doesn't exist here.
  - **Per-operator operand grammars:** `$elemMatch` recurses as a nested _query document_; `$not` takes an operator document or a regex; `$expr` switches to the §3 expression grammar (already noted above). Comparison/element/array operators take literal operands, subject to the literal rule above.
- **`$regex` (and the expression forms `$regexMatch`/`$regexFind`/`$regexFindAll`) is allowed but capped, not just time-bounded.** This is a real capability increase the design must flag: the current compiler _escapes_ every regex input (`compileMongo.js` `escapeRegExp`), so the AI can't author a pattern at all; the open engine hands the AI a raw, unescaped regex operator. `maxTimeMS` is a weak backstop here — it's checked at document/yield boundaries, and a single catastrophic-backtracking pattern against one large string can run well past the deadline without an interrupt point. So the validator adds hard caps: **max regex pattern length**, **rejection of the `x`/verbose and other exotic flags** beyond `imsu`, and reliance on the field/array caps below to bound input size. Server regex is PCRE (not JS), which limits but does not eliminate backtracking blowup. Accepted with these caps at launch; revisit if it bites.

### 4. Collection scoping and authorization

- Recursively collect every collection named by `$lookup.from`, plus the base `collection`. (The same mechanism extends to `$unionWith`/`$graphLookup` if those deferred stages are ever enabled.)
- Each must exist in the collections catalog (see [Collection namespace](#collection-namespace--the-catalog-is-keyed-by-collection-name)); otherwise reject.
- Compute the union of `roles` across all touched collections and enforce the caller holds at least one of each required set (extends today's single-dataset role gate in `validateQuerySpec.js`).
- Recurse the **full** validation (stage allowlist + expression walk + auth + caps — not just the expression walk) into each `$lookup` sub-`pipeline` with the joined collection as its base, and into each `$facet` branch with the current collection as its base.
- **`$lookup.let` scope:** the `let` value-expressions are validated in the _outer_ collection's scope; the names they bind are the only `$$`-vars (beyond the system set) permitted inside the inner sub-pipeline's `$expr`. This lexical scope is threaded through the walk (§3).
- **View leak (documented residual risk):** a cataloged collection may be a MongoDB view whose own definition `$lookup`s into an **undeclared** collection. The validator sees only the view name and never inspects the view's pipeline, and the DB-wide-`read` principal _can_ read the target — so the "dictionary is the confidentiality boundary" guarantee leaks through views. **Mitigation:** view definitions exposed in the catalog must be treated as part of the trusted dictionary and audited at declaration time (documented as an operator responsibility), or the principal narrowed to per-collection grants if a deployment can't audit its views.

### 5. Field scoping (confidentiality)

Default: any field is readable once its collection is allowed (pipelines synthesize computed fields and, after a `$lookup`, joined fields merge into a single document namespace — so field-level allowlisting both fights "ask anything" and can't cleanly attribute a merged path back to its origin collection). **Decided: no field-level scoping at launch** — collection-level declaration plus the read-only principal is the boundary. **Stated exposure:** with no field scoping, an empty pipeline (or `$replaceRoot: { newRoot: "$$ROOT" }`, or `$getField`) returns **every field of every touched collection**, up to the injected row limit — i.e. a full, bounded dump of any cataloged collection is available to any authorized caller. This is an accepted consequence of "declare the collection = expose the collection." A deployment that must expose a collection while hiding some of its fields declares a MongoDB view that projects them away (views remain first-class catalog citizens; note the view-leak caveat in §4). An opt-in `deny_fields` list per collection is recorded as future work if a concrete PII need surfaces — but `$getField`/`$setField` mean it could never be a hard guarantee, only a hint.

### 6. Resource governance

- **Mandatory trailing `$limit`** — the engine **always appends** a `$limit: CAP` as the final top-level stage, unconditionally (CAP e.g. 1000–5000). A limit the agent wrote is not trusted to be the bound: "inject only if absent" would let the agent supply `$limit: 100000` and defeat the cap. A redundant trailing `$limit` is harmless, so appending always is the simplest guarantee. The same rule recurses into each `$facet` branch (see below).
- **Structural caps:** max total stages (e.g. 50, counted _including_ stages inside `$facet` branches and sub-pipelines, not just top-level), max sub-pipeline nesting depth (e.g. 5), max `$lookup` count (e.g. 10), max `$facet` branches (e.g. 10).
- **Expression-tree caps (validator self-protection).** The structural caps above don't bound what's _inside_ a single stage, and the validator's own recursion is a DoS target: a `$match: { $expr: { $and: [ { $and: [ … 100k deep … ] } ] } }` overflows the recursive walker's call stack and crashes the Node process _before_ any stage cap is consulted. So the walker enforces, per pipeline: **max expression nesting depth**, **max total node count**, **max array-literal length** (carrying forward today's `MAX_IN_VALUES = 100` for `$in`/`$nin`/`$all`, which is otherwise dropped in the new model), and **max serialized pipeline size**. The walker is either iterative or has an explicit depth guard so it fails with a validation error, never a stack overflow.
- **Generative / expensive-but-legal stages (bounded, not just time-capped).** `maxTimeMS` bounds _time_, not _memory_ or _intermediate cardinality_, and an OOM kills the process rather than timing out cleanly. Specific bounds:
  - **`$densify`** can _synthesize_ documents to fill a range (`range: { step: 1, bounds: [0, 1e12] }` → ~10¹² docs mid-pipeline). **Decided: deferred** (moved to the opt-in row in §2) — no concrete gap-filling need exists, and deferring deletes the span×step cap (arithmetic over numeric _and_ date ranges) from the validator. If a deployment enables it later, it must bring that cap with it.
  - **`$range`/`$map`/`$reduce`/`$concatArrays`** build large in-memory arrays _inside one document's_ expression evaluation — `allowDiskUse` does not help here. Bounded by the array-length and node caps above.
  - **`$sample`** on a large collection without a suitable index is a blocking scan; cap the `size` argument.
  - **Uncorrelated `$lookup`** (sub-pipeline with no `localField`/`foreignField`) is a cartesian product; the `$lookup`-count and nesting-depth caps bound the multiplication.
- **`$facet` branch output bound:** the top-level trailing `$limit` does **not** bound array sizes inside a `$facet` (each branch emits an array within one document). The engine appends a trailing `$limit: CAP` to every `$facet` branch pipeline the same way it does the top level, so no branch can return an unbounded array.
- **`maxTimeMS`** retained (configurable; default 30000) — a DoS backstop for unindexed `$lookup`/large `$group`, but explicitly _not_ sufficient alone (see regex interruptibility in §3b and memory/cardinality above); it complements, not replaces, the caps.
- **`allowDiskUse`** — **decided: `true`.** Joins and large groups spill to disk past the 100MB per-stage memory limit; with `allowDiskUse: false` those queries just error. The trade-off is that high-cardinality `$group`/`$sort` now spill large temp files and disk becomes its own bounded-but-real resource — `maxTimeMS` + the row limit are the bound.
- **Indexing:** the engine can't guarantee an index exists on a `$lookup` `foreignField`; `maxTimeMS` catches the pathological case. Optionally emit a slow-query log/warning.

## Query shape and agent surface

### Collection namespace — the catalog is keyed by collection name

**Finding from review:** today's dictionary is a list of _datasets_ keyed by `id`, each with a `source.collection`; two datasets can point at the same collection, and a dataset id need not equal its collection name (the demo has `contact_companies` → `demo_contact_companies_report`). The open engine names raw collection names in `collection` and `$lookup.from`, so it needs an unambiguous collection → policy lookup.

**Decision:** the engine reads from an explicit **collections catalog** — a map keyed by collection name (`{ [collectionName]: { roles, description, fields, relationships } }`). `collection` and every `$lookup.from` must be a key of this catalog; the base plus every touched collection are resolved through it, and the engine enforces the union of their `roles`. An entry with no `roles` (absent or empty list) is queryable by **any authenticated user** — today's dataset semantics, kept and now stated explicitly: role-gating is opt-in per collection, and declaring a collection at all is the act of exposure. The catalog **replaces** the dataset list outright: with charts, reports, and exports also on the pipeline engine (see [Charts, reports, and exports](#charts-reports-and-exports-on-the-open-engine)), nothing consumes `dimensions`/`measures` anymore, and keeping two policy sources would just invite drift. Keying by collection name removes the id-vs-name ambiguity.

Each catalog field entry carries: `type`, `description`, optional enum `values` (feeding select-filter options — the one affordance the report path still wants from the old dictionary), and optional **display hints** (`format: currency`, `currency`, `locale`, `decimals`). Display hints are prompt material, not enforcement: the agent is instructed to copy them into its presentation contracts (see [Charts, reports, and exports](#charts-reports-and-exports-on-the-open-engine)) so that a given field formats consistently across every chart, KPI, and table that touches it — but the contract remains the authoritative input and the engine gains no new validation surface. This carries forward the old dataset list's measure-level `format`/`currency`/`locale`, which the pipeline model otherwise dropped without replacement.

The `query_data` tool input changes from a structured spec to:

```yaml
collection: activities # dictionary-declared base collection
pipeline: # AI-authored, validated read-only aggregation
  - $lookup:
      {
        from: companies,
        localField: company_id,
        foreignField: _id,
        as: company,
      }
  - $unwind: $company
  - $group: { _id: "$company.region", n: { $sum: 1 } }
  - $sort: { n: -1 }
  - $limit: 20
```

The **data dictionary becomes the collections catalog** (see [Collection namespace](#collection-namespace--the-catalog-is-keyed-by-collection-name)): keyed by collection name, each entry declaring its `roles`, a human/AI-readable description of fields and their types, optional enum `values` and display hints per field, and declared relationships (which fields join to which collections) so the agent can author correct `$lookup`s. The `dimensions`/`measures` dataset list is **removed** — the catalog is what every path authorizes and prompts against. The richer the catalog, the better the agent's pipelines — but querying no longer requires pre-enumerating every dimension/measure.

### Catalog authoring: AI-generated, human-curated

For the demo, the catalog is small enough to hand-write. For the production schemas this design targets, field-by-field descriptions across dozens of collections are the single biggest adoption cost — so the catalog gets a **bootstrap workflow** (added 2026-07-22):

- A standalone script (`scripts/` — run by an operator, never at runtime) connects with the **read-only principal** (the same credential the engine uses — provisioning it is already a deployment step), samples each collection (`$sample` + key/type inference over the sampled documents), and asks a model to draft per-field descriptions, spot likely enum fields (low-cardinality strings → candidate `values`), propose display hints (money-shaped names/values → `format: currency`), and infer relationships from naming and value overlap (`company_id` → `companies._id`).
- The script **emits catalog YAML for a human to curate and check in.** The curated file is the trusted artifact; generation is scaffolding.
- **Security posture is unchanged by generation.** The catalog is the confidentiality/authorization boundary, and it stays part of the trusted, reviewed dictionary — the AI drafts it, but a human decides which collections are declared at all, what `roles` gate them, and whether descriptions leak anything they shouldn't. `roles` are never AI-drafted, and because an active entry with empty `roles` is open to any authenticated user, **the draft fails closed: the script emits every collection entry commented out.** Uncommenting an entry is the curator's act of declaring that collection — an unedited draft checked in declares nothing, and the roles question arises only for collections the curator deliberately activated. Re-running the script against a drifted schema produces a fresh draft to diff against the curated file, which doubles as schema-drift detection.

## Charts, reports, and exports on the open engine

**Scope decision (2026-07-22, supersedes the 2026-07-20 "report path stays structured" resolution):** charts and saved reports must work on the open engine, in this design — not as a follow-on. And per direction, **backwards compatibility is a non-goal**: every change may be breaking until the solution settles, so the structured spec is removed rather than kept alongside, and reports persisted in the old format are simply invalid (there are no production saved reports to preserve; if that changes before implementation, a one-time mechanical migration exists — `compileMongo` already turns a structured spec into exactly the pipeline shape, and `select`/`measures` yield the presentation contract — but it is not built speculatively). The same waiver covers the second persisted surface: saved-conversation result parts — their **download** entries carry a structured query spec that re-runs on click and will fail validation afterwards, while persisted **charts** keep rendering (they are stored pre-rendered).

### The problem: presentation is currently derived from query structure

Every renderer downstream of a query gets its contract from the _shape of the spec_, not from the data: `buildEChartsOption` encodes axes as `select[0]` / `measures[].key`; KPI sections read `measures[0].key` out of row 0 with currency/locale from the measure's dictionary entry; table columns (and their tag/numeric styling) come from `select` + `measures`; and report filter controls work by appending `{ field, op, value }` entries to the section query's `filters` array at compile time. A raw pipeline has no derivable structure, so the query and the presentation contract must separate.

### Declared presentation contract

Wherever a query feeds a renderer, the agent declares — alongside `{ collection, pipeline }` — which output columns mean what:

- **Chart sections / `render_chart`:** `x` (category / pie item column) and `y: [column]` (value series). Replaces the derived `select[0]` / `measures[].key` encode.
- **KPI sections:** `valueKey` — the column read from row 0.
- **Table sections:** `columns: [{ key, label?, format? }]`. Today's derived enum **tag styling is deliberately dropped** — the old validator set `tag: true` on columns whose dimension declared `values`, rendering them as colored pills; the new contract renders those cells as plain text. A leaner contract and prompt were judged worth the cosmetic regression; a `tag` flag can be added to the contract if it's missed (decided 2026-07-22).
- **Number formatting:** a per-column `format` descriptor (`{ style: decimal|currency, currency?, locale?, decimals? }`) declared in the contract, replacing format/currency/locale lookups on measure dictionary entries. The agent is prompted to source these from the catalog's per-field **display hints** where the column maps to a cataloged field, so a field formats consistently across every chart/KPI/table that touches it; for computed columns the agent declares the format itself.

The contract is **inert data** — length- and count-capped strings and enums, no query grammar, zero security surface. It cannot be checked against the pipeline statically (the output shape of an arbitrary pipeline isn't statically known), but at every point where the contract is consumed the rows are already in hand (`emit-data-parts` at turn end, `resolve-report` at view time), so the engine verifies it **against the actual rows**: declared keys must exist in the result, chart `y` / KPI values must be numeric. Verification applies only to **non-empty** results — zero rows is a legitimate outcome (a filter can narrow to nothing) and renders as an empty chart / zero-valued KPI / empty table exactly as today (`compileReport` already falls back to `0` for a missing KPI row); `null` cells in a `y`/value column are likewise tolerated, since null group keys are normal pipeline output. A mismatch fails with an actionable message the agent can self-correct on at chat time, and renders as the existing Alert-card fallback at report-view time — a graceful rendering failure, never a safety one.

### Filter binding: server-built `$match` prepended to the pipeline

Report filter controls can no longer append entries to a spec's `filters` array. Instead:

- A section binds to filters exactly as today (`filterBy: [field]`); the filter control's `onChange` re-query calls the query endpoint with `{ query: { collection, pipeline }, filters: [{ field, op, value }] }`, where values are deferred `__state` reads as today.
- The **server builds the `$match` itself** from those fixed triples (`select` control → `eq`; `daterange` → `gte`/`lte`; a small fixed op→operator map, **default-deny** — an unknown `op` rejects the request; the agent contributes no query syntax at re-query time), **prepends** it to the pipeline, drops null-valued triples ("no constraint", as today), and runs the combined pipeline through `validatePipeline` → `AnalyticsPipeline` like any other.
- **The triples are untrusted client input.** The re-query is a browser-side CallAPI, so a direct caller can post arbitrary `field`/`op`/`value` — and `field` lands in _key position_ of the server-built `$match`. That's contained because the built stage is **not exempt from validation**: the combined pipeline passes through the same walk, which rejects `$`-prefixed and `__proto__`/`constructor` field keys and fails closed on operator-shaped values.
- Prepending preserves today's semantics (filters apply pre-aggregation, and can use indexes). **Documented limitation:** a bound filter field must exist on the base collection's documents — not a post-`$group`/post-`$lookup` alias. This matches today's behaviour, where filters bind to raw dictionary paths; the agent is prompted to keep filterable fields at the source grain.
- Select-control options come from the agent's declared `options` (already supported) or the catalog field's enum `values`.

### What changes, what doesn't

Execution sites swap mechanically: `render-chart` / `generate-report` payload schemas and validators accept `{ collection, pipeline }` + contract, `export-data` accepts the pipeline alone; the `emit-data-parts` and `resolve-report` `:for` loops run `AnalyticsPipeline`; `compileReport` consumes the declared contract instead of derived `select`/`measures`; `buildEChartsOption` takes an explicit encode; CSV export takes columns from the row keys as today. One new deployment floor: persisting raw pipelines stores nested `$`-prefixed field names (`$match`, `$lookup`, …) inside report documents, which MongoDB allows only from **5.0** — the old spec (no `$` keys anywhere) had no such requirement.

The security model does not change at all. Validation and catalog role-gating live _inside_ `AnalyticsPipeline`, so a persisted report's pipelines are revalidated against the **current** catalog with the **viewing** user's roles on every resolve — the same posture as today's per-resolve revalidation, enforced mechanically at the one choke point every path shares. `validatePipeline` and the three allowlists need nothing extra for charts or reports.

## Relationship to the current approach

- **Dotted `field` paths** — subsumed: the agent references any field path directly inside a validated pipeline (scoped to allowed collections). The structured-spec `field` mapping is no longer needed on the exploratory path.
- **Date `bucket`s** — subsumed: `$dateTrunc` is an allowed operator the agent uses directly.
- **MongoDB views** — retained as an optional convenience (an author can still expose a pre-baked, correct-grain view as a "collection" in the dictionary), but no longer the required mechanism for joins/arrays.
- **Report generation** (`validateReportSpec.js`, `compileReport.js`, `render_chart`, `export_data`) — **in scope**: sections carry `{ collection, pipeline }` plus a declared presentation contract, and the structured spec is removed entirely (see [Charts, reports, and exports](#charts-reports-and-exports-on-the-open-engine)). An earlier revision of this design deliberately scoped the report path out on the grounds that sections need stable, typed output columns an arbitrary pipeline doesn't guarantee; the declared contract, checked against actual rows at each render point, is the answer to that objection.

## Architecture / data flow

1. Agent calls `query_data` (or `render_chart`/`generate_report`/`export_data`, whose sections carry the same query shape) with `{ collection, pipeline }`.
2. New `validatePipeline` (pure) validates against the catalog + allowlists + caps, collecting touched collections and enforcing roles.
3. **`AnalyticsPipeline` replaces `AnalyticsQuery`** as the `ReportingData` connection's single request — with the report path also on pipelines, there is no second validator to keep separate, so one request is the "one correct way." It runs the validated (reconstructed) pipeline with `maxTimeMS`/`allowDiskUse: true`/appended `$limit`, on the **same `ReportingData` connection**, whose `databaseUri` is now provisioned to a **read-only MongoDB principal**. `meta: { checkRead: true, checkWrite: false }` as today. Validation and role-gating happen **inside the request**, so every caller — the chat tool, `resolve-report`'s per-viewer section loop, `emit-data-parts`, filter re-queries, CSV downloads — passes the same gate mechanically. The **catalog binds at the connection, not per request**: `ReportingData`'s connection properties carry it (wired once via `_module.var` in the module's connection YAML), so every request validates against the same catalog by construction — a caller can't substitute a stale or trimmed one. Requests carry only `{ query, roles }`; `roles` stays per-request (`_user: roles` — it's the calling user's), and an endpoint that omits it fails closed for role-gated collections. **Implementation finding (task 7):** the connection-binding guarantee has one qualification — an app that _remaps_ `reporting-data` onto its own connection entry replaces the module's connection definition entirely, including the catalog binding, so the remapping app must bind `catalog` on its connection itself (without it the engine has no catalog and rejects every pipeline — fail closed, not open). The demo binds the same `catalog.yaml` file the module var references so the two cannot drift; documented in `docs/reporting/reference/catalog.md`.
4. Rows returned to the agent (and to the charts/downloads panel) exactly as today.

`AnalyticsQuery`, `validateQuerySpec`, and `compileMongo` are **deleted** — breaking, no transition path (see the scope decision in [Charts, reports, and exports](#charts-reports-and-exports-on-the-open-engine)).

## Non-goals

- Implementation — this is a plan only.
- Write capability of any kind — the engine stays strictly read-only.
- Guaranteed-correct aggregates — explicitly deferred (see the correctness decision).
- Backwards compatibility — no migration for persisted structured-spec reports, no coexistence of the two query shapes; every change may be breaking until the solution settles (per direction, 2026-07-22).
- Multi-database queries — all touched collections live in the one `reporting-data` database.

## Risks

- **Validator completeness** is now safety-critical, and for JS/eval it is the **sole** defense: the read-only principal explicitly does _not_ stop `$where`/`$function` (JS runs fine under a read-only user, per the two-layers decision). So a miss in the `$match` **query-grammar** walk (distinct from the expression grammar — the classic conflation trap) that admits `$where` is directly a server-side CPU/JS-execution vector with no second layer behind it. Mitigated by default-deny + the generic reconstruct-don't-forward walker (§0), but the validator needs an adversarial test suite: `$match: { $where }` and `$match: { $expr: { $function } }`; a mixed operator document (`{ f: { $gt: 1, $where: "…" } }`, expect reject) and literal-position `$where` (`{ f: { a: { $where: "…" } } }`, expect fail-closed reject); `$function`/`$accumulator` buried in `$group._id`, `$bucket.output`, `$setWindowFields.output`, `$replaceRoot.newRoot`; allowlist-bypass keys (`constructor`, `__proto__`, `toString`); `$$USER_ROLES`/`$$CLUSTER_TIME` leaks; `$in` array over the length cap; filter-triple abuse posted straight to the re-query endpoint (`field: "$where"`, `field: "__proto__"`, `value: { $gt: … }`); a pathological-depth `$expr` (walker must not stack-overflow); `$literal` wrapping operator-shaped data (must pass); nested `$lookup.pipeline` with `$merge`; deferred stages (`$unionWith`/`$graphLookup`) rejected in every position.
- **DoS** via expensive-but-legal pipelines — cartesian `$lookup`, deep `$facet`, in-expression array building (`$range`/`$map`), unindexed `$sample`, catastrophic-backtracking `$regex` that outruns `maxTimeMS`, and the validator's own recursion. Bounded by the structural + expression-tree + generative-stage caps and `maxTimeMS`; reduced, not eliminated.
- **Wrong numbers** from fan-out — accepted for now; the top user-visible risk, revisited when it bites.
- **Presentation-contract mismatch** — the contract can't be checked against the pipeline statically, so a wrong `x`/`y`/`valueKey`/column key surfaces only when rows are in hand. Contained: at chat time the tool fails with an actionable message the agent self-corrects on; at report-view time the section renders the existing Alert-card fallback. A rendering failure, never a safety one — but saved reports can rot if the underlying schema drifts, same as today's dictionary drift (and surfaced the same way).
- **Persisted pipelines execute later, as other users** — a report saved by one user runs its pipelines on every view. Contained by the same invariant as today: validation + catalog role-gating live inside `AnalyticsPipeline`, so every resolve revalidates against the current catalog with the _viewer's_ roles; a pipeline touching a collection the viewer can't access fails as an Alert-card section.
- **Operator drift:** new MongoDB operators need triage into the allowlist; default-deny makes the failure "can't use it yet" rather than "silently unsafe."
