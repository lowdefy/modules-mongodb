# Task 3: `gen-var-docs.mjs` — Generate Var Reference Tables (Phase 1b)

## Context

`module.lowdefy.yaml` is the source of truth for every var's `type`, `default`, `required`, `enum`, and `description` (including nested object properties). Hand-restating these in docs guarantees drift, so each module's `docs/{module}/reference/vars.md` is **generated** from its manifest by a script. Task 1 cleaned the manifests; Task 2 established the output tree and front-matter contract. This task writes the generator.

Generation alone doesn't stop drift — `vars.md` is a committed file, so it goes stale the moment a manifest changes and nobody re-runs the generator. The fix (wired in Task 5) is a `--check` mode that CI runs on every PR. This task must implement both the write mode and `--check`.

The repo's existing scripts (`scripts/release-notes.mjs`, `scripts/sync-module-versions.mjs`) are ESM `.mjs` with an Apache 2.0 license header — match that style.

## Task

Create `scripts/gen-var-docs.mjs`:

**Input:** walk `modules/*/module.lowdefy.yaml`, read the `vars:` section (top-level and nested object properties).

**Output:** for each module, write `docs/{module}/reference/vars.md` with:

- Front-matter: `title: Vars`, `module: {name}`, `type: reference`.
- A **generated marker** comment at the top of the body (e.g. `<!-- GENERATED by scripts/gen-var-docs.mjs — do not edit by hand -->`) so editors know not to hand-edit.
- A var table. Per open question 1 in the design: **lean table** (columns: Name, Type, Default, Required, Description) for flat vars; **indented sub-lists / nested sub-sections** for vars with deep object properties (e.g. workflows, companies, layout `logo`). Pick a consistent rendering for nested properties — a nested heading per object var with its own sub-table is acceptable; the goal is readability, not one giant flat table.

**Modes:**

- Default (write): regenerate every `vars.md` in place.
- `--check`: regenerate to a temp location and compare against the committed files; exit non-zero on any diff, printing which module(s) drifted. Do not modify committed files in `--check`.
- Consider a `--module {name}` filter for authoring a single module (optional, only if cheap).

**YAML parsing:** use a YAML parser already available in the workspace (check `pnpm-lock.yaml` / existing scripts for `yaml` or `js-yaml`; reuse whatever the repo already depends on rather than adding a new dep).

## Acceptance Criteria

- `node scripts/gen-var-docs.mjs` writes a `docs/{module}/reference/vars.md` for every module that has vars, each with correct front-matter, a generated marker, and a table/nested layout reflecting the manifest.
- Descriptions, defaults, `required`, and `enum` values are surfaced verbatim from the manifest (no manual restatement).
- Nested object vars (verify against workflows and layout `logo`) render readably, not as an unparsed blob.
- `node scripts/gen-var-docs.mjs --check` exits 0 immediately after a clean generate, and exits non-zero (naming the drifted module) if a manifest is edited without regenerating.
- Script carries the Apache 2.0 header and matches the existing `.mjs` style.

## Files

- `scripts/gen-var-docs.mjs` — create — generator with write + `--check` modes.
- `docs/{module}/reference/vars.md` — create (generated output) — committed for each module with vars.

## Notes

- Output paths must match the design's tree (`docs/{module}/reference/vars.md`). Other reference pages (`exports.md`, `indexes.md`, grammar) are hand-written by later tasks — this generator only owns `vars.md`.
- The generated-marker comment matters: later tasks and reviewers must be able to tell generated files from hand-written ones at a glance.
- Don't add a runtime dependency for YAML if the repo already has one transitively used by scripts; prefer the existing parser.
