# User Admin on BetterAuth

Redesign of the `user-admin` module for the BetterAuth-based auth engine (the auth-upgrade designs). A major breaking change — the module is rebuilt against the new `contact` / `user` / `member` model and the sanctioned admin-step mutation surface, not adapted from the current interface.

**Scope**: apps under the **`pinned` active-org policy** (org = app — the replacement for the multi-app adapter pattern). One module instance administers one pinned org. Multi-tenant administration is a separate future module; the self-service counterpart (`user-account`) is a separate follow-on design.

**Dependency**: the auth-upgrade designs — [admin](../../../lowdefy-design/designs/auth-upgrade/admin/design.md) (step catalog, ownership rubric, module contract), [user-model](../../../lowdefy-design/designs/auth-upgrade/user-model/design.md) (records, lifecycle, hard wall), [mongodb](../../../lowdefy-design/designs/auth-upgrade/mongodb/design.md) (collection names, native reads). This design assumes `attributes` are stored as native BSON sub-documents (not JSON strings) — resolved upstream (mongodb Decision 5: the adapter is **vendored** into `@lowdefy/connection-mongodb` with `supportsJSON: true` plus a legacy string-parse on read); native reads here filter and project attribute contents on that settled shape.

---

## Problem

Today the module owns everything about a user as raw MongoDB writes against the fused `user_contacts` collection (`apps.{app}` map, `is_user`, `disabled`, `global_attributes`). The new model splits that record across app-owned data (`contact`) and auth-owned records (`user`, `member`, `invitation`, `session`, `account`), and the auth-owned side may only be written through the platform's admin routine steps — raw writes bypass BetterAuth's invariants. Every page, form, and API in the module is affected, and the new engine offers capabilities the old module could never have (ban, session revocation, impersonation, native invitations, auth-method visibility). Rebuild, don't port.

## The reframe

The module is no longer CRUD over one collection — it is **the operator console for a person's access lifecycle in one app**, composing contact requests + admin steps + audit events into per-concern routines:

| Concern                                   | Record                            | Write pathway                                                                                                                   |
| ----------------------------------------- | --------------------------------- | ------------------------------------------------------------------------------------------------------------------------------- |
| The person (profile, CRM data)            | `contact` (`user-contacts`)       | Shared `write-profile` fragment — change-stamped `contact` write + `UpdateUserProfile` re-denorm of the target's `user.profile` |
| Login identity + global attributes        | `user` (`users`)                  | `UpdateUserAttributes`, `BanUser`/`UnbanUser`, `DeleteUser`, `RevokeUserSessions` steps                                         |
| This app's access (roles, app attributes) | `member` (`user-members`)         | `UpdateMemberRoles`, `UpdateMemberAttributes`, `RemoveMember` steps                                                             |
| Pending access                            | `invitation` (`user-invitations`) | `InviteMember`, `CancelInvitation` steps                                                                                        |

Reads stay native — aggregations over these collections joined by `userId` / `contactId` — per admin Decision 4.

Under `pinned` policy the word "organization" never appears in the UI: the pinned org _is_ the app, and the admin sees "this app's users."

The access lifecycle the UI makes legible:

```
Invited ──accept──► Active ◄──unban──── Suspended (ban: global, reversible,
   │                  │    ──ban─────►             sessions revoked)
cancel / expire       │
                 Remove from app (member row deleted; re-invite to restore)
                      │
                 Delete login identity (user row hard-deleted; contact survives)
```

## Key decisions

### 1. One module instance = one pinned org; the users list is the members list

The module administers the deployment's pinned organization. The list page reads `user-members` for that org, joined to `users` and `user-contacts`. There is no `app_name` var — per-app scoping by field path dies with the `apps.{app}` map.

Native reads see `member.role` as BetterAuth stores it — a CSV string (`"admin,branch-manager"`, user-model Decision 5); the split-to-array happens only in `resolveAuthentication`. Every pipeline that displays, filters, or exports roles therefore splits the string itself (`$split`) — nothing reads a role array from the database. **The role filter matches exact elements of the split array** (`$in` / equality against the post-`$split` array), never a substring/regex over the raw CSV — otherwise a filter for `admin` would also match `super-admin`, and `editor` would match `content-editor`. This is distinct from the free-text name/email filters (Decision 2), which are deliberately regex.

**Org scoping — resolved.** The admin steps default `organizationId` to the deployment's pinned org, and the module's native reads `$match` on `user-members.organizationId` using the engine's server-side **`_organization`** operator (`_organization: id`), which exposes the resolved pinned org id. The module carries **no `org` var** (formerly upstream ask 1, now delivered — steps default in admin, config read via `_organization` in engine).

**Same-database co-location is a hard precondition.** Every native read is a single aggregation that `$lookup`-joins the auth collections to `user-contacts`, and MongoDB's `$lookup` cannot cross databases (let alone clusters). So the BetterAuth adapter's database ([mongodb](../../../lowdefy-design/designs/auth-upgrade/mongodb/design.md) Decision 1 — the adapter takes its own explicit `_secret` URI), the `user-contacts` connection (mongodb Decision 2 — an app collection on the app's connection), and the module's read-only connections **must all resolve to one MongoDB database** — the natural config being a single shared `_secret` (e.g. `MONGODB_URI`). This is the mongodb design's own intended shape: its `user-` naming rationale envisions `user-contacts` "landing inside" the auth-collection block in one database listing. **The failure mode is silent, so it is documented, not enforced** (per the enforcement decision): a cross-database `$lookup` treats the missing collection as empty rather than erroring, so a divergent deployment shows blank contact data everywhere instead of failing loudly. There is no clean home for a build/startup check (the platform is not told which connection needs co-location, and modules have no startup hook), so the precondition is stated here and in the consumer docs; the blank-data symptom surfaces immediately in dev/test. A deployment that genuinely cannot co-locate would need the app-layer-stitching alternative (rejected here — it abandons the native single-aggregation reads admin Decision 4 charters).

**Why not the platform's tenant wall?** The declarative tenant wall (`tenant: true` on a connection — [permissions](../../../lowdefy-design/designs/auth-upgrade/permissions/design.md)) auto-scopes reads to the caller's active org, which under `pinned` is the pinned org — tempting, but it does not fit this module, for two reasons. (1) The wall injects its org `$match` recursively into every `$lookup` sub-pipeline, assuming each joined collection carries the tenant field; but the module joins `user-members` → `users` → `user-contacts` (the workspace adds `user-sessions` / `user-accounts`), and only `user-members` / `user-invitations` carry `organizationId` — the injected filter would match nothing and every join would fail closed (the wall's own "global reference collection" case). v1 offers no per-`$lookup` exemption, and `tenant: none` disables the whole wall including stage 0. (2) Decisions 4 and 6 deliberately read `user-members` across _all_ orgs (ban blast-radius, cross-app badges), which the wall forbids. So the module scopes explicitly with `_organization: id` at the root and leaves joins and cross-org reads free — one mechanism, uniform.

### 2. Members and invitations are sibling tables; the export merges them

The list page presents two tables — **Members** (the primary view) and **Invitations** (pending / expired, with a pending-count badge) — as tabs on the `all` page, one menu entry. They stay separate because they are different lifecycles over different collections with different columns and actions (members: roles, status, last-active; invitations: inviter, expiry, resend/cancel). Forcing them into one union list muddies both.

**Search is plain `$match` text/regex — Atlas Search is gone from this module.** The old list page's `$search` stage and its Atlas index requirement do not carry over: the members and invitations tables filter with regex/text `$match` clauses over the relevant fields (contact name / emails, `user.email`, roles, invitation email). This is not just a simplification — in the split model it is the only search that spans the data at all. The old module could use Atlas `$search` because everything lived on one fused `user_contacts` document, so `$search` ran at stage 0 and saw every searchable field. Now name is on `user-contacts`, email on `users`, roles on `user-members`, so the searchable fields only coexist **after** the `$lookup` joins. Atlas `$search` **must be the pipeline's first stage** over a single collection, so it structurally cannot search across the join; a post-`$lookup` regex `$match` can. The cost is that a post-join `$match` cannot use an index (the joined shape is a computed stream, and substring regex wouldn't use a btree index even pre-join), so each query is a linear scan of the org's members.

**Sizing:** this is sized for pinned orgs in the **low thousands of members** — a linear scan over a `$lookup`-joined pipeline is fine at that scale, and dropping `$search` removes the module's only Atlas-specific infrastructure dependency. A deployment that outgrows it (tens of thousands) needs a **pipeline restructure**, not a mid-pipeline seam: root a `$search` from an Atlas Search index built on a _joined view_, or denormalize the searchable fields onto one collection so `$search` can run at stage 0. Both reintroduce Atlas-specific infrastructure and are a documented future option, not built now (no concrete large-org case). The `request_stages.filter_match` slot appends plain `$match` clauses (Decision 8) — it extends the regex filter; it is **not** a path to reintroduce Atlas `$search`, which cannot live mid-pipeline.

The **Excel export merges both** into one sheet with a `status` column (Active / Suspended / Invited / Expired) — the union lives only in the export pipeline, where a flat "everyone with or pending access" snapshot is exactly what the consumer wants. The export sits behind a **`download` var (default `false`)**: unless enabled, the download button and its export pipeline are excluded.

Status derivation is uniform: member row + `user.banned` not true → **Active**; member row + `user.banned: true` → **Suspended**; invitation with `status: "pending"` and future `expiresAt` → **Invited**; invitation with `status: "pending"` and past `expiresAt` → **Expired**. There is **no `expired` status** in BetterAuth (`invitationStatus` is `pending` / `accepted` / `rejected` / `canceled`, verified against `better-auth@1.6.23`) — Expired is derived, `status: "pending" AND expiresAt < now`. The native pipelines read `user-invitations` directly, bypassing the org plugin's `findPendingInvitation`/`listInvitations` helpers, so they must do this filtering themselves: `$match status: "pending"` (to exclude `accepted` / `rejected` / `canceled`, which appear in neither the Invitations tab nor the export) and split Invited vs Expired on `expiresAt` against now.

The ban fields are BetterAuth's admin-plugin schema on the `user` row (verified against `better-auth@1.6.23`): `banned` (boolean, default `false`), `banReason` (string), `banExpires` (date). **The module issues permanent bans only** — its suspend action never passes a ban duration, so `banExpires` is never set. Ban _expiry_ is therefore out of scope (Non-goals), and the Suspended read is simply `banned === true` — no `banExpires` comparison. (BetterAuth does support timed bans and auto-clears a lapsed one lazily at the banned user's next sign-in; since this module never creates one, that path is never exercised and the native read needs no expiry check.)

### 3. A single user workspace with section-scoped edits

One detail page (`view`) replaces today's view + edit + check pages. Tiles: **Profile** (contact fields), **Access** (roles + member attributes), **Global attributes** (user attributes), **Security**, **Apps** (cross-app badges, Decision 6), **Activity** (event timeline). Each tile edits through its own modal and its own routine.

This is the data model speaking, not a UI preference: a profile save runs the shared `write-profile` fragment (change-stamped `contact` write **+ `UpdateUserProfile` re-denorm** — see below); an access save is `UpdateMemberRoles` + `UpdateMemberAttributes`; a suspension is `BanUser`. One "edit everything" form would smear three write pathways behind one Save button, blur partial-failure semantics, and produce mushy audit events. Per-section routines give crisp events ("roles changed", "profile updated", "suspended") and line up with the permissions milestone, which will gate these capabilities individually.

**Why profile save is `write-profile`, not a bare contact write.** Profile fields live on the app-owned `contact` (source of truth), but the auth `user` row carries a denormalized `profile` copy plus `name` / `image` display copies — and those are what feed `_user.name` / `_user.image` / `_user.profile.*`, the header, avatar, and menus the _target_ sees in their own sessions across the suite. An admin editing another user's profile with a bare contact write would leave that target's `user.profile` bag stale (session refresh touches only the caller, not the target), so their header/avatar would show old values until they next wrote their own profile. The shared **`write-profile`** fragment (Decision 8, sourced from `modules/shared/`, `_ref`'d by both this module and `user-account`) pairs the change-stamped contact write with a `UpdateUserProfile` re-denorm of the target's bag in one routine, so profile data can't be written without refreshing it — `resolveAuthentication` reads the target's `user` row per request, so the re-denormed bag is fresh on the target's next request. The re-denorm targets the _edited_ user (never the admin), so it is floored by `auth.userAdminRole` like every other admin write, and the `request_stages.write` seam still extends the contact write.

Every routine endpoint is role-gated by the hosting endpoint's `auth.api.roles` (admin Decision 2), supplied by a module var. The platform now also enforces `auth.userAdminRole` as a step-level floor — `handleAuthStep` mechanically checks the caller holds the configured role before **every** auth step runs (uniform across the catalog, not a hand-listed set; only `system: true` invocations bypass it, plus `UpdateUserProfile`'s self-target exemption — which this module never hits, since its profile writes target the edited user, not the admin). So every auth-owned write this module drives (the reframe table's step columns — user attributes, member roles/attributes, ban/unban, session revocation, membership removal, invitations, delete, plus the `UpdateUserProfile` re-denorm on profile saves) is floored, and the module's endpoint gates compose on top as defense in depth rather than the sole guard (upstream ask 3).

**Partial-failure semantics — specified and accepted.** A routine is a sequential step array that **halts on the first error with no rollback** — auth steps are adapter-direct and there is no cross-step transaction (standalone MongoDB has no sessions, mongodb Decision 5), so a step that already committed stays committed. Two routines chain writes: the **access save** (`UpdateMemberRoles` + `UpdateMemberAttributes`) and the **invite** (create-or-link `contact` → `InviteMember`). This is safe because every step is an **idempotent** set/upsert — `UpdateMemberRoles`/`UpdateMemberAttributes` _set_ (not append), create-or-link reconciles on duplicate-key — so a halted routine is safe to re-run and converges; **retry is the recovery path**. The **audit event is the routine's final step**, so it fires only on full success: a partial failure emits no event and returns the error to the admin (no misleading "roles changed" event for a write that then failed). The two named partial states are benign — access save leaves new roles with old attributes until the admin retries; an _unknown_-email invite may leave an orphan `contact` if `InviteMember` fails, which the next check resolves to "existing contact, no membership" and links (create-or-link's reconcile makes the retry a no-op on the contact). Cross-step **transaction support is planned platform-side**; this module adopts it when it lands, at which point these routines become atomic and the retry-convergence reasoning is superseded — out of scope now.

### 4. Two revocations, honestly labelled — and ban's blast radius is global

The old `disabled` flag conflated two things the new model separates:

- **Suspend / reinstate** — `BanUser` / `UnbanUser`. Reversible, revokes sessions, blocks sign-in. Suspend issues a **permanent ban** — no duration is passed, so the ban stays until an admin reinstates (timed bans / auto-expiry are out of scope, Non-goals). **Ban is user-level, so it applies across every app in the suite** — the UI must say so, never present it as app-scoped. The confirm dialog **enumerates the user's other memberships** (a native read over `user-members` + `user-organizations`) so the admin sees exactly which apps the suspension reaches before confirming.
- **Remove from this app** — `RemoveMember`. Deletes the member row; the person keeps other apps and their contact record survives. Restore is a re-invite (roles/attributes are not retained — membership is the row's existence).

**Delete login identity** (`DeleteUser`) is offered only when the user holds **no other memberships** (a native read guards the button and the routine re-checks): app A's admin must not destroy an identity app B depends on. The contact always survives; contact soft-delete stays the contacts-side convention.

**Ban's suite-wide authority is deliberate — and gated by a var.** Under `pinned`, suspend hands every app instance's admin roles a capability that reaches the whole suite: the one place the module's authority exceeds its app scope. **This rests on an explicit deployment premise: the pinned suite is administered by one trusted operator group.** The canonical shape is a multi-app deployment — e.g. a team portal + a customer portal on one suite — where a suspended person should be locked out _everywhere_, so a global block is the correct, "proper" suspend, not an over-reach. The admin already chooses scope at the point of action: **Suspend** (`BanUser`, global, reversible, keeps membership) versus **Remove from this app** (`RemoveMember`, app-scoped, deletes the member row) are two distinct buttons — there is deliberately no half-scoped "suspend from one app but keep membership" (the org `member` row carries no status field, and a partial suspend would be a worse suspend). Because the trusted-operator premise is the intended model, suspend defaults on — but it sits behind a **`suspension` var (default `true`)** gating the suspend/reinstate surface, the deployment-level switch for a suite that does _not_ match the premise and wants its app-admins limited to app-scoped revocation (`RemoveMember`).

**Self-targeting is allowed.** No guard stops an admin suspending or removing themselves; lock-out is accepted as recoverable by another admin (or direct database access as a last resort). Guards against self-suspend / self-remove are surface we can add later if this bites in practice — not worth carrying up front.

### 5. Security tile — adopt the new engine capabilities

- **Suspend / reinstate** (Decision 4) — behind the `suspension` var, default `true`.
- **Sessions** — active sessions listed (native read over `user-sessions`: created, expiry, IP/user-agent) with a "sign out everywhere" action (`RevokeUserSessions`). The read **`$project`s out `token`** (and any other bearer/secret fields) — the `session` row's `token` is the bearer credential (`@better-auth/core@1.6.23` session schema), and this read shows an admin _another_ user's sessions, so returning it would hand the admin a credential to replay that user's cookie. Matches the sibling module's self-service treatment (`user-account` Decision 5, "`token` projected out").
- **Auth methods** — read-only: linked providers (`user-accounts`), passkeys, MFA enrolment, email-verified. Native reads; answers "why can't she log in?" without touching anything.
- **Impersonation** — "View as user" via the `ImpersonateUser` client action, **behind a module var** (`impersonation: false` by default). It is a client action (session-scoped, cookie-setting), governed by BetterAuth's own admin-plugin AC. Under the member-only role model no caller satisfies that check by default; the engine resolves this (upstream ask 3, shipping in phase 9) with the `auth.userAdminRole` config option that gates the whole user-administration surface. Impersonation is its client-action case: the engine bridges BetterAuth's user-level check via an internal `user.role` denormalization and grants the role a curated admin-plugin scope set (`user: ['impersonate']`, `impersonate-admins` excluded). So the var gates the UI and the engine's user-admin-role AC makes the action actually work; the two align on the same administering role (Decision 3). The capability works once the app configures `auth.userAdminRole` and the caller holds that role.

### 6. Cross-app badges — visibility, not management

The workspace shows the user's other memberships in the suite as read-only **app badges** (org name tags — native read over `user-members` + `user-organizations`), answering "which apps does this person have?" for support. The cross-app disclosure — any app's admins see which other apps a person belongs to (here and in the ban dialog, Decision 4) — is deliberate: it rests on the **same explicit deployment premise as suite-wide suspend (Decision 4): the pinned suite is administered by one trusted operator group.** Cross-app _visibility_ is strictly weaker than the cross-app _ban_ that premise already licenses by default, so it too is uniform with no var — in a single-org app the badge list is naturally empty. Managing that access still belongs to each app's own admin instance. **If a suite with siloed per-app admin teams ever needs it** (not a case in view today, so no var on a guess), the badges and the ban-dialog membership enumeration are the same disclosure and would share a single opt-out gate — noted as the future seam, not built now.

### 7. Invite flow — email-first, BetterAuth-native

One `invite` page. The admin enters an email first; a check routine resolves it before the form opens:

- **Already a member** → link to their workspace (today's check-page guarantee, now in-page).
- **Pending invitation** → show it, offer resend / cancel.
- **Existing contact (no membership)** → link the contact, prefill profile fields.
- **Unknown** → blank form; the routine creates the contact.

Submit = create-or-link `contact` → `InviteMember` with `profile: { contactId }` + roles → audit event. Create-or-link runs the **shared `create-or-link-contact` fragment** — a generic email-keyed upsert living in `modules/shared/` (e.g. `modules/shared/contact/`), `_ref`'d **by relative path** from both this module's invite flow and [`user-account`](../user-account-better-auth/design.md)'s merge-on-signup hook, so both use one upsert (keyed on `lowercase_email`, reconcile-on-duplicate-key) rather than two implementations that could drift. It is a **shared file, not a module export**: it carries no module vars (each caller wraps it with its own ids and audit event), so it resolves in the consuming module and needs **no `user-admin → user-account` dependency** — the export-plus-dependency model is for fragments that encapsulate one module's vars (e.g. a company-selector bound to the companies module), which these are not. (The sibling design currently frames this fragment and `write-profile` as _exported by_ `user-account`; that framing should likewise move to the shared folder — flagged there.) The invitation email is sent by BetterAuth through the deployment's **`auth.email`** config — the same unified send path as verification, password-reset, and magic-link emails — so the module ships **no email endpoint, hook, or binding**. The Lowdefy email redesign that was pending when this was first written has since landed ([auth-emails](../../../lowdefy-design/designs/auth-upgrade/_completed/auth-emails/design.md)), and it settles how the invite is branded: by default the invite renders the **stock branded `InvitationEmail`** template (a React Email component themed from the deployment's `app.email` brand), so a configured `auth.email` connection ships an HTML, on-brand invite with **no module wiring** — not the plain-text stub the earlier draft assumed. The CTA's accept link is built by the engine from **`authPages.acceptInvitation`** (the accept-page role `user-account`'s manifest contributes — see [user-account](../user-account-better-auth/design.md) Decisions 1/4) plus `?invitationId=…`; when that role is unset (e.g. `user-account` not deployed) the template falls back to instruction copy. An app that wants bespoke invite copy points **`auth.email.templates.invitation`** at one of its own `notifications:` entries — an app-level `auth:` override the engine reads, not a module dependency. This **supersedes the earlier `invitation.send` escape hatch**: that synthetic hook point is **removed** upstream (auth-emails), so there is no binding to bind and none to drop — branding is `auth.email.templates.invitation`, uniform with the other three flows. Resend rides `InviteMember`'s native `resend` (re-renders through the same `auth.email` send path); cancel is `CancelInvitation`; expiry and auto-cancel-if-member are BetterAuth's. This preserves the contact-uniqueness invariant the old `check` page existed for (one contact per email), backed by the **unique index on the contact's `lowercase_email`** and the invitation-accept `profile` merge carrying `contactId` (admin Decision 5). (The module's partial-unique `users` index on `profile.contactId` enforces one `user` per `contact`, a different invariant — it does not prevent duplicate contacts for one email.)

**Member attributes are captured on the invite form**, exactly as today's invite captures `app_attributes` — they carry authorization parameters an invited user must hold from their first session, so "edit them after acceptance" would open a window where the user is in with no attributes. The invitation stores them (`invitation.additionalFields`) and an accept-time hook applies them to the minted member row, alongside the engine's `profile` merge (carrying `contactId`). This platform support is **already delivered** — admin Decision 5 has `InviteMember` accept an `attributes` object stored on the invitation and the `afterAcceptInvitation` hook copy it onto the minted member row (formerly upstream ask 4, now resolved).

The public **accept page is not this module's** — it belongs with the auth-pages modules, like the login page.

### 8. Extension surface — same slot philosophy, renamed to the model

- **`fields.profile`** (contact fields — binds `state.profile.*`), **`fields.user_attributes`** (was `global_attributes`), **`fields.member_attributes`** (was `app_attributes`). Same blocks serve tile display and edit modals.
- **Roles come from the platform role catalog, not a module var.** The module reads the app's **explicit role catalog** — the authored `auth.roles` list (`{ id, label, description }` entries), exposed to the module through the `_build.authConfig.roles` projection ([role-catalog](../../../lowdefy-design/designs/auth-upgrade/features/role-catalog/design.md), which resolves upstream ask 7) — for the assignable set, display labels, **and help text**: the module renders each role's `label` as the option text and its optional `description` as a subtitle/tooltip beside it in the role picker (the access-edit modal and the invite form's role selector), so an admin assigning roles sees what each grants. A role with no `description` renders label-only. This makes the catalog the single source of truth: a role exists for the app because it is _declared_, independent of whether any page gates on it (an app may assign a role purely to branch display-only UI off `_user.roles`). The old author-supplied `roles` var (`[{label, value}]`, a second list that could drift from what the steps accept) is **retired** — every declared id is registered in the org plugin's AC, so `UpdateMemberRoles` accepts exactly the declared ids, and page/api/ws `roles:` references are build-validated against the catalog. Under `pinned`, `auth.userAdminRole` is itself declared in the catalog (role-catalog Decision 6), so the admin-roles gate of Decision 3 references a catalog role like any other.
  - **Orphaned roles (in data, not in the catalog).** Config never rewrites `member.role`, so a member can hold a role later removed from the catalog. The module **displays** such a value as a raw, flagged "no longer configured" chip (never hides it) and lets the admin **remove** it; it is never silently stripped, and a role save submits only catalog-valid ids to the AC (so an untouched orphan can't fail the write with `ROLE_NOT_FOUND`). Bulk cleanup of orphaned strings across data is a migration/consumer concern — removing a role from config is deliberate and does not mutate data.
- **`components.*`** slots carry over: `table_columns`, `download_columns`, `filters` (+ `filter_requests`), `main_slots`, `sidebar_slots`, tile overrides.
- **`request_stages.*`** survives for **reads** (list pipeline, filter match, export) and for the **contact write** (`request_stages.write` on the profile routine, per the module contract). `filter_match` now takes plain `$match` clauses — there is no `$search` stage to extend (Decision 2). The auth-side write seam is routine steps, not pipeline stages — no slot re-creates one.
- `event_display`, `avatar_colors`, `app_title` carry over in kind. New vars: `impersonation` (Decision 5), `suspension` (Decision 4), `download` (Decision 2); an admin-roles var gates the routine endpoints (Decision 3).

**Dependencies**: `layout`, `events` unchanged. `notifications` is **dropped** — the invite email now rides `auth.email` (Decision 7), so the module no longer depends on notifications for dispatch. The email redesign has since landed ([auth-emails](../../../lowdefy-design/designs/auth-upgrade/_completed/auth-emails/design.md)) and did **not** reintroduce a module-shipped dispatch path: an app that wants branded invite copy points `auth.email.templates.invitation` at one of its own `notifications:` entries (app `auth:` config the engine reads), which adds no module dependency. The two shared fragments — `create-or-link-contact` (Decision 7, invite upsert) and `write-profile` (Decision 3, contact write + `UpdateUserProfile` re-denorm) — are **not** dependencies: both are `modules/shared/` files `_ref`'d by relative path (also `_ref`'d by `user-account`), adding no module edge.

| ----------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Pages       | `all` (Members + Invitations tabs, filters; export behind the `download` var), `view` (workspace), `invite`                                                                                                                                                                                             |
| APIs        | `invite`, `check-invite-email`, `cancel-invitation`, `resend-invitation`, `update-profile`, `update-access`, `update-user-attributes`, `suspend`, `reinstate`, `revoke-sessions`, `remove-member`, `delete-user` (invite email is sent by BetterAuth via `auth.email` — no module endpoint, Decision 7) |
| Connections | `user-contacts-collection` (app connection), plus one read-only connection per auth collection the module reads natively: `users`, `user-members`, `user-invitations`, `user-sessions`, `user-accounts`, `user-organizations` — names are adapter-fixed (mongodb Decision 2)                            |
| Menus       | `default` (Users)                                                                                                                                                                                                                                                                                       |

Each native read is an aggregation starting from the relevant auth-collection connection, `$lookup`-joining the others. This rests on the same-database co-location precondition (Decision 1): the auth adapter, the `user-contacts` connection, and the module's read connections must all resolve to one MongoDB database, or the joins to `user-contacts` silently return empty.

Retired vs today: `edit` and `check` pages (`new` is renamed `invite`), `app_name` var, `resend-invite` as a bespoke notification flow, the Atlas `$search` stage and its index requirement (Decision 2), all raw writes to auth-owned data. A consumer migration guide (v0.9 vars/slots/pages → this surface) is an implementation task, written once this design is finalised.

## Upstream asks (feedback into the auth-upgrade designs)

The platform-side changes this design depends on are specified in **[upstream-asks.md](upstream-asks.md)**. Verified against the current auth-upgrade designs, **asks 1–5 are resolved upstream**, **ask 6 is dropped** (the invite email now rides `auth.email`, so there is no binding to wire), and **ask 7 is resolved** (the [role-catalog](../../../lowdefy-design/designs/auth-upgrade/features/role-catalog/design.md) design delivers the authored `auth.roles` catalog) — so this design has **no open platform dependency**:

1. ~~**Pinned-org defaulting**~~ — **Resolved.** Steps default `organizationId` to the pinned org; the engine's server-side `_organization` operator exposes the resolved org id for native `$match` reads (no `org` var). (Decision 1.)
2. ~~**Admin identity-field stance**~~ — **Resolved.** admin Decision 1 makes "no set-email / set-name / set-password steps" an explicit decision (Decision 4 relies on cancel + re-invite).
3. ~~**Impersonation AC**~~ — **Resolved.** The engine's _The user-admin role_ section (phase 9) adds an `auth.userAdminRole` config option gating the whole user-administration surface — a step-level floor on every auth admin step plus impersonation, its client-action case (bridged via a `user.role` denormalization and a curated `user: ['impersonate']` admin-plugin AC). Impersonation stays behind its var, now backed by a working capability (Decision 5).
4. ~~**Attributes on invitations**~~ — **Resolved.** admin Decision 5 stores invite-time attributes on the invitation and copies them to the member at accept (Decision 7).
5. ~~**Attributes as native BSON**~~ — **Resolved.** mongodb Decision 5 settled on native sub-documents, delivered by **vendoring** the adapter into `@lowdefy/connection-mongodb` (`supportsJSON: true` + string-parse-on-read for legacy rows) — the earlier pnpm-patch/fork approach was abandoned (`patchedDependencies` don't reach consumer installs of the published `@lowdefy/server`). Attribute filters are safe; the upstream PR is now optional goodwill, not a live dependency.
6. ~~**Module-exported hook bindings**~~ — **Dropped.** The module no longer ships the `invitation.send` dispatch endpoint (invite email rides `auth.email` — Decision 7), so there is no binding to export. The email redesign has since landed ([auth-emails](../../../lowdefy-design/designs/auth-upgrade/_completed/auth-emails/design.md)) and **removes the `invitation.send` hook point entirely** — branded invites are now `auth.email.templates.invitation` — so this ask does not resurface via that path.
7. ~~**Explicit role catalog**~~ — **Resolved.** The [role-catalog](../../../lowdefy-design/designs/auth-upgrade/features/role-catalog/design.md) design makes `auth.roles` an authored `{ id, label, description }` catalog: every declared id registers in the org plugin's AC (gate-less display-only roles included), page/api/ws `roles:` references are build-validated against it, `userAdminRole` is declared in it, and it is exposed to modules via the `_build.authConfig.roles` projection — retiring the module's `roles` var (Decision 8) and grounding orphaned-role handling (role-catalog Decision 8 mirrors Decision 8 here).

## Non-goals

- **Multi-tenant administration** (`policy: tenant` — org switching, tenant creation, seat management). Separate module, designed after this one lands.
- **`user-account` redesign** — the self-service side (own profile, own sessions, passkey enrolment, MFA setup). Designed in [user-account-better-auth](../user-account-better-auth/design.md), which also owns the auth pages (login, signup, accept, verify).
- **Auth pages** — login, signup, invitation accept, verify. Auth-page modules own these.
- **Contact administration** beyond what the invite flow needs — the `contacts` module owns pure contacts.
- **Role-type management** — the module assigns catalog roles; minting role names is out of scope platform-wide.
- **Timed / expiring bans** — suspend issues a permanent ban only; the module never sets `banExpires` and the status read never checks it. BetterAuth supports timed bans, so this is available if a concrete need appears — it isn't one here.
- **Admin email change for active users** — a wrong login email caught after acceptance is handled as a new user: remove / delete and re-invite (upstream ask 2 records the platform stance; pre-accept it's cancel + re-invite). If a concrete need surfaces, the path is a designed upstream step with verification semantics, exposed behind a module var — admin-side only, never self-service.
